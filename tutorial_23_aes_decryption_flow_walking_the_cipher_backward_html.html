AES Decryption Flow: Walking the Cipher Backward

How AES-128 decryption starts from ciphertext, uses round keys in reverse order, applies inverse layers, and recovers the plaintext block.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 22 mapped AES-128 encryption from plaintext to ciphertext. This tutorial walks that same machine backward. Decryption is not a new unrelated cipher. It is the carefully ordered reversal of encryption, using inverse layers and round keys in reverse order.

AES decryption is the return journey through the maze. You do not undo the first door first. You undo the last door first, then step backward through the path until the original plaintext is recovered.

What you should know first

You should know the AES encryption flow, the normal round structure, the final round difference, and the inverse operations: InvSubBytes, InvShiftRows, InvMixColumns, and AddRoundKey. Remember that AddRoundKey is its own inverse because XOR undoes itself.

The full AES-128 decryption flow

Intuition first

Encryption ends with AddRoundKey using \(K_{10}\). Therefore decryption begins by applying AddRoundKey with \(K_{10}\). From there, decryption uses inverse layers and round keys in reverse order until \(K_0\) is removed at the end.

Encryption flow reversed

AES-128 encryption flow is:

\[ K_0\ \text{initial add}\rightarrow\text{rounds }1\text{-}9\rightarrow\text{round }10\rightarrow\text{ciphertext} \]

AES-128 decryption walks backward:

\[ K_{10}\ \text{first}\rightarrow\text{rounds }9\text{-}1\rightarrow K_0\ \text{last}\rightarrow\text{plaintext} \]
Reverse-order rule

If encryption applies A, then B, then C, decryption must undo C first, then B, then A. This rule drives the entire AES decryption flow.

Step 1: Undo the final encryption round

The final encryption round was:

\[ \text{SubBytes}\rightarrow\text{ShiftRows}\rightarrow\text{AddRoundKey}(K_{10}) \]

To undo it, decryption applies the inverse steps in reverse order:

\[ \text{AddRoundKey}(K_{10})\rightarrow\text{InvShiftRows}\rightarrow\text{InvSubBytes} \]
Why no InvMixColumns here?

The final encryption round did not include MixColumns. Since there was no MixColumns in the final round, there is no InvMixColumns needed when undoing that final round.

Step 2: Undo rounds 9 through 1

A normal encryption round applies:

\[ \text{SubBytes}\rightarrow\text{ShiftRows}\rightarrow\text{MixColumns}\rightarrow\text{AddRoundKey}(K_r) \]

The matching inverse round applies:

\[ \text{AddRoundKey}(K_r)\rightarrow\text{InvMixColumns}\rightarrow\text{InvShiftRows}\rightarrow\text{InvSubBytes} \]

Decryption performs this for \(r=9,8,7,\ldots,1\).

Decryption phaseRound keyInverse layers
Undo round 9\(K_9\)AddRoundKey, InvMixColumns, InvShiftRows, InvSubBytes
Undo round 8\(K_8\)AddRoundKey, InvMixColumns, InvShiftRows, InvSubBytes
⋮⋮Same inverse main-round structure
Undo round 1\(K_1\)AddRoundKey, InvMixColumns, InvShiftRows, InvSubBytes

Step 3: Undo the initial AddRoundKey

Encryption began with:

\[ \text{State}_0=\text{Plaintext}\oplus K_0 \]

Decryption ends by applying AddRoundKey with \(K_0\) again:

\[ \text{Plaintext}=\text{State}_0\oplus K_0 \]
Why the same operation works

XOR is its own inverse. If \(x\oplus k=y\), then \(y\oplus k=x\). This is why AddRoundKey is used both for encryption key addition and decryption key removal.

AES-128 decryption timeline

Here is AES-128 decryption as a timeline.

Input
Load the 16-byte ciphertext block into the AES state.
Undo final
Apply AddRoundKey with \(K_{10}\), then InvShiftRows, then InvSubBytes.
Undo 9-1
For \(r=9\) down to \(1\), apply AddRoundKey with \(K_r\), InvMixColumns, InvShiftRows, and InvSubBytes.
Undo initial
Apply AddRoundKey with \(K_0\).
Output
Read the resulting state as the plaintext block.

AES-128 decryption pseudocode

In readable pseudocode, AES-128 decryption looks like this:

state = ciphertext
state = AddRoundKey(state, K10)
state = InvShiftRows(state)
state = InvSubBytes(state)

for round = 9 down to 1:
    state = AddRoundKey(state, K_round)
    state = InvMixColumns(state)
    state = InvShiftRows(state)
    state = InvSubBytes(state)

state = AddRoundKey(state, K0)

plaintext = state

This pseudocode matches the conceptual reversal of the encryption flow. Some optimized implementations rearrange equivalent steps, but this is the clearest learning version.

Layer inverses

Each encryption layer has a matching undo operation.

Encryption layerDecryption inverseWhy it works
SubBytesInvSubBytesThe AES S-box is bijective.
ShiftRowsInvShiftRowsShiftRows is a reversible byte permutation.
MixColumnsInvMixColumnsThe MixColumns matrix is invertible over \(\mathrm{GF}(2^8)\).
AddRoundKeyAddRoundKeyXOR undoes itself.

Round keys are used backward

Encryption uses round keys in increasing order:

\[ K_0,K_1,K_2,\ldots,K_9,K_{10} \]

Decryption uses them in decreasing order:

\[ K_{10},K_9,K_8,\ldots,K_1,K_0 \]
Memory hook

The last key used in encryption is the first key used in decryption. The first key used in encryption is the last key used in decryption.

Common mistakes

Mistake 1: starting decryption with \(K_0\)

Decryption starts with \(K_{10}\), because \(K_{10}\) was the final key applied during encryption.

Mistake 2: applying inverse layers in the forward order

Decryption must reverse the order of encryption layers. The last operation applied is the first one undone.

Mistake 3: using InvMixColumns when undoing the final round

The final encryption round omitted MixColumns, so the first decryption phase does not use InvMixColumns.

Mistake 4: thinking AddRoundKey needs a special inverse

AddRoundKey is XOR. Applying the same round key again undoes it.

Practice problems

  1. Which round key is used first during AES-128 decryption?
    Show solution

    \(K_{10}\).

  2. Which round key is used last during AES-128 decryption?
    Show solution

    \(K_0\).

  3. Why is there no InvMixColumns immediately after the first AddRoundKey in decryption?
    Show solution

    Because the final encryption round did not include MixColumns.

  4. What inverse operation undoes ShiftRows?
    Show solution

    InvShiftRows.

  5. Why does AddRoundKey undo itself?
    Show solution

    Because XOR undoes itself: \(x\oplus k\oplus k=x\).

Why this matters for cryptography

AES decryption demonstrates a major principle of block-cipher design: secure scrambling can still be reversible. Each encryption layer is designed to be undone with the correct inverse operation and the correct round key.

  • Decryption begins with ciphertext, not plaintext.
  • Round keys are used from \(K_{10}\) down to \(K_0\).
  • The final encryption round is undone first.
  • Main encryption rounds are undone from round 9 down to round 1.
  • The original plaintext appears only after \(K_0\) is removed.

One-page recap

What to remember
  • AES-128 decryption starts from the ciphertext block.
  • The first decryption key used is \(K_{10}\).
  • The last decryption key used is \(K_0\).
  • Decryption uses round keys in reverse order.
  • The final encryption round is undone first.
  • Undoing the final round uses AddRoundKey, InvShiftRows, InvSubBytes.
  • No InvMixColumns is used in that first decryption phase because final encryption omitted MixColumns.
  • Main rounds are undone from round 9 down to round 1.
  • Each inverse main round uses AddRoundKey, InvMixColumns, InvShiftRows, InvSubBytes.
  • The final decryption step is AddRoundKey with \(K_0\), producing plaintext.
