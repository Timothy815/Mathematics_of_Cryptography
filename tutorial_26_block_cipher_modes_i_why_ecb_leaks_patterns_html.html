Block Cipher Modes I: Why ECB Leaks Patterns

Why encrypting each block independently can reveal structure, and why a secure block cipher still needs a safe mode of operation.

🏠 Dashboard

Why this tutorial exists

Purpose

So far, we have studied AES as a machine that encrypts one 128-bit block. Real messages are often longer than one block. A block cipher mode tells us how to use a block cipher repeatedly on multi-block data. This tutorial studies the simplest mode, Electronic Codebook mode, and why it is usually unsafe.

AES can be mathematically strong, but a poor mode can still leak information. Electronic Codebook mode is the classic warning: identical plaintext blocks produce identical ciphertext blocks. The lock may be excellent, but the filing system gives away the pattern.

What you should know first

You should know that AES encrypts one 128-bit block at a time. You should also understand that encryption should hide not only the exact content of a message, but also useful patterns about that content.

What is a block cipher mode?

Definition: mode of operation

A block cipher mode of operation is a rule for using a block cipher on data. It tells us how to handle one block, many blocks, repeated blocks, initialization values, counters, chaining, and sometimes authentication.

AES itself encrypts exactly one 128-bit block at a time. A mode of operation answers a practical question:

\[ \text{How do we safely encrypt a message with many blocks?} \]

What is ECB?

Definition: Electronic Codebook mode

Electronic Codebook mode, or ECB, encrypts each plaintext block independently using the same key.

\[ C_i=E_K(P_i) \]

Friendly meaning: take each plaintext block, feed it into AES with the same key, and write down the resulting ciphertext block. Then do the same thing for the next block, without using any information from the previous block.

ECB pattern example

Suppose a message has repeated blocks:

Plaintext \(P_1\)
HELLO-HELLO-0001
Plaintext \(P_2\)
PAYLOAD-BLOCK-02
Plaintext \(P_3\)
HELLO-HELLO-0001
Plaintext \(P_4\)
ENDING-BLOCK-004

In ECB, identical plaintext blocks become identical ciphertext blocks:

Ciphertext \(C_1\)
A91F...
Ciphertext \(C_2\)
72C4...
Ciphertext \(C_3\)
A91F...
Ciphertext \(C_4\)
D88B...
Main ECB problem

If \(P_1=P_3\), then ECB gives \(C_1=C_3\). The attacker may not know what the repeated block says, but they can see that the same block occurred twice.

The formal pattern leak

In ECB mode:

\[ C_i=E_K(P_i) \]

Since the same key \(K\) is used for each block, the same plaintext block always produces the same ciphertext block:

\[ P_i=P_j\quad\Longrightarrow\quad E_K(P_i)=E_K(P_j)\quad\Longrightarrow\quad C_i=C_j \]
What is leaked?

ECB does not necessarily reveal the plaintext bytes directly. It reveals equality patterns: which encrypted blocks came from identical plaintext blocks. Depending on the data, that may be enough to reveal useful information.

The image-pattern problem

ECB is famously bad for structured data such as images. Even after encryption, repeated regions can leave visible structure because identical plaintext blocks become identical ciphertext blocks.

ECB does not make repeated regions behave independently. Equal input blocks still create equal output blocks, so large-scale structure can survive as a visible ghost.

Why is it called a codebook?

The name “Electronic Codebook” hints at its weakness. If the same block always maps to the same encrypted block under the same key, then the encryption behaves like a giant codebook lookup.

Plaintext blockECB ciphertext block
AAAA...F13C...
BBBB...19D0...
AAAA...F13C...

The attacker may not know the full codebook, but they can still recognize repeated entries.

Why ECB is not enough for most messages

ECB lacks chaining, randomness, and position-dependent variation. That means it does not hide repeated blocks. In practical cryptography, this is usually unacceptable for multi-block data.

Preview: what better modes try to fix

Safer modes avoid ECB's direct “same plaintext block gives same ciphertext block” behavior. They do this by adding chaining, counters, nonces, initialization vectors, or authentication.

Preview vocabulary

An initialization vector, or IV, is a starting value used by some modes to make encryption depend on more than just the plaintext block and key. A nonce is a number used once. These ideas will matter in later tutorials.

Later tutorials will cover modes such as CBC, CTR, and authenticated encryption modes. For now, the lesson is simple: do not judge a system only by the block cipher. Judge the mode too.

Common mistakes

Mistake 1: thinking strong AES automatically means safe message encryption

AES is a strong block cipher, but unsafe use can still leak information. ECB is the classic example.

Mistake 2: thinking ECB leaks the key

ECB does not normally leak the key directly. Its major problem is pattern leakage.

Mistake 3: thinking different positions fix repeated blocks

In ECB, block position does not change the encryption. The same plaintext block encrypts the same way wherever it appears.

Mistake 4: thinking encryption also proves authenticity

ECB only encrypts blocks. It does not authenticate the message or protect against block rearrangement.

Why this matters for cryptography

ECB teaches a humbling cryptographic lesson: a strong primitive is not enough. Correct construction matters. AES can hide a single block well, but ECB can reveal patterns when many blocks are encrypted independently.

  • AES encrypts fixed-size blocks.
  • Modes define how AES is used on messages.
  • ECB encrypts each block independently.
  • Identical plaintext blocks produce identical ciphertext blocks.
  • This leaks structure, especially in repeated or patterned data.
  • Better modes are designed to hide these repeated-block patterns.
