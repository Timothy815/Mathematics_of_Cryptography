Block Cipher Modes II: CBC, Initialization Vectors, and Chaining

How Cipher Block Chaining mode uses an initialization vector and previous ciphertext blocks to reduce ECB-style repeated-block pattern leakage.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 26 showed why ECB leaks patterns: identical plaintext blocks produce identical ciphertext blocks. This tutorial introduces Cipher Block Chaining mode, or CBC, which mixes each plaintext block with the previous ciphertext block before encryption.

CBC makes each block carry a little history. Instead of encrypting every block as if it lived alone, CBC chains blocks together so the same plaintext block can encrypt differently depending on what came before it.

What you should know first

You should know that AES encrypts one 128-bit block at a time, that ECB encrypts blocks independently, and that XOR combines bits in a reversible way. You should also remember that ECB's main weakness is repeated-block pattern leakage.

The big picture

Intuition first

In ECB, each plaintext block walks straight into AES. In CBC, each plaintext block first gets XORed with the previous ciphertext block. The first block has no previous ciphertext, so CBC uses an initialization vector as the starting block.

CBC encryption formulas

CBC needs a starting value for the first block. That value is the initialization vector, written as \(IV\).

\[ C_1=E_K(P_1\oplus IV) \] \[ C_i=E_K(P_i\oplus C_{i-1})\quad\text{for }i>1 \]
Definition: initialization vector

An initialization vector, or IV, is a block-sized starting value used by CBC for the first block. It should be unpredictable or random for CBC encryption and does not need to be secret, but it must be available for decryption.

Why does CBC need an IV?

For block 2 and later, CBC uses the previous ciphertext block. But block 1 has no previous ciphertext block. The IV fills that role.

Memory hook

The IV is the “previous ciphertext” for the first block. After that, each real ciphertext block becomes the chaining value for the next block.

CBC versus ECB

ECB encrypts equal plaintext blocks the same way. CBC changes the input to the block cipher by XORing each plaintext block with a chaining value first.

ModeBlock input to AESEffect on repeated plaintext blocks
ECB\(P_i\)Repeated plaintext blocks produce repeated ciphertext blocks.
CBC block 1\(P_1\oplus IV\)The IV changes the first block's AES input.
CBC block \(i>1\)\(P_i\oplus C_{i-1}\)The previous ciphertext changes the next block's AES input.
Plain-language version

CBC does not feed raw plaintext blocks into AES. It seasons each block with the previous ciphertext first. Same plaintext, different previous ciphertext, different AES input.

In ECB, \(P_1=P_3\) would force \(C_1=C_3\). In CBC, the AES inputs are different unless the chaining values are also the same:

\[ \text{AES input for block 1}=P_1\oplus IV \] \[ \text{AES input for block 3}=P_3\oplus C_2 \]

Even when \(P_1=P_3\), the values \(IV\) and \(C_2\) will generally differ, so the AES inputs differ.

CBC decryption formulas

CBC decryption reverses the process. First decrypt the ciphertext block, then XOR with the previous ciphertext block, or with the IV for the first block.

\[ P_1=D_K(C_1)\oplus IV \] \[ P_i=D_K(C_i)\oplus C_{i-1}\quad\text{for }i>1 \]

Why CBC decryption works

For block \(i>1\), encryption is:

\[ C_i=E_K(P_i\oplus C_{i-1}) \]

Decrypting \(C_i\) gives back the AES input:

\[ D_K(C_i)=P_i\oplus C_{i-1} \]

Then XOR with \(C_{i-1}\):

\[ D_K(C_i)\oplus C_{i-1}=(P_i\oplus C_{i-1})\oplus C_{i-1}=P_i \]
The XOR cancellation

The same chaining value is XORed in during encryption and XORed out during decryption. Since \(x\oplus y\oplus y=x\), the original plaintext block is recovered.

What must be true about the IV?

For CBC, the IV has important requirements:

IV propertyMeaningWhy it matters
Block-sizedFor AES-CBC, the IV is 128 bits.It must XOR with the first plaintext block.
Unpredictable/random for encryptionAn attacker should not be able to predict it in advance.This helps prevent first-block pattern leakage and chosen-plaintext tricks.
Not secretThe IV can be sent with the ciphertext.The key is secret; the IV is needed for decryption.
Not reused with the same keyFresh encryption should use a fresh IV.Reusing IVs can reveal repeated first-block relationships.

What CBC does not solve by itself

CBC improves on ECB's repeated-block leakage, but CBC encryption by itself does not provide authentication. An attacker may still be able to tamper with ciphertext unless the message is authenticated.

Important warning

Encryption is not the same as authentication. CBC needs a secure authentication method, or a modern authenticated encryption mode, when integrity matters.

Common mistakes

Mistake 1: thinking the IV must be secret

The IV does not need to be secret. It usually travels with the ciphertext. The key must be secret.

Mistake 2: reusing the same IV with the same key

CBC should use a fresh IV for each encryption under the same key.

Mistake 3: forgetting the first block is special

The first block uses the IV because there is no previous ciphertext block yet.

Mistake 4: thinking CBC automatically authenticates data

CBC hides patterns better than ECB, but it does not prove that the ciphertext was not modified.

Why this matters for cryptography

CBC shows how a mode can repair a major weakness of ECB by making each block depend on earlier ciphertext. It also introduces a crucial practical idea: encryption schemes often need starting values, freshness, and separate integrity protection.

  • CBC mixes each plaintext block with a chaining value before encryption.
  • The first chaining value is the IV.
  • Later chaining values are previous ciphertext blocks.
  • This prevents simple ECB-style repeated-block leakage.
  • The IV should be fresh and unpredictable for CBC encryption.
  • CBC encryption alone does not authenticate the message.
