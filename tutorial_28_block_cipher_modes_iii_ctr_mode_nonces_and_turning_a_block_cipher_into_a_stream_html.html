Block Cipher Modes III: CTR Mode, Nonces, and Turning a Block Cipher into a Stream

How Counter mode uses encrypted nonce-counter blocks as a keystream, why encryption and decryption both use XOR, and why nonce reuse is dangerous.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorials 26 and 27 studied ECB and CBC. This tutorial introduces Counter mode, or CTR. CTR uses a block cipher such as AES to generate a stream of pseudorandom-looking blocks. Those blocks are XORed with the plaintext to produce ciphertext.

CTR mode turns AES into a keystream generator. Instead of encrypting the plaintext block directly, AES encrypts a nonce-plus-counter value. The result becomes a keystream block, and the keystream is XORed with the message.

What you should know first

You should know that AES encrypts 128-bit blocks, that XOR undoes itself, and that modes of operation define how a block cipher is used on longer messages. You should also remember that IVs and nonces are not usually secret, but they must be handled correctly.

The big picture

Intuition first

In CBC, plaintext is mixed before encryption. In CTR, plaintext is not sent into AES at all. AES encrypts a sequence of nonce-counter blocks to create a keystream. The plaintext is then XORed with that keystream.

CTR encryption formulas

A nonce is combined with a counter value to create a unique input block for AES. We will write this as \(N\|i\), meaning “nonce \(N\) combined with counter \(i\).”

\[ S_i=E_K(N\|i) \] \[ C_i=P_i\oplus S_i \]
Definition: nonce

A nonce is a value used once with a given key. In CTR mode, the nonce helps ensure that the same counter sequence does not create the same keystream under the same key.

Definition: keystream

A keystream is a sequence of bits or blocks that are XORed with plaintext to produce ciphertext. In CTR, the keystream is made by encrypting nonce-counter blocks.

CTR decryption uses the same keystream

CTR decryption regenerates the same keystream block and XORs it with the ciphertext:

\[ S_i=E_K(N\|i) \] \[ P_i=C_i\oplus S_i \]

Notice that CTR decryption uses the block cipher's encryption function \(E_K\), not the block cipher's decryption function \(D_K\). The block cipher is being used to generate the keystream.

Why XOR makes this work

Encryption computes \(C_i=P_i\oplus S_i\). Decryption computes \(C_i\oplus S_i=(P_i\oplus S_i)\oplus S_i=P_i\). The same keystream block cancels itself out.

The counter sequence

CTR mode changes the counter for each block. The nonce stays the same for one message, while the counter increases.

Memory hook

CTR is a numbered ticket machine for keystream blocks: same key, same nonce, new counter value for each block.

The danger of nonce reuse

Golden rule

Never reuse the same nonce with the same key in CTR mode. Reusing the nonce repeats the keystream. Repeating the keystream can expose relationships between plaintexts.

If two messages use the same key and nonce, then they generate the same keystream:

\[ C_i=P_i\oplus S_i \] \[ C'_i=P'_i\oplus S_i \]

XOR the two ciphertext blocks together:

\[ C_i\oplus C'_i=(P_i\oplus S_i)\oplus(P'_i\oplus S_i)=P_i\oplus P'_i \]

The keystream cancels out. That does not automatically reveal both plaintexts, but it leaks a dangerous relationship between them. If one plaintext is known or guessable, the other can often be recovered.

What CTR does not solve by itself

CTR mode provides confidentiality when used correctly, but it does not authenticate the message. An attacker who flips a bit in the ciphertext flips the corresponding bit in the decrypted plaintext.

Malleability warning

CTR encryption is malleable. That means ciphertext changes can cause predictable plaintext changes after decryption. Modern systems usually pair CTR-like encryption with authentication, or use an authenticated encryption mode such as GCM.

Common mistakes

Mistake 1: encrypting the plaintext directly with AES in CTR

CTR encrypts nonce-counter blocks to make a keystream. The plaintext is XORed with the keystream.

Mistake 2: using AES decryption during CTR decryption

CTR decryption uses AES encryption again to regenerate the same keystream.

Mistake 3: reusing a nonce with the same key

Nonce reuse repeats the keystream. That is one of the most dangerous CTR mistakes.

Mistake 4: assuming CTR authenticates data

CTR provides encryption, not authentication. It must be paired with integrity protection or used inside an authenticated encryption mode.

Why this matters for cryptography

CTR mode is powerful because it turns a block cipher into a stream-like construction. It is efficient, parallelizable, and elegant. But it also shows a recurring cryptographic law: freshness values are not decorative. If a nonce is reused with the same key, the safety of the keystream collapses.

  • CTR encrypts nonce-counter blocks, not plaintext blocks.
  • The AES outputs become keystream blocks.
  • Plaintext is XORed with the keystream to make ciphertext.
  • Decryption regenerates the same keystream and XORs again.
  • Nonce reuse with the same key repeats the keystream.
  • CTR needs authentication for integrity.
