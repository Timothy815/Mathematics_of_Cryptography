How to use this workbook

CTR mode is easiest if you separate two jobs: first generate the keystream by encrypting nonce-counter values, then XOR the keystream with plaintext or ciphertext. The same keystream must never be reused with the same key.

Quick reference

CTR formulas
\[ S_i=E_K(N\|i) \] \[ C_i=P_i\oplus S_i \] \[ P_i=C_i\oplus S_i \]
Nonce-reuse danger
\[ C_i\oplus C'_i=P_i\oplus P'_i \]

Same key plus same nonce means same keystream. That is dangerous.

Part I: Concept checks

I1 sample answer. CTR encrypts nonce-counter blocks to create a keystream, then XORs that keystream with plaintext. This makes AES behave like a stream cipher construction.

I2 sample answer. Nonce reuse is dangerous because it repeats the keystream. If two ciphertexts use the same keystream, XORing them cancels the keystream and reveals the XOR of the plaintexts.

I3 sample answer. No. CTR encryption alone does not authenticate a message. An attacker may be able to modify ciphertext bits and cause predictable changes in plaintext.

I4 sample answer. CTR mode uses AES to encrypt nonce-counter inputs and produce keystream blocks. Each plaintext block is XORed with its matching keystream block to create ciphertext. Decryption regenerates the same keystream and XORs again. CTR is efficient and stream-like, but nonce reuse with the same key is dangerous and authentication is still needed.
