Block Cipher Modes IV: Padding, Message Length, and Why Blocks Need Filling

Why some block cipher modes need padding, how PKCS#7-style padding works, and why padding must be validated safely.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

AES works on 16-byte blocks. But real messages are not always a perfect multiple of 16 bytes. Padding is a rule for filling the final block so a block cipher mode can process it correctly.

Padding is the art of making the last box full without losing the ability to tell what was real message and what was filler. Done well, it is quiet housekeeping. Done badly, it can become a side-channel trapdoor.

What you should know first

You should know that AES has a 128-bit block size, which equals 16 bytes. You should also know that modes like ECB and CBC process data in full blocks, while CTR mode works more like a stream and does not need padding in the same way.

The big picture

Intuition first

A block cipher mode that requires full blocks cannot encrypt a half-filled final block. Padding adds carefully labeled filler bytes so the final block reaches the required size and can be removed later without confusion.

Why padding is needed

AES encrypts exactly 16 bytes at a time. If a message is 30 bytes long, it can be split into one full 16-byte block and one 14-byte final chunk. That final chunk is too short for modes that require complete blocks.

Important distinction

Padding is needed for block-oriented modes such as ECB and CBC when the message is not naturally encoded into full blocks. CTR does not need this kind of padding because it can generate as much keystream as needed and XOR only the message bytes.

PKCS#7-style padding

Definition: PKCS#7-style padding

PKCS#7-style padding fills the final block by adding \(n\) bytes, each with value \(n\), where \(n\) is the number of padding bytes needed.

For AES, the block size is 16 bytes. If the final chunk is 14 bytes, it needs 2 padding bytes:

... data data 02 02

If the final chunk is 10 bytes, it needs 6 padding bytes:

... data data 06 06 06 06 06 06

If the message already ends exactly on a block boundary, PKCS#7 adds a full padding block:

10 10 10 10 10 10 10 10 10 10 10 10 10 10 10 10
Why add a full block when already aligned?

Without a full padding block, the receiver could not always tell whether the final byte was real data or a padding length marker. The padding must be unambiguous.

How many padding bytes?

Let \(B\) be the block size in bytes and \(L\) be the message length in bytes. For AES, \(B=16\). First compute the remainder:

\[ r=L\bmod B \]

Then the PKCS#7 padding length is:

\[ n=\begin{cases} B-r, & \text{if }r\ne 0\\ B, & \text{if }r=0 \end{cases} \]

Removing padding

To remove PKCS#7-style padding after decryption, the receiver checks the final byte. That byte says how many padding bytes should be present.

Padding validation and padding oracles

Padding must be checked carefully. Some systems have accidentally revealed whether padding was valid through error messages, timing differences, or different server behavior. This can create a padding oracle.

Definition: padding oracle

A padding oracle is a system behavior that tells an attacker whether decrypted padding was valid. Even a tiny yes/no signal can sometimes be used to learn information about the plaintext.

For this beginner tutorial, the key lesson is not to implement your own padding checks casually. In real systems, use vetted cryptographic libraries and authenticated encryption modes when possible.

Common mistakes

Mistake 1: adding zero bytes and hoping for the best

Zero padding can be ambiguous if the original message may naturally end with zero bytes. Padding should be unambiguous.

Mistake 2: forgetting the full padding block

With PKCS#7, a message that already fits perfectly still gets a full block of padding.

Mistake 3: assuming padding provides security

Padding only makes the message fit the block size. It does not provide confidentiality, integrity, or authentication by itself.

Mistake 4: leaking padding errors

Different error messages or timing behavior can create padding oracle vulnerabilities.

Practice problems

  1. What is the AES block size in bytes?
    Show solution

    16 bytes.

  2. A message is 30 bytes long. How many PKCS#7 padding bytes are needed for AES?
    Show solution

    30 mod 16 is 14, so 2 padding bytes are needed: 02 02.

  3. A message is exactly 16 bytes long. How many PKCS#7 padding bytes are added?
    Show solution

    16 padding bytes are added, each with value 10 in hexadecimal.

  4. Why can zero padding be ambiguous?
    Show solution

    If the original message can end with real zero bytes, the receiver may not know which zeros are data and which are padding.

  5. Does CTR mode need PKCS#7-style padding?
    Show solution

    No. CTR can generate a keystream and XOR only the bytes needed.

Why this matters for cryptography

Padding seems like a small formatting issue, but cryptography is full of small formatting issues that matter deeply. A block cipher mode may be mathematically sound, but message encoding, padding, validation, and error behavior must also be safe.

  • AES has a 16-byte block size.
  • Some modes need complete blocks.
  • Padding fills the final block.
  • PKCS#7-style padding records the padding length in the padding bytes.
  • A full padding block is added when the message already fits exactly.
  • Bad padding validation can leak information.

What to remember
  • AES block size is 16 bytes.
  • ECB and CBC usually need padding for messages that do not end on a block boundary.
  • CTR does not need PKCS#7-style padding because it acts like a keystream mode.
  • PKCS#7-style padding adds \(n\) bytes, each with value \(n\).
  • If 2 bytes are needed, add 02 02.
  • If 6 bytes are needed, add 06 06 06 06 06 06.
  • If the message already fits exactly, add a full block of padding.
  • Padding must be unambiguous.
  • Padding does not authenticate a message.
  • Padding validation mistakes can create padding oracles.
