Block Cipher Modes V: Authentication, AEAD, and Why Encryption Alone Is Not Enough

Why confidentiality is only one part of secure communication, how authentication detects tampering, and why modern systems often use authenticated encryption.

🏠 Dashboard

Why this tutorial exists

Purpose

ECB, CBC, and CTR can provide confidentiality when used correctly. But confidentiality only means the message is hidden. It does not automatically mean the message has not been changed. This tutorial introduces authentication, Message Authentication Codes, and AEAD.

Encryption is a curtain. Authentication is a tamper seal. A secure system often needs both: hide the message from outsiders and detect whether someone changed it in transit.

What you should know first

You should know that block cipher modes such as CBC and CTR focus on encryption. You should also remember that CTR is malleable and CBC can be vulnerable if padding or errors are mishandled. These issues point toward a larger rule: encryption alone is not enough.

The big picture

Intuition first

Encryption answers, “Can outsiders read this?” Authentication answers, “Has this been changed, and was it produced by someone with the right secret?” These are related but different questions.

Confidentiality, integrity, and authentication

These three ideas are often mentioned together, but they are not identical.

Security propertyQuestion it answersExample failure
ConfidentialityCan an attacker read the message?An eavesdropper learns the plaintext.
IntegrityWas the message changed?An attacker flips bits or rearranges blocks.
AuthenticationWas this produced by someone with the right secret?An attacker forges a message or tag.
Precise relationship

Message authentication usually provides integrity and origin authentication under a shared secret key. It does not necessarily identify a human person by itself; it shows that the message was produced by someone who had the correct cryptographic key.

Why encryption alone is not enough

Some encryption modes allow meaningful changes to ciphertext. CTR is the easiest example: flipping a bit in the ciphertext flips the corresponding bit in the plaintext after decryption.

\[ C=P\oplus S \] \[ P=C\oplus S \]

If an attacker changes \(C\) to \(C'\), the decrypted plaintext becomes \(P'=C'\oplus S\). Without authentication, the receiver may not know the ciphertext was modified.

Core warning

Encryption without authentication can hide a message while still allowing tampering. Hidden does not mean trustworthy.

Message Authentication Codes

Definition: Message Authentication Code

A Message Authentication Code, or MAC, is a short tag computed from a message and a secret key. The receiver recomputes the tag and checks whether it matches.

\[ tag=MAC_K(message) \]

Friendly version: a MAC is a keyed tamper seal. If the message changes, the tag should no longer verify.

StepSenderReceiver
1Compute tag from message and key.Receive message and tag.
2Send message plus tag.Recompute tag from received message and key.
3Done.Accept only if tags match.

Encrypt-then-MAC

A common safe design principle is encrypt-then-MAC: first encrypt the plaintext, then authenticate the ciphertext and relevant metadata.

On receipt, the system should verify the tag before releasing plaintext. If verification fails, reject the message.

Plain-language rule

Check the tamper seal before trusting the contents. In many designs, do not hand decrypted data to the application until authentication succeeds.

Authenticated Encryption with Associated Data

Definition: AEAD

Authenticated Encryption with Associated Data, or AEAD, is a construction that provides both encryption and authentication. It can also authenticate extra data that is not encrypted.

AEAD usually takes:

  • a secret key,
  • a nonce or IV,
  • plaintext to encrypt,
  • associated data to authenticate but not encrypt.

It outputs ciphertext and an authentication tag.

\[ (ciphertext,tag)=AEAD\_Encrypt_K(nonce,plaintext,AAD) \]

During decryption, the tag must verify. If verification fails, the plaintext must be rejected.

What is associated data?

Definition: Additional Authenticated Data

Additional Authenticated Data, or AAD, is data that is not encrypted but is included in the authentication check. If AAD changes, verification fails.

Example: A network packet may have a header that routers need to read. The header cannot be encrypted if the network must route it, but it can still be authenticated so tampering is detected.

DataEncrypted?Authenticated?
Message bodyYesYes
Visible header / metadataNoYes, if included as AAD
Authentication tagNoUsed for verification

Examples of authenticated encryption modes

Two common authenticated encryption designs are AES-GCM and ChaCha20-Poly1305.

Important practical note

In real systems, use well-reviewed library implementations. Do not assemble encryption and authentication from scratch unless you are studying in a controlled learning environment.

Verify before trusting

In authenticated encryption, decryption is not just “decrypt and hope.” The authentication tag must be checked. If the tag is invalid, the message should be rejected.

StepActionResult
1Receive nonce, ciphertext, AAD, and tag.Inputs for verification.
2Check authentication tag.Accept only if valid.
3Release plaintext only after successful verification.Tampered messages are rejected.

Common mistakes

Mistake 1: thinking encryption proves integrity

Encryption hides content. It does not automatically prove that the ciphertext was not modified.

Mistake 2: ignoring the tag

In authenticated encryption, the tag is essential. If tag verification fails, reject the message.

Mistake 3: authenticating only part of the needed data

If visible metadata matters, include it as AAD so tampering is detected.

Mistake 4: nonce reuse in AEAD modes

Many AEAD modes require nonce uniqueness. Reusing nonces with the same key can be catastrophic.

Why this matters for cryptography

Modern cryptography is not only about hiding. It is also about detecting manipulation. Many historical and practical vulnerabilities came from treating encryption as if it automatically meant authenticity. AEAD reflects a more mature design principle: confidentiality and integrity should travel together.

  • Encryption provides confidentiality.
  • Authentication helps provide integrity and origin assurance under a key.
  • A MAC produces a keyed tag.
  • AEAD combines encryption and authentication.
  • AAD is authenticated but not encrypted.
  • Invalid tags should cause rejection.
