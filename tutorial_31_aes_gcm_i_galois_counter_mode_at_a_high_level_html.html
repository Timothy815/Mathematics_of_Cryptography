AES-GCM I: Galois Counter Mode at a High Level

A beginner-friendly overview of AES-GCM as authenticated encryption: CTR-style confidentiality plus Galois-field authentication.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 30 introduced authenticated encryption. AES-GCM is one of the most widely used authenticated encryption modes. This tutorial gives the high-level map before we study the math inside GHASH.

AES-GCM has two major jobs. It encrypts data using counter-mode ideas, and it authenticates data using arithmetic in a Galois field. One side hides the message; the other side checks the tamper seal.

What you should know first

You should know AES encrypts 128-bit blocks, CTR mode turns AES into a keystream generator, and AEAD means authenticated encryption with associated data. You do not need to know the internal GHASH multiplication yet. That comes later.

What AES-GCM provides

AES-GCM is an AEAD mode. That means it provides confidentiality for plaintext and authentication for both ciphertext and associated data.

Input / outputEncrypted?Authenticated?Purpose
PlaintextYesYesHidden and protected from tampering.
CiphertextAlready encryptedYesSent output that must verify.
AADNoYesVisible metadata protected from tampering.
TagNoUsed for verificationReceiver checks this before accepting.
AEAD reminder

AEAD does not mean everything is encrypted. It means the plaintext is encrypted, and both ciphertext and associated data are authenticated.

The encryption side: CTR-style keystream

GCM uses AES in a counter-like way for encryption. AES encrypts counter blocks derived from the nonce. The outputs become keystream blocks, which are XORed with plaintext.

\[ S_i=E_K(counter_i) \] \[ C_i=P_i\oplus S_i \]

This is why GCM inherits an important CTR rule: nonce reuse with the same key is dangerous. Reusing a nonce can repeat keystream and damage authentication security too.

The authentication side: GHASH

Definition: GHASH

GHASH is the authentication function used inside GCM. It combines AAD and ciphertext using multiplication and addition in \(GF(2^{128})\), producing a value that contributes to the final authentication tag.

Friendly version: GHASH is a mathematical mixing bowl. It takes visible associated data and ciphertext, stirs them with a secret-derived hash subkey \(H\), and helps produce a tag that changes if the protected data changes.

Why “Galois” is in the name

The authentication part of GCM uses arithmetic in the finite field \(GF(2^{128})\). That is the Galois-field part. The encryption part uses counters. Together: Galois/Counter Mode.

The hash subkey \(H\)

GHASH uses a special value called the hash subkey, written \(H\). It is created by encrypting the all-zero block with AES under the same key:

\[ H=E_K(0^{128}) \]

The notation \(0^{128}\) means 128 zero bits. AES encrypts that all-zero block, and the result becomes the GHASH subkey.

Friendly explanation

\(H\) is not a password and is not sent as the message. It is a key-derived ingredient used by the authentication mixer.

The tag verification rule

AES-GCM outputs ciphertext and a tag. During decryption, the receiver recomputes the expected tag. If the tag does not match, the message must be rejected.

StepReceiver actionResult
1Receive nonce, ciphertext, AAD, and tag.Inputs are ready for checking.
2Recompute expected tag.Compare expected tag with received tag.
3If tag matches...Accept and release plaintext.
4If tag fails...Reject the message.
Critical rule

Never ignore an authentication tag failure. A failed tag means the message may have been modified, the wrong key or nonce may have been used, or the data may be corrupted.

Why nonce uniqueness matters

AES-GCM requires nonce uniqueness for a given key. The nonce does not need to be secret, but it must not repeat with the same key.

Nonce propertyMeaningWhy it matters in GCM
Not secretCan be sent with ciphertext.Security comes from the key, not hiding the nonce.
Unique with keyDo not reuse the same nonce under the same key.Prevents keystream reuse and protects authentication security.
Often 96 bitsCommon recommended size.Efficient standard GCM processing.
Nonce reuse warning

Reusing a nonce with AES-GCM under the same key can be catastrophic. This is not a small bookkeeping mistake; it can break the security promises of the mode.

Common mistakes

Mistake 1: thinking GCM is just CTR

GCM uses CTR-style encryption, but it also includes GHASH authentication and a tag.

Mistake 2: ignoring AAD

AAD is not encrypted, but it is authenticated. If metadata matters, it must be included correctly.

Mistake 3: reusing a nonce with the same key

AES-GCM nonce reuse can be catastrophic. Nonce uniqueness is a core requirement.

Mistake 4: accepting plaintext after tag failure

If the tag fails, reject the message. Do not treat the decrypted bytes as trustworthy.

Why this matters for cryptography

AES-GCM is a bridge between the practical world and the mathematical world. It uses AES, counter-mode thinking, nonce discipline, associated data, authentication tags, and arithmetic in \(GF(2^{128})\). It is an ideal place to see why cryptography is not one trick, but a carefully engineered orchestra.

  • AES-GCM is an AEAD mode.
  • It encrypts plaintext using CTR-style keystream generation.
  • It authenticates ciphertext and AAD using GHASH.
  • GHASH uses arithmetic in \(GF(2^{128})\).
  • The tag must verify before the message is accepted.
  • Nonce uniqueness with a key is essential.
