AES-GCM II: GHASH Inputs, Blocks, and the Authentication Tag

How GCM arranges associated data, ciphertext, length information, and the hash subkey so GHASH can help produce the authentication tag.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 31 introduced AES-GCM at a high level. This tutorial looks more closely at the authentication side: what goes into GHASH, how those inputs are divided into blocks, why length information is included, and how the GHASH result helps form the final authentication tag.

GHASH is not merely hashing the ciphertext. It authenticates a carefully arranged record: associated data, ciphertext, and length information. The tag is the sealed summary that lets the receiver detect tampering.

What you should know first

You should know that AES-GCM is an AEAD mode, that AAD means Additional Authenticated Data, that ciphertext is the encrypted message body, and that the tag must verify before a message is accepted. You should also know that \(H=E_K(0^{128})\) is the GHASH hash subkey.

What goes into GHASH?

GHASH processes three conceptual regions:

RegionContainsWhy it is included
AAD regionAssociated data \(A\), padded to a full GHASH block if needed.Protects visible metadata from tampering.
Ciphertext regionCiphertext \(C\), padded to a full GHASH block if needed.Protects encrypted message bytes from tampering.
Length blockBit lengths of \(A\) and \(C\).Prevents ambiguity about where one region ends and the other begins.
Important distinction

GHASH zero padding is internal authentication formatting. It is not the same as PKCS#7 message padding from Tutorial 29. GHASH padding helps align authentication input to 128-bit blocks.

GHASH uses 128-bit blocks

GHASH works with 128-bit blocks, which is 16 bytes. AAD and ciphertext are split into 16-byte chunks. If either region does not end exactly on a 16-byte boundary, zeros are added internally for GHASH processing.

Input regionOriginal lengthGHASH block treatment
AAD16 bytesOne full block, no internal zero padding needed.
AAD20 bytesOne full block plus 4 bytes, then 12 zero bytes for GHASH alignment.
Ciphertext32 bytesTwo full blocks, no internal zero padding needed.
Ciphertext37 bytesTwo full blocks plus 5 bytes, then 11 zero bytes for GHASH alignment.

Example GHASH input layout

Suppose AAD is 20 bytes and ciphertext is 37 bytes. GHASH arranges them like this:

AAD block 1
16 bytes of AAD
AAD block 2
4 AAD bytes + 12 zero bytes
Cipher block 1
16 bytes of ciphertext
Cipher block 2
16 bytes of ciphertext
Cipher block 3
5 cipher bytes + 11 zero bytes
Length block
len(A) and len(C), in bits

The length block would record:

\[ len(A)=20\times 8=160\text{ bits} \] \[ len(C)=37\times 8=296\text{ bits} \]

Why include the lengths?

The length block prevents ambiguity. Without length information, different combinations of AAD, ciphertext, and padding could potentially produce confusing internal layouts. Lengths pin down exactly how much original AAD and ciphertext were intended.

Friendly explanation

The length block is the receipt at the bottom of the package: “This much visible metadata, this much encrypted cargo.” If someone alters the amount or arrangement, the receipt no longer matches.

GHASH as an accumulator

Internally, GHASH processes the arranged blocks one at a time. It uses the hash subkey \(H\) and Galois-field multiplication to update an accumulator.

Definition: accumulator

An accumulator is a running value that gets updated as each block is processed. GHASH starts from zero and updates the accumulator with every input block.

A simplified structural view is:

\[ Y_0=0 \] \[ Y_i=(Y_{i-1}\oplus X_i)\cdot H \]

Here \(X_i\) is one 128-bit GHASH input block, and multiplication by \(H\) happens in \(GF(2^{128})\). We will study that multiplication later. For now, focus on the flow: combine block, multiply by \(H\), carry the result forward.

From GHASH output to authentication tag

GHASH produces an authentication value, often written here as \(S\). GCM then combines this with an AES-encrypted counter block to make the final tag.

\[ S=GHASH_H(A,C) \]

High-level tag construction:

\[ tag=E_K(J_0)\oplus S \]
Definition: \(J_0\)

\(J_0\) is the starting counter block derived from the nonce. In common 96-bit nonce cases, it is built efficiently from the nonce and a final counter value. For this tutorial, treat \(J_0\) as the special starting counter used to mask the GHASH output into the final tag.

How verification uses the tag

The receiver takes the received nonce, AAD, ciphertext, and tag. It recomputes GHASH and the expected tag. If the expected tag does not match the received tag, the message is rejected.

StepReceiver actionMeaning
1Arrange AAD, ciphertext, zero padding, and length block.Rebuild the GHASH input.
2Run GHASH with \(H\).Recompute \(S\).
3Compute expected tag.Use \(E_K(J_0)\oplus S\).
4Compare tags.Accept if they match, reject if they do not.

Common mistakes

Mistake 1: thinking GHASH authenticates only ciphertext

GHASH processes AAD, ciphertext, and a final length block. AAD matters.

Mistake 2: confusing GHASH zero padding with message padding

GHASH zero padding is internal formatting for authentication blocks. It is not PKCS#7 padding and is not part of the decrypted plaintext.

Mistake 3: measuring lengths in bytes instead of bits in the length block

GCM's final GHASH length block records lengths in bits.

Mistake 4: accepting plaintext when the tag fails

If tag verification fails, reject the message. The tag is the tamper seal.

Why this matters for cryptography

AES-GCM's authentication is not a vague “checksum.” It is a structured mathematical process over precisely arranged data. AAD, ciphertext, padding for alignment, lengths, the hash subkey, and the final AES mask all have jobs. This is where cryptographic engineering starts to look like clockwork with teeth.

  • GHASH authenticates AAD and ciphertext.
  • AAD and ciphertext are padded internally to 128-bit GHASH blocks.
  • The final length block records AAD and ciphertext lengths in bits.
  • GHASH uses an accumulator updated with \(H\).
  • The GHASH output helps form the authentication tag.
  • The receiver recomputes the tag and rejects the message if verification fails.

What to remember
  • GHASH is GCM's authentication mixer.
  • GHASH uses the hash subkey \(H=E_K(0^{128})\).
  • GHASH processes AAD, ciphertext, and a final length block.
  • AAD is authenticated but not encrypted.
  • Ciphertext is authenticated.
  • AAD and ciphertext are internally zero-padded only for GHASH block alignment.
  • GHASH uses 128-bit blocks.
  • The length block records lengths in bits.
  • The GHASH output is combined with \(E_K(J_0)\).
  • The final tag must verify or the message is rejected.
