How to use this workbook

This workbook focuses on GHASH structure, not yet the full field multiplication. Track the order of inputs, how many 16-byte GHASH blocks are needed, how many internal zero bytes are added, and why the final length block matters.

Quick reference

GHASH input order
  1. AAD \(A\)
  2. Zero padding for AAD alignment, if needed
  3. Ciphertext \(C\)
  4. Zero padding for ciphertext alignment, if needed
  5. Length block: \(len(A)\), \(len(C)\)
Core formulas
\[ Y_0=0 \] \[ Y_i=(Y_{i-1}\oplus X_i)\cdot H \] \[ tag=E_K(J_0)\oplus S \]

Part I: Concept checks

I1 sample answer. GHASH zero padding is internal formatting used to align AAD and ciphertext to 128-bit authentication blocks. PKCS#7 padding is message padding used by some encryption modes to fill plaintext blocks. They are not the same.

I2 sample answer. AAD may need to remain visible, such as a header, but it should still be protected from tampering. GHASH includes AAD so changes to it cause tag verification to fail.

I3 sample answer. The receiver recomputes the tag to check whether the received AAD and ciphertext match what was originally protected. If the recomputed tag differs, something is wrong.

I4 sample answer. GHASH processes AAD first, then internal zero padding if needed, then ciphertext, then ciphertext zero padding if needed, and finally a length block containing the bit lengths of AAD and ciphertext. This arrangement lets GCM authenticate both visible metadata and encrypted message bytes.
