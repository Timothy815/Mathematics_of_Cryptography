AES-GCM III: GHASH Multiplication in \(GF(2^{128})\)

How GHASH treats 128-bit blocks as polynomials, multiplies them by the hash subkey, and reduces the result back into the finite field used by GCM.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 32 showed the structure of GHASH inputs. This tutorial explains the arithmetic used inside the GHASH accumulator: multiplication in \(GF(2^{128})\). The goal is not to memorize implementation tricks yet, but to understand what kind of multiplication GHASH is doing.

GHASH multiplication is not ordinary integer multiplication. It is polynomial multiplication over bits, followed by a reduction step that folds the result back into 128 bits. The machine is precise, but once the parts are named, it stops looking like a thundercloud with equations inside.

What you should know first

You should know that a bitstring can represent a polynomial, that addition in \(GF(2)\) is XOR, and that AES-GCM's GHASH accumulator uses the update rule \(Y_i=(Y_{i-1}\oplus X_i)\cdot H\). You should also remember that \(H=E_K(0^{128})\).

The big picture

Intuition first

GHASH multiplication has three layers. First, treat a 128-bit block as a polynomial. Second, multiply polynomials using XOR-style addition. Third, reduce the product so the answer is again a 128-bit field element.

128-bit blocks as polynomials

A GHASH block is 128 bits. Mathematically, GHASH treats that block as a polynomial with bit coefficients. A bit of 1 means “include this power of \(x\).” A bit of 0 means “do not include it.”

What field does GHASH use?

Definition: \(GF(2^{128})\)

\(GF(2^{128})\) is a finite field with \(2^{128}\) possible elements. In GHASH, each element is represented by a 128-bit block interpreted as a polynomial over \(GF(2)\).

The field is defined using the irreducible polynomial:

\[ p(x)=x^{128}+x^7+x^2+x+1 \]

That polynomial is the folding rule. Whenever multiplication creates terms with degree 128 or higher, the result is reduced modulo \(p(x)\) so it fits back into 128 bits.

Addition is XOR

In \(GF(2)\), coefficients are only 0 or 1. Adding coefficients follows this table:

Bit ABit BA + B in \(GF(2)\)A XOR B

Key idea

In GHASH arithmetic, adding polynomial terms means XORing matching coefficients. If the same term appears twice, it cancels out.

Toy polynomial multiplication over bits

Before 128-bit GHASH multiplication, use a tiny example:

\[ A(x)=x+1 \] \[ B(x)=x^2+x \]

Multiply like ordinary polynomials first:

\[ (x+1)(x^2+x)=x^3+x^2+x^2+x \]

The two \(x^2\) terms cancel because addition is XOR:

\[ x^3+x^2+x^2+x=x^3+x \]
Why cancellation happens

In ordinary arithmetic, \(x^2+x^2=2x^2\). In bit arithmetic, \(1+1=0\), so the repeated term disappears.

Why reduction is needed

Multiplying two 128-bit polynomials can produce terms as high as \(x^{254}\). But GHASH field elements must fit in 128 bits, which means the highest allowed degree is 127.

StagePossible degreeFits in 128-bit field element?
Original GHASH blockAt most 127Yes
Raw product of two blocksAt most 254No
Reduced product modulo \(p(x)\)At most 127Yes

Reduction is the step that turns a too-large polynomial product back into a valid 128-bit field element.

The GHASH reduction rule

GHASH uses:

\[ p(x)=x^{128}+x^7+x^2+x+1 \]

Since we reduce modulo \(p(x)\), this polynomial is treated as equivalent to zero:

\[ x^{128}+x^7+x^2+x+1=0 \]

Rearranging in characteristic 2 gives the folding rule:

\[ x^{128}\equiv x^7+x^2+x+1 \]
What this means

Whenever a term reaches \(x^{128}\), it can be folded back into lower-degree terms \(x^7+x^2+x+1\). Higher terms are folded similarly by multiplying this relationship by more powers of \(x\).

Where multiplication appears in GHASH

The GHASH accumulator update from Tutorial 32 was:

\[ Y_0=0 \] \[ Y_i=(Y_{i-1}\oplus X_i)\cdot H \]

Now we can interpret this more clearly:

PartMeaning
\(Y_{i-1}\)The previous running GHASH value.
\(X_i\)The next 128-bit input block.
\(Y_{i-1}\oplus X_i\)Add the new block into the running value using XOR.
\(\cdot H\)Multiply in \(GF(2^{128})\) by the hash subkey.

Each block changes the accumulator, then multiplication by \(H\) spreads the effect through the field structure.

Implementation preview: shift and conditional XOR

Real implementations often compute GHASH multiplication using shifts and conditional XORs rather than writing giant polynomials by hand. The idea is still the same: multiply bit-by-bit, and whenever a shifted value crosses the field boundary, apply the reduction rule.

Mental model

Think of the 128-bit value as moving along a conveyor belt. Shifts move powers of \(x\). When a term would fall off the high end, the reduction polynomial redirects it back into selected lower positions.

Later, a workbook can practice this with tiny 4-bit or 8-bit fields first, then connect it back to GHASH's 128-bit field.

Common mistakes

Mistake 1: treating GHASH multiplication like integer multiplication

GHASH uses polynomial multiplication over bits, not ordinary base-10 or base-2 integer multiplication.

Mistake 2: forgetting that addition is XOR

Matching terms cancel when they appear twice because \(1\oplus1=0\).

Mistake 3: skipping reduction

Raw products can be too large. Reduction modulo \(x^{128}+x^7+x^2+x+1\) brings the result back to 128 bits.

Mistake 4: thinking \(H\) is arbitrary public data

\(H\) is derived from the AES key as \(H=E_K(0^{128})\). It is part of the keyed authentication machinery.

Why this matters for cryptography

GHASH multiplication is the mathematical heart of AES-GCM authentication. It is fast in software and hardware because bit operations naturally support XORs, shifts, and finite-field reduction. This is a beautiful example of cryptography choosing a mathematical world that computers can inhabit efficiently.

  • GHASH treats 128-bit blocks as polynomials.
  • Coefficients are bits from \(GF(2)\).
  • Polynomial addition is XOR.
  • Raw multiplication can produce terms too large for 128 bits.
  • Reduction modulo \(x^{128}+x^7+x^2+x+1\) folds the result back into the field.
  • The accumulator update multiplies by \(H\) after each block is XORed in.
