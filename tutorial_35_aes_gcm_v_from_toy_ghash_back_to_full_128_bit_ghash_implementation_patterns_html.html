Why this tutorial exists

Purpose

Tutorial 34 used a toy 4-bit GHASH so the whole accumulator cycle could be worked by hand. This tutorial returns to real GHASH and explains the implementation patterns that make \(GF(2^{128})\) multiplication practical: XOR, shifting, conditional reduction, and constant-time care.

The full 128-bit version is not a new creature. It is the toy creature grown into armor: same bones, more bits, stricter engineering.

What you should know first

You should know the toy GHASH update rule \(Y_i=(Y_{i-1}\oplus X_i)\cdot H\), the real GHASH field polynomial \(x^{128}+x^7+x^2+x+1\), and the idea that field multiplication means polynomial multiplication followed by reduction.

Pattern 1: bit-by-bit schoolbook multiplication

The simplest conceptual method is bit-by-bit multiplication. For each bit of one operand, decide whether to XOR in a shifted copy of the other operand.

Definition: carry-less multiplication

Carry-less multiplication is polynomial multiplication over bits. Unlike ordinary integer multiplication, it does not carry from one bit position to another. Matching terms combine by XOR.

Conceptual pseudocode:

result = 0
for each bit of A:
    if that bit is 1:
        result = result XOR shifted_copy_of_B
reduce result back to 128 bits
Why this matches the math

Each 1-bit in \(A\) selects one power of \(x\). Multiplying by \(B\) creates a shifted copy of \(B\). XOR combines all selected shifted copies.

Pattern 2: shift and reduce as you go

Instead of producing a giant raw product and reducing afterward, implementations can fold the reduction into the shifting process. Each shift moves the polynomial. If a bit crosses the boundary, the reduction rule is applied immediately.

The real GHASH reduction polynomial is:

\[ p(x)=x^{128}+x^7+x^2+x+1 \] \[ x^{128}\equiv x^7+x^2+x+1 \]

The taps \(x^7,x^2,x,1\) tell the implementation where the overflowed term gets folded back.

Reduction constants

Implementations often encode the folding rule as a constant. Depending on the bit-ordering convention, you may see a constant such as 0xe1 used in GHASH reduction logic.

Important caution about bit order

GHASH specifications and implementations can describe bit order differently. The same field polynomial may appear as different-looking constants depending on whether the algorithm shifts left or right and how the 128-bit value is mapped to polynomial coefficients.

Constant-time care

Some data used in GHASH is related to secret keys, especially \(H\). Implementations should avoid timing behavior that depends on secret values.

Definition: constant-time implementation

A constant-time implementation avoids branches, memory lookups, and timing differences that depend on secret data. The goal is to make side-channel observations less useful to an attacker.

A naive implementation might say:

if secret_bit == 1:
    result = result XOR value

A safer implementation pattern may use masks so the same operations happen either way:

mask = all_ones_if_bit_is_1_else_all_zeroes
result = result XOR (value AND mask)
Beginner warning

Do not write your own production GHASH implementation. The purpose here is understanding. Real systems should use vetted cryptographic libraries.

Hardware acceleration: carry-less multiply

Modern processors often include carry-less multiplication instructions. These can multiply binary polynomials much faster than bit-by-bit loops.

Common mistakes

Mistake 1: treating implementation constants as new math

Constants like 0xe1 are compact encodings of the same field polynomial idea, shaped by bit ordering and shift direction.

Mistake 2: forgetting bit-order conventions

Two implementations can look different while representing the same field arithmetic.

Mistake 3: branching on secret data

Branching or table lookups based on secret values can leak timing information.

Mistake 4: trying to hand-roll production cryptography

Understanding the machinery is excellent. Deploying self-written cryptographic primitives is a different matter. Use vetted libraries for real systems.

Why this matters for cryptography

AES-GCM sits at the meeting point of algebra and engineering. The algebra says “multiply in \(GF(2^{128})\) and reduce by this polynomial.” The engineering says “do that quickly, correctly, and without leaking secrets through timing.” Strong cryptography needs both the theorem-room and the machine-shop.

  • The toy GHASH accumulator scales directly to real GHASH.
  • Carry-less multiplication is polynomial multiplication over bits.
  • Reduction constants encode the field polynomial.
  • Bit-order conventions affect how code looks.
  • Constant-time discipline matters.
  • Production systems should use vetted libraries.
