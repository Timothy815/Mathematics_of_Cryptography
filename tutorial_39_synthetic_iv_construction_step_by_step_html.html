Synthetic IV Construction Step by Step

A guided walkthrough of how AES-GCM-SIV turns nonce, AAD, plaintext, and POLYVAL output into a synthetic IV/tag and then uses it for encryption.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 37 introduced AES-GCM-SIV as a nonce-misuse-resistant AEAD mode. Tutorial 38 introduced POLYVAL as GHASH’s little-endian cousin. This tutorial connects those ideas by walking through the synthetic IV construction step by step at a beginner-friendly level.

The synthetic IV is the hinge of AES-GCM-SIV. It is where authentication-like processing becomes the starting point for encryption. The mode first listens to the message, then chooses how to begin encrypting it.

What you should know first

You should know that AES-GCM-SIV is an AEAD mode, that POLYVAL processes formatted blocks in a finite field, and that a synthetic IV is computed from cryptographic inputs rather than simply supplied from outside. You do not need to implement AES-GCM-SIV yet. This tutorial focuses on the conceptual pipeline.

The big picture

AES-GCM-SIV has a different rhythm from AES-GCM. AES-GCM relies heavily on the externally supplied nonce being unique. AES-GCM-SIV uses the nonce too, but it also derives a synthetic IV from the protected inputs.

ModeHigh-level orderNonce misuse behavior
AES-GCMUse nonce/counter for encryption, then authenticate.Can fail catastrophically if nonce repeats.
AES-GCM-SIVCompute synthetic IV/authentication value first, then encrypt.Designed to fail less catastrophically under nonce reuse.
Memory hook

AES-GCM starts the engine using the nonce. AES-GCM-SIV first builds a custom ignition key from the nonce and message, then starts the engine.

Stage 1: Format the inputs

Authenticated encryption must know exactly what is being protected. AES-GCM-SIV processes associated data, plaintext, and length information in a structured way before producing the synthetic IV.

Definition: associated authenticated data

Associated Authenticated Data, or AAD, is data that is authenticated but not encrypted. It remains visible, but any change to it should be detected.

Examples of AAD might include protocol headers, packet numbers, sender IDs, or metadata that must remain readable but protected against tampering.

AAD

Visible but authenticated.

Plaintext

Authenticated and encrypted.

Lengths

Included so boundaries cannot be confused.

Why lengths matter

Without lengths and formatting, two different arrangements of data could be accidentally interpreted as the same input. Formatting is the label-maker of cryptography.

Stage 2: Run POLYVAL

POLYVAL processes formatted blocks with an accumulator pattern:

\[ S_0=0 \] \[ S_i=(S_{i-1}\oplus X_i)\cdot H \]

Each block affects the running state. The output is not the final ciphertext. It is an authentication-style summary that will help create the synthetic IV.

SymbolRole in this stage
\(S_i\)The running POLYVAL accumulator.
\(X_i\)The current formatted input block.
\(H\)The POLYVAL hash key.
\(\oplus\)XOR, the field addition operation.
Key idea

POLYVAL turns the protected input structure into a compact, keyed mathematical fingerprint. That fingerprint helps determine the synthetic IV.

Stage 3: Mix in the nonce

AES-GCM-SIV still accepts a nonce. The difference is that the nonce is not simply trusted as the direct encryption starting point. Instead, it is mixed into the process that creates the synthetic IV.

Careful simplification

The exact AES-GCM-SIV specification includes precise byte layouts, masking, and AES calls. This tutorial is not yet implementing those details. The beginner-level idea is that the nonce and POLYVAL result are combined before the final synthetic IV/tag value is produced.

InputRole
POLYVAL resultRepresents the protected AAD/plaintext/length structure.
NonceMakes the encryption instance-specific.
AES/keyed processingFinalizes a value only the key-holder can reproduce correctly.

Stage 4: Produce the synthetic IV/tag

The synthetic IV is tag-like: it is derived from the data being protected and is checked during decryption. It is also IV-like: it guides the counter-style encryption step.

Definition: synthetic IV/tag

In AES-GCM-SIV, the synthetic IV is a computed value that both authenticates the protected inputs and provides the starting material for encryption.

It acts like...Because...
A tagIt is checked to verify authenticity.
An IVIt is used to set up counter-style encryption.
A keyed fingerprintIt depends on the key and protected inputs.

Stage 5: Use the synthetic IV for encryption

After the synthetic IV/tag value is produced, AES-GCM-SIV uses counter-style encryption to encrypt the plaintext. The crucial difference from ordinary AES-GCM is that the starting value is derived from the protected inputs, not simply from a raw external nonce.

\[ C=P\oplus S \]

This equation still represents stream/CTR-style encryption: plaintext is XORed with a keystream. But the keystream setup is tied to the synthetic IV, which itself depends on the message and associated data.

Why this helps under nonce reuse

If the same external nonce is reused with different plaintexts, the synthetic IV can still differ because the protected inputs differ. That helps avoid the repeated-keystream disaster that ordinary AES-GCM can suffer.

Decryption: recompute and check

During decryption, the receiver does not simply decrypt and hope. The receiver recomputes the synthetic IV/tag from the received data and checks whether it matches the provided tag. If it does not match, the ciphertext is rejected.

Decryption stepPurpose
Use tag/synthetic IV to decrypt.Recover candidate plaintext.
Recompute POLYVAL over AAD, plaintext, and lengths.Recreate the authentication summary.
Recreate the expected synthetic IV/tag.Check whether the data is authentic.
Reject if mismatch.Prevent tampered data from being accepted.
Security rule

In authenticated encryption, failed verification should mean the message is rejected. Do not use unauthenticated plaintext output from a failed decryption.

Repeated input warning

AES-GCM-SIV is more forgiving under nonce reuse, but it is deterministic for the same key, nonce, plaintext, and AAD. This means repeated identical protected inputs may produce repeated outputs.

What this tutorial is not yet doing

This tutorial is intentionally high-level. It does not yet teach every byte-order detail, key derivation detail, mask bit, counter increment rule, or test vector. Those belong in a later implementation-focused lesson.

The learning sequence

First learn the architecture. Then learn the exact specification. Then study implementation. Trying to do all three at once is how the swamp gets its boots on.

Common mistakes

Mistake 1: thinking the synthetic IV is just random

It is computed from cryptographic inputs. It is not merely sampled randomly.

Mistake 2: thinking AES-GCM-SIV ignores the nonce

The nonce is still part of the construction. It is just not trusted in the same fragile way as ordinary GCM.

Mistake 3: thinking misuse resistance means careless use is fine

Unique nonces are still best practice. Misuse resistance reduces damage from certain mistakes.

Mistake 4: ignoring verification failure

If the tag check fails, reject the message. Do not use failed-decryption plaintext.

  2. Why does AES-GCM-SIV include length information in the authenticated structure?
    Show solution

    Length information prevents ambiguity between different arrangements of AAD and plaintext.

  3. What does POLYVAL contribute to the synthetic IV construction?
    Show solution

    POLYVAL creates a keyed authentication-style summary of the formatted protected inputs.

  4. Why does computing the synthetic IV from protected inputs help under nonce reuse?
    Show solution

    Different protected inputs can produce different synthetic IVs even if the external nonce repeats, reducing repeated-keystream danger.

  5. What should happen if decryption verification fails?

    Show solution

    The repeated output may reveal that the exact same protected input was repeated.

Why this matters for cryptography

AES-GCM-SIV is a lesson in defensive construction. Instead of trusting the nonce alone to set up encryption, it derives a starting value from the actual protected data. This is cryptography as engineering humility: assume the outside world may stumble, and design the inside of the machine to absorb the stumble better.

  • Synthetic IV construction binds encryption setup to the protected data.
  • POLYVAL provides a keyed finite-field summary.
  • The nonce still matters, but it is not used in the same fragile way.
  • The synthetic IV acts like both a tag and an IV.
  • Verification failure means rejection.
  • Misuse resistance reduces damage; it does not erase all risks.
