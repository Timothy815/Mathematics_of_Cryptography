What Is a Cryptographic Hash Function?

The opening lesson of the hash-function arc: fixed-length fingerprints, one-way behavior, collision resistance, and why hashing is not encryption.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Volume 1 focused heavily on reversible transformations: encryption, decryption, modes, authenticated encryption, and finite-field authentication. Volume 2 begins a different kind of machine: the cryptographic hash function. A hash function takes data of almost any size and produces a fixed-length fingerprint. In cryptography, that fingerprint must be hard to reverse, hard to duplicate accidentally, and hard to fake deliberately.

A cryptographic hash is not a secret box. It is more like a mathematical seal press: feed it a scroll, a book, or a library, and it stamps out a fixed-size mark. The magic is that tiny changes should make a wildly different mark, and finding two scrolls with the same mark should be brutally difficult.

What you should know first

You should know what bits and bytes are, what a fixed-length output means, and why authentication matters. You do not need to know SHA-2, SHA-3, Merkle-Damgård, sponge constructions, or HMAC yet. This tutorial builds the front door.

Basic definition

Definition: cryptographic hash function

A cryptographic hash function is a deterministic function that takes an input of arbitrary or variable length and produces a fixed-length output called a digest, while aiming to provide strong one-way and collision-resistant behavior.

\[ h=H(M) \]

Read this as: “The digest \(h\) is produced by applying the hash function \(H\) to the message \(M\).”

Hashes are deterministic

A cryptographic hash function is deterministic. The same input always produces the same digest.

\[ H(M)=h \] \[ H(M)=h \quad \text{again} \]

This is necessary because hashes are often used for checking whether data stayed the same. If the same file gave a different digest every time, it would be useless as a fingerprint.

InputDigest behavior
Same inputSame digest.
Slightly changed inputShould produce a very different-looking digest.

Fixed-length output

A hash function compresses information into a digest of fixed size. A tiny message and a huge file can both produce a digest of the same length.

InputPossible hash functionOutput size
helloSHA-256256 bits
A 500-page documentSHA-256256 bits
A software installerSHA-256256 bits
A disk imageSHA-256256 bits
Important consequence

Since there are more possible inputs than possible fixed-length outputs, collisions must exist. The goal is not to make collisions impossible. The goal is to make them infeasible to find.

Hashing is not encryption

Hashing and encryption are often confused, but they are fundamentally different.

FeatureEncryptionHashing
Main purposeHide information so it can later be recovered by someone with the key.Create a fixed-length fingerprint.
Reversible?Yes, with the right key.No, not by design.
Uses a key?Usually yes.Ordinary hash functions do not use a secret key.
Output sizeUsually related to input size.Fixed length.
ExampleAES encryption.SHA-256 digest.
Common language trap

People sometimes say “encrypt the password” when they mean “hash the password.” In cryptography, these are not the same. Password storage needs special password hashing, not reversible encryption.

Three core security properties

Cryptographic hash functions are usually judged by three major resistance properties.

Preimage resistance

Given a digest, it should be hard to find an input that produces it.

Second preimage resistance

Given one input, it should be hard to find a different input with the same digest.

Collision resistance

It should be hard to find any two different inputs with the same digest.

PropertyAttacker challengeCompact math form
Preimage resistanceGiven \(h\), find \(M\) such that \(H(M)=h\).Find \(M\) from \(h\).
Second preimage resistanceGiven \(M_1\), find \(M_2\ne M_1\) with the same digest.\(H(M_1)=H(M_2)\)
Collision resistanceFind any two different messages with the same digest.\(M_1\ne M_2\), \(H(M_1)=H(M_2)\)

The avalanche effect

A good cryptographic hash should react dramatically to small input changes. Changing one bit of the input should change many bits of the output.

Input 1: The package arrives at noon.
Input 2: The package arrives at moon.

Those messages differ by one letter, but a strong hash function should produce very different-looking digests.

Why avalanche matters

If small changes caused small predictable changes in the digest, attackers could use patterns to manipulate messages or search for collisions more efficiently.

What cryptographic hashes are used for

Hash functions appear all over modern cryptography and security.

What hashes do not do by themselves

A plain hash function is powerful, but it is not a universal security potion.

MisunderstandingCorrection
A hash encrypts data.No. A hash is not designed to be reversed.
A hash proves who created a message.No. A plain hash has no secret key and no identity by itself.
A fast hash is enough for password storage.No. Password storage needs slow, salted password-hashing functions.
A hash prevents tampering by itself.Only if the expected digest is protected or authenticated somehow.

A toy hash warning

Suppose we create a silly toy hash that adds all byte values and keeps only the last 8 bits:

\[ H_{toy}(M)=\left(\sum \text{bytes of } M\right)\bmod 256 \]

This has a fixed-length output, but it is not cryptographically secure. It is easy to find different messages with the same sum. This teaches an important lesson: being a hash function is not the same as being a secure cryptographic hash function.

Important distinction

Checksums and ordinary hashes can detect accidental changes. Cryptographic hashes must also resist deliberate attack.

Common mistakes

Mistake 1: thinking hashing is encryption

Hashing is not reversible. Encryption is designed to be reversible with the right key.

Mistake 2: thinking collisions are impossible

Collisions must exist because the output space is fixed. The security goal is making useful collisions infeasible to find.

Mistake 3: using fast hashes directly for passwords

Passwords need specialized slow password-hashing functions with salts, not plain fast hashes.

Mistake 4: thinking a public digest proves authenticity

Anyone can hash data. Authenticity requires a protected digest, a MAC, or a digital signature.

Practice problems

  1. What does \(h=H(M)\) mean?
    Show solution

    It means the digest \(h\) is produced by applying the hash function \(H\) to message \(M\).

  2. Why is hashing not encryption?
    Show solution

    Encryption is reversible with the right key. Hashing is designed to be one-way and produce a fixed-length digest.

  3. Why must collisions exist for fixed-length hashes?
    Show solution

    There are more possible inputs than fixed-length outputs, so at least two different inputs must map to the same digest.

  4. What is preimage resistance?
    Show solution

    Given a digest \(h\), it should be hard to find an input \(M\) such that \(H(M)=h\).

  5. What is collision resistance?
    Show solution

    It should be hard to find any two different inputs \(M_1\ne M_2\) such that \(H(M_1)=H(M_2)\).

  6. Why is a plain fast hash not enough for password storage?
    Show solution

    Fast hashes allow attackers to test guesses quickly. Password storage needs slow, salted password-hashing functions.

Why this matters for cryptography

Hash functions are everywhere because they give cryptography a way to compress, identify, bind, and check data. They are used in signatures, MACs, password storage, certificate systems, Merkle trees, blockchains, software downloads, and key derivation. They are the fingerprints, rivets, and wax seals of the cryptographic workshop.

  • Hashes produce fixed-length digests.
  • Hashes are deterministic.
  • Cryptographic hashes should be one-way.
  • Cryptographic hashes should resist collisions.
  • Hashing is not encryption.
  • Plain hashes do not prove authenticity by themselves.
  • Hash functions prepare us for MACs, HMAC, Merkle trees, SHA-2, and SHA-3.

One-page recap

What to remember
  • A cryptographic hash function maps data to a fixed-length digest.
  • The basic notation is \(h=H(M)\).
  • The same input always produces the same digest.
  • Small input changes should produce large digest changes.
  • Hashing is not encryption.
  • Encryption is reversible with a key; hashing is one-way by design.
  • Collisions must exist, but they should be hard to find.
  • Preimage resistance means hard to find an input for a chosen digest.
  • Second preimage resistance means hard to find another input matching a given input’s digest.
  • Collision resistance means hard to find any two different inputs with the same digest.
  • Plain fast hashes are not enough for password storage.
  • Hashes are building blocks for MACs, signatures, Merkle trees, and protocols.
