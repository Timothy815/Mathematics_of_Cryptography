Toy Hash Functions: Building and Breaking a Tiny Hash

A hands-on lesson where we build a tiny hash, compute examples, find collisions, and learn why real cryptographic hashes need far more than fixed-length output and scrambled-looking behavior.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

The last three tutorials defined cryptographic hashes, separated the major resistance properties, and explained the birthday bound. This tutorial turns those ideas into a small machine. We will build a toy hash function, compute it by hand, then attack it on purpose.

A toy hash is not secure. That is exactly why it is useful. It is a glass engine: the gears are crude, the casing is cracked, and the flaws are visible. By breaking it, we learn what real hash functions must defend against.

What you should know first

You should know that a hash function maps a message \(M\) to a fixed-length digest \(h\), written \(h=H(M)\). You should also know the difference between preimage resistance, second preimage resistance, and collision resistance, and you should remember that collisions must exist for fixed-length hashes.

Our toy hash function

We will define a deliberately weak 8-bit hash. It processes the message one byte at a time and keeps only an 8-bit state.

Toy hash definition

Start with state \(s=0\). For each byte \(b_i\), update the state by XORing the byte into the state, rotating left by one bit, and adding the byte again modulo 256. The final state is the digest.

\[ s_0=0 \] \[ s_i=\bigl((s_{i-1}\oplus b_i)\lll 1 + b_i\bigr)\bmod 256 \] \[ H_{toy}(M)=s_{\text{final}} \]
Careful note about notation

The expression above is intentionally compact. In practice, read it as: first XOR, then rotate left by 1, then add the byte, then keep only 8 bits.

First lesson: fixed output is easy

Our toy hash can process messages of different lengths, but it always outputs one byte.

MessageOutput sizeComment
A8 bitsOne-byte input, one-byte digest.
AB8 bitsTwo-byte input, one-byte digest.
HELLO8 bitsFive-byte input, one-byte digest.
A whole book8 bitsStill only one-byte digest.
Important lesson

Fixed-length output is not hard to achieve. Secure fixed-length output is the hard part.

Breaking it: output size is tiny

Since the digest is only 8 bits, there are only:

\[ 2^8=256 \]

possible digests. That means collisions are not merely possible; they are easy. If we hash more than 256 different messages, a collision is guaranteed by the pigeonhole principle.

Failure 1

The digest is too short. Even if the mixing were clever, an 8-bit output cannot provide serious collision resistance.

Breaking it: birthday search

By the birthday bound, collisions for an 8-bit hash become likely after roughly:

\[ 2^{8/2}=2^4=16 \]

random messages, not 256. That is the birthday paradox returning with a tiny hammer.

Digest lengthPossible outputsRough collision search scale
8 bits\(2^8=256\)\(2^4=16\)

Breaking it: structure leaks patterns

Our toy hash uses simple operations: XOR, rotate, add. Those are not automatically bad, but in this tiny design they are too simple and too exposed.

WeaknessWhy it matters
Only 8-bit stateThe entire internal state can be searched easily.
No strong compression functionThe byte update is too simple to hide structure.
No padding or length bindingDifferent message layouts may be easier to manipulate.
Linear-ish piecesXOR and rotation preserve patterns unless surrounded by stronger nonlinear structure.
Important distinction

Real hash functions often use XOR, rotations, additions, Boolean functions, substitutions, permutations, and compression structures. The issue is not that our toy uses these ingredients. The issue is that the recipe is tiny, shallow, and easy to exhaustively analyze.

How the toy hash fails the three properties

PropertyWhat the attacker triesWhy the toy hash fails
Preimage resistanceGiven a digest, find an input.Only 256 digests; brute force is easy.
Second preimage resistanceGiven one message, find a different message with the same digest.Small output and simple state make alternatives easy to search for.
Collision resistanceFind any two different messages with the same digest.Birthday search makes collisions appear after roughly 16 random tries.

Avalanche testing

We can also test whether small input changes cause large output changes. For example, compare A and B.

MessageToy digestObservation
A0xC3Computed earlier.
B0xC6Close-looking output, not impressive avalanche.

This is a bad sign. But even if the outputs looked wildly different, that still would not prove security. Avalanche is necessary-ish as a design smell test, but not sufficient as a proof.

What real hash functions add

Real cryptographic hash functions use much larger state, carefully designed compression or permutation structures, many rounds, strong diffusion, and detailed security analysis.

Toy hash weaknessReal hash design response
8-bit digestUse large outputs such as 256 bits or more.
Tiny internal stateUse much larger internal state.
One shallow update per byteUse many rounds of mixing.
Easy structureUse carefully designed nonlinear and diffusive operations.
No formal constructionUse studied structures such as Merkle-Damgård or sponge constructions.

Why build weak things?

Building weak cryptography in a controlled learning environment is extremely useful. It lets us see the cracks. The danger is not in building a toy. The danger is forgetting that the toy is a toy.

Learning rule

Toy cryptography is for understanding, not protection. It belongs in notebooks, classrooms, simulations, and experiments, not production systems.

Common mistakes

Mistake 1: thinking fixed output means secure hash

Fixed output is easy. Secure fixed output requires strong resistance properties.

Mistake 2: thinking avalanche proves security

Avalanche behavior is useful, but it does not prove preimage, second preimage, or collision resistance.

Mistake 3: ignoring output length

A tiny digest makes collision attacks easy no matter how clever the update function looks.

Mistake 4: using toy crypto outside learning

Toy designs are useful microscopes. They are not vault doors.

Practice problems

  1. What is the starting state of our toy hash?
    Show solution

    The starting state is \(s_0=0\), or 0x00.

  2. What are the four operations in one byte update?
    Show solution

    XOR the byte into the state, rotate left by one bit, add the byte, then reduce modulo 256.

  3. Why is an 8-bit digest insecure?
    Show solution

    There are only 256 possible outputs, so brute force and collision search are easy.

  4. Roughly how many random messages are needed before collisions become likely for an 8-bit hash?
    Show solution

    About \(2^{8/2}=2^4=16\) messages.

  5. Why is avalanche not enough to prove security?
    Show solution

    A function can produce different-looking outputs while still having small output space, structural weaknesses, or easy collisions.

  6. Name two things real hash functions add beyond this toy hash.
    Show solution

    Possible answers include larger digest size, larger internal state, many rounds, strong diffusion, nonlinear operations, padding, length binding, and studied constructions.

Why this matters for cryptography

Toy hash functions reveal why real hash functions are carefully engineered. A digest must be long enough, the internal state must be large enough, the update function must diffuse and hide structure, and the overall construction must survive attacks against preimage resistance, second preimage resistance, and collision resistance. This lesson prepares us for compression functions, Merkle-Damgård, padding, length extension, and sponge constructions.

  • Fixed-length output is not enough.
  • Small digests make collisions easy.
  • Birthday attacks punish short outputs.
  • Avalanche is useful but not sufficient.
  • Toy hashes are safe only as learning tools.
  • Real hash functions need strong constructions and detailed analysis.
