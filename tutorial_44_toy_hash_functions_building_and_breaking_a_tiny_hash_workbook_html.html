Printable practice for computing a tiny hash by hand, testing fixed output, identifying weaknesses, applying the birthday bound, and explaining why toy hashes are learning tools rather than security tools.

Directions

How to use this workbook

Work through the toy hash as a visible machine. The goal is not to admire the design, but to learn how fixed-length output, small state, simple structure, birthday search, and avalanche testing relate to real hash security.

Quick reference

Toy hash rule
\[ s_0=0 \] \[ s_i=\bigl((s_{i-1}\oplus b_i)\lll 1+b_i\bigr)\bmod 256 \] \[ H_{toy}(M)=s_{\text{final}} \]
Learning warning

Toy cryptography is for understanding, not protection. A weak machine can teach excellent lessons because its cracks are visible.

Note for self-study

The toy hash is intentionally weak. The point is to make the weaknesses visible: tiny output, tiny state, simple structure, weak avalanche, and easy collisions.

H1 sample answer. With only 256 possible digest values, 257 different messages force at least two messages to land in the same digest slot.

Part I: Breaking it by birthday search

Digest lengthPossible outputsRough collision search scale
8 bits\(2^8=256\)\(2^4=16\)
16 bits\(2^{16}\)\(2^8=256\)
32 bits\(2^{32}\)\(2^{16}\)

I1 sample answer. Birthday search says collisions become likely around the square root of the output space. For 8 bits, \(\sqrt{2^8}=2^4=16\).

Part J: Structural weaknesses

WeaknessWhy it matters
Only 8-bit stateThe entire state can be searched easily.
No strong compression functionThe update is too simple to hide structure.
No padding or length bindingDifferent message layouts may be easier to manipulate.
Linear-ish piecesXOR and rotation can preserve patterns without stronger nonlinear structure.

Part K: How the toy hash fails the three properties

PropertyWhat the attacker triesWhy the toy hash fails
Preimage resistanceGiven a digest, find an input.Only 256 digests; brute force is easy.
Second preimage resistanceGiven one message, find a different message with the same digest.Small output and simple state make alternatives easy to search for.
Collision resistanceFind any two different messages with the same digest.Birthday search makes collisions likely after roughly 16 random tries.

Part L: What real hash functions add

Toy hash weaknessReal hash design response
8-bit digestUse large outputs such as 256 bits or more.
Tiny internal stateUse much larger internal state.
One shallow update per byteUse many rounds of mixing.
Easy structureUse carefully designed nonlinear and diffusive operations.
No formal constructionUse studied structures such as Merkle-Damgård or sponge constructions.

Part N: Concept checks

N1 sample answer. The toy hash is useful because its flaws are visible. It lets us practice computation, collision reasoning, avalanche testing, and attack analysis without being buried inside a full real-world hash function.

N2 sample answer. It fails collision resistance because it has only 256 possible outputs, so collisions are guaranteed after enough messages and become likely after roughly 16 random samples by the birthday bound.

N3 sample answer. Avalanche testing helps reveal whether small input changes spread through the output, but a function can have avalanche-like behavior and still have short outputs, structural weaknesses, or easy collisions.

N4 sample answer. This tutorial built a deliberately weak 8-bit toy hash and computed examples by hand. The toy hash shows that fixed-length output is easy, but security is hard. Because it has only an 8-bit output and tiny state, brute force and birthday collisions are easy. Real hash functions need larger outputs, larger states, many rounds, stronger mixing, padding, length handling, and studied constructions.
