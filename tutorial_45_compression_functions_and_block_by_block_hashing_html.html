Compression Functions and Block-by-Block Hashing

How hash functions process long messages by splitting them into blocks, feeding each block into a compression function, and carrying forward a chaining state until a final digest is produced.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 44 used a tiny toy hash to show that fixed-length output is easy but secure hashing is hard. This tutorial introduces the central structural idea behind many real hash functions: block-by-block processing using a compression function and a chaining value.

A hash function cannot usually swallow an entire book in one bite. Instead, it cuts the message into blocks, feeds each block into a mixing chamber, and carries a state forward. The final state becomes the digest. It is less like one giant crunch and more like a careful mill: block in, state updated, next block in, state updated again.

What you should know first

You should know that a hash function maps a message \(M\) to a fixed-length digest \(h\), written \(h=H(M)\). You should also know that real hash functions need large output sizes, strong diffusion, and resistance to preimage, second preimage, and collision attacks.

The basic problem: messages can be long

Hash functions need to process messages of many different sizes. A message might be one byte, one file, one software installer, or a whole disk image.

Memory hook

Block-by-block hashing is the conveyor belt of hash design. The message moves through in chunks, and the state keeps the memory of what has already passed through.

What is a compression function?

Definition: compression function

A compression function is an internal hash component that takes a previous chaining value and a message block, then produces a new chaining value of fixed size.

\[ CV_i=f(CV_{i-1},M_i) \]

Read this as: “The new chaining value \(CV_i\) is produced by feeding the old chaining value \(CV_{i-1}\) and the current message block \(M_i\) into compression function \(f\).”

Why call it “compression”?

The compression function usually takes more input bits than it outputs. For example, it may take a chaining value plus a message block and produce only one chaining-value-sized output.

Important caution

Compression here does not mean file compression like ZIP. It means reducing a larger internal input into a fixed-size state while trying to preserve cryptographic security properties.

Chaining values: the memory of the hash

The chaining value carries forward information from all earlier blocks. If block 1 changes, then \(CV_1\) changes, which should affect \(CV_2\), \(CV_3\), and the final digest.

\[ CV_0=IV \] \[ CV_1=f(CV_0,M_1) \] \[ CV_2=f(CV_1,M_2) \] \[ CV_3=f(CV_2,M_3) \]
Key idea

The chaining value is the hash function’s running memory. It is not supposed to reveal the message, but it should be affected by all previous blocks.

Splitting a message into blocks

Suppose we use tiny 4-byte blocks for learning. Real hash functions use larger block sizes, but tiny blocks make the idea visible.

Message:  ATTACKATDAWN
Blocks:   ATTA | CKAT | DAWN
Block numberBlock textMeaning
\(M_1\)ATTAFirst block.
\(M_2\)CKATSecond block.
\(M_3\)DAWNThird block.

The hash function processes \(M_1\), then \(M_2\), then \(M_3\), updating the chaining value each time.

A tiny worked example

Let us create a very small educational compression function. This is not secure. It is only for seeing the shape of block-by-block hashing.

Toy compression function

Use a one-byte chaining value and one-byte message block. Define:

\[ f(CV,M)=((CV\oplus M)\lll 1 + M)\bmod 256 \]

In Tutorial 44, we called this “processing bytes.” Now we are learning the more general architecture: old chaining value plus current block goes into a compression function, producing a new chaining value.

The initialization value

The first block needs a previous chaining value. Since no previous block exists yet, the construction starts from an initialization value, usually written \(IV\).

\[ CV_0=IV \]

In real hash functions, the initialization value is normally a fixed public constant chosen by the specification. It is not a secret key.

TermRoleSecret?
\(IV\)Starting chaining value.No, usually public and fixed.
\(CV_i\)Running chaining value after block \(i\).No, internal but not secret in ordinary hashing.

Padding preview

What if the message does not divide evenly into blocks? The hash function must pad the final block so it fits the required block size.

4-byte blocks:
Message:  ATTACK!
Blocks:   ATTA | CK!?

The last block is short, so padding must fill it.

Padding seems like a boring detail, but it is not. Bad padding or missing length information can create serious structural problems. We will study padding and length encoding more carefully soon.

Preview

Padding and length encoding help prevent different messages from being interpreted as the same block sequence.

Why block order matters

Because each chaining value feeds into the next compression step, changing an early block should affect every later state.

ChangeExpected effect
Change \(M_1\)Changes \(CV_1\), then \(CV_2\), then the final digest.
Change \(M_2\)Leaves \(CV_1\) the same but changes \(CV_2\) and the final digest.
Reorder blocksShould produce a different digest.
Append a blockShould produce a different final digest.
Chain reaction

Good block-by-block hashing should behave like a trail of falling gears: touch an early gear, and the motion changes downstream.

Compression is not automatically secure

A compression function is a structure, not a guarantee. A weak compression function creates a weak hash, even if the block-by-block architecture looks professional.

RequirementWhy it matters
Strong diffusionChanges should spread through the state.
NonlinearityAttackers should not exploit simple linear patterns.

Careful padding and length handlingMessage boundaries must be unambiguous.

Design warning

“It processes blocks” does not mean “it is secure.” A bad compression function wrapped in a neat loop is still bad cryptography wearing a tie.

Bridge to Merkle-Damgård

One famous way to build a hash function from a compression function is called the Merkle-Damgård construction. The basic pattern is exactly what we have studied here:

\[ CV_0=IV \] \[ CV_i=f(CV_{i-1},M_i) \] \[ H(M)=CV_t \]

Later, we will add the important details: padding, length encoding, security assumptions, and length extension attacks.

Common mistakes

Mistake 1: thinking compression means ZIP-style compression

In hash design, compression means combining a chaining value and message block into a fixed-size new chaining value.

Mistake 2: thinking the IV is secret

In ordinary hash functions, the initialization value is usually a public fixed constant.

Mistake 3: ignoring padding

Padding and length encoding are not decorative. They help define exactly what message was hashed.

Mistake 4: trusting a weak compression function

A block-by-block structure cannot rescue a badly designed compression function.

Practice problems

  1. What does a compression function take as input?
    Show solution

    It takes the previous chaining value and the current message block.

  2. What does a compression function output?
    Show solution

    It outputs a new chaining value.

  3. What does \(CV_i=f(CV_{i-1},M_i)\) mean?
    Show solution

    The new chaining value \(CV_i\) is produced by applying compression function \(f\) to the previous chaining value \(CV_{i-1}\) and current message block \(M_i\).

  4. Why is an initialization value needed?
    Show solution

    The first block needs a previous chaining value, so the construction starts from an initialization value \(IV\).

  5. Why is padding needed?
    Show solution

    Padding is needed when the message does not divide evenly into fixed-size blocks.

  6. Why does a compression function need more than a professional-looking loop around it?
    Show solution

    It must provide strong diffusion, resist structural attacks, use sufficient state/output size, and be part of a carefully studied construction.

Why this matters for cryptography

Compression functions are one of the great architectural ideas in hash design. They let a hash function process long messages with a repeated internal operation. This leads directly to Merkle-Damgård, SHA-1, SHA-2-style thinking, padding rules, length encoding, length extension attacks, and eventually the contrast with sponge constructions such as SHA-3.

  • Long messages are split into fixed-size blocks.
  • Each block is processed with a chaining value.
  • The compression function updates the chaining value.
  • The final chaining value becomes, or helps produce, the digest.
  • The IV starts the chain.
  • Padding and length encoding matter.
  • A secure hash needs a strong compression function and a sound construction.
