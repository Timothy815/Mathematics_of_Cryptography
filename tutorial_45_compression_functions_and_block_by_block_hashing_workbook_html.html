How to use this workbook

Focus on the structure: a message is split into blocks, each block is combined with a chaining value, and the compression function creates the next chaining value. Keep asking, “What is the current block? What is the previous state? What becomes the new state?”

Quick reference

Core pattern
\[ CV_0=IV \] \[ CV_i=f(CV_{i-1},M_i) \] \[ H(M)=CV_t \]
Memory phrase
Old state + current block → compression function → new state

The chaining value is the running memory of the hash.

Note for self-study

The big idea is that an iterated hash carries a chaining value forward. Each block changes the state, and the final state becomes the digest or helps produce it.

Part B: Compression function definition

B1 sample answer. A compression function combines a previous chaining value and a current message block to produce a new fixed-size chaining value.

B2 sample answer. \(CV_i=f(CV_{i-1},M_i)\) means that the new chaining value is made by applying compression function \(f\) to the previous chaining value and the current message block.

Input / outputRole
\(CV_{i-1}\)Previous chaining value.
\(M_i\)Current message block.
\(f\)Compression function.
\(CV_i\)New chaining value.

Part C: Why block-by-block hashing exists

InputProblemHashing strategy
HiVery small.Pad and process as one or more blocks.
A paragraphMedium length.Split into fixed-size blocks.
A 100 MB fileToo large to treat as one simple input.Process block by block.

C1 sample answer. Block-by-block hashing lets the same repeated compression function process messages of many different sizes without needing to handle the entire message at once.

Part E: Chaining values

StepFormulaMeaning
Start\(CV_0=IV\)Start from the initialization value.
Block 1\(CV_1=f(CV_0,M_1)\)Process the first block.
Block 2\(CV_2=f(CV_1,M_2)\)Process the second block using the previous state.
Block 3\(CV_3=f(CV_2,M_3)\)Process the third block using the previous state.
Final\(H(M)=CV_3\)The final chaining value becomes the digest in this simplified model.

E1 sample answer. The chaining value is the running memory because it carries the effect of all previously processed blocks into the next compression step.

Part F: Initialization value

TermRoleSecret?
\(IV\)Starting chaining value.No, usually public and fixed.

F1 sample answer. The first block needs a previous chaining value, but no previous block exists yet. The initialization value provides the starting state.

Part I: Why block order matters

ChangeExpected effect
Change \(M_1\)Changes \(CV_1\), then \(CV_2\), then the final digest.
Change \(M_2\)Leaves \(CV_1\) the same but changes \(CV_2\) and the final digest.
Reorder blocksShould produce a different digest.
Append a blockShould produce a different final digest.

I1 sample answer. Changing an early block changes the chaining value produced from that block. Since that chaining value becomes input to the next block, the change should spread downstream.

Part J: Compression is not automatically secure

RequirementWhy it matters
Strong diffusionChanges should spread through the state.
NonlinearityAttackers should not exploit simple linear patterns.
Large state and outputSmall states invite brute force and collisions.
Careful padding and length handlingMessage boundaries must be unambiguous.
Studied constructionThe design should have public cryptanalytic scrutiny.

J1 sample answer. A loop only repeats the weakness if the compression function is weak. Strong hash design requires a strong internal function and a sound construction.

Part K: Bridge to Merkle-Damgård

LineFormulaMeaning
Start\(CV_0=IV\)Start from the initialization value.
Iterate\(CV_i=f(CV_{i-1},M_i)\)Process each message block with the previous chaining value.
Finish\(H(M)=CV_t\)The final chaining value becomes the digest in this simplified model.

K1 sample answer. Merkle-Damgård is built around the same repeated compression pattern: start with an IV, process each block with a compression function, and output the final chaining value after proper padding and length handling.

Part M: Concept checks

M1 sample answer. Block-by-block hashing splits a message into fixed-size blocks. The construction starts with an initialization value, combines each block with the previous chaining value using a compression function, and carries the resulting chaining value forward until the final digest is produced.

M2 sample answer. The chaining value is the running state of the hash. It carries the effect of previously processed blocks into the next compression step.

M3 sample answer. Changing \(M_1\) should change \(CV_1\). Since \(CV_1\) is input to the next compression step, the change should continue forward and alter the final digest.

M4 sample answer. This tutorial explained how compression functions allow hash functions to process long messages block by block. Each block \(M_i\) is combined with the previous chaining value \(CV_{i-1}\) using a compression function \(f\), producing a new chaining value \(CV_i\). The process begins with an IV and continues until the final digest is produced. Padding, length handling, diffusion, nonlinearity, and strong construction design are essential for security.
