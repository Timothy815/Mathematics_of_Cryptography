Padding, Length Encoding, and Length Extension Attacks

Why Merkle-Damgård padding includes message length, how the final digest can behave like a resumable internal state, and why naive constructions such as Hash(secret || message) can be dangerous.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 46 introduced Merkle-Damgård construction. This tutorial zooms in on two details that are easy to underestimate: padding and length encoding. Then it shows one of the most important consequences of the Merkle-Damgård chaining structure: length extension attacks.

Length extension attacks feel strange at first because the attacker does not “reverse the hash.” They do something subtler: they continue the hash from the visible digest as though it were an internal chaining value. The locked door is not opened; the hallway is extended from the last visible room.

What you should know first

You should know that Merkle-Damgård-style hashes process padded message blocks using a chaining value:

\[ CV_0=IV \] \[ CV_i=f(CV_{i-1},M_i) \] \[ H(M)=CV_t \]

You should also know that the final digest is often the final chaining value, or closely related to it, in the basic construction.

Padding review

Merkle-Damgård-style padding often follows this conceptual shape:

\[ pad(M)=M\,\|\,1\,\|\,0\ldots0\,\|\,\text{len}(M) \]
PiecePurpose
\(M\)The original message.
\(1\)Marks where padding begins.
\(0\ldots0\)Fills enough space to align the final length field.
\(\text{len}(M)\)Records the original message length.
Memory hook

Padding is not packing peanuts. It is part of the machine-readable grammar of the message.

Why length encoding matters

Length encoding helps bind the padded block sequence to the original message length. Without length information, different messages may be easier to interpret ambiguously after padding.

Padding featureWhat it helps prevent
End markerConfusion about where the original message stops.
ZerosMisalignment with required block size.
Original lengthAmbiguity about which message produced the padded block sequence.
Precise ruleDifferent implementations disagreeing about the exact input.
Important distinction

Length encoding helps support the Merkle-Damgård collision-resistance argument. It does not, by itself, prevent length extension attacks in naive keyed uses.

The final digest as a visible state

In a basic Merkle-Damgård construction, the final digest is the final chaining value:

\[ H(M)=CV_t \]

That means the public digest may reveal enough information to continue the compression process from that point, if the attacker also knows or can guess the length of the original hidden prefix.

Central length-extension insight

If the digest acts like the final chaining value, an attacker may be able to continue hashing extra blocks from that digest without knowing the original message contents.

What is a length extension attack?

Definition: length extension attack

A length extension attack is an attack against certain Merkle-Damgård-style hashes where, given a hash of an unknown message and the length of that message or a good guess of it, an attacker can compute the hash of the unknown message plus valid padding plus attacker-chosen extra data.

\[ \text{Given } H(M), \text{ compute } H(M\,\|\,pad(M)\,\|\,X) \]

Here \(X\) is attacker-chosen extension data. The attacker does not need to know \(M\) itself if they can continue from the digest state and correctly account for padding length.

The classic danger: naive secret-prefix MAC

Suppose someone tries to authenticate a message like this:

\[ tag=H(secret\,\|\,message) \]

This is called a secret-prefix construction. With Merkle-Damgård-style hashes, it can be vulnerable to length extension. The attacker may not know the secret, but if they know or can guess the secret length, they may be able to compute a valid tag for:

\[ secret\,\|\,message\,\|\,pad(secret\,\|\,message)\,\|\,extension \]
Security warning

Do not use Hash(secret || message) as a home-made MAC with Merkle-Damgård-style hashes. Use a standard construction such as HMAC.

What the attacker does not get

A length extension attack is often misunderstood. The attacker does not recover the secret and does not reverse the hash.

MisunderstandingCorrection
The attacker finds the secret.No. The secret remains unknown.
The attacker reverses the hash.No. The attacker continues the hash state forward.
The attacker removes padding.No. The padding becomes part of the forged longer message.
The attacker breaks all uses of hashes.No. The issue affects certain constructions and misuse patterns.
Memory hook

Length extension is not rewinding the film. It is splicing on another scene after the visible final frame.

Why padding appears inside the forged message

In a length extension attack, the forged message often contains the original message, then padding bytes, then attacker-chosen data. This looks strange, but it follows directly from the hash construction.

Original authenticated data:
message

What attacker forges conceptually:
message || glue_padding || extension

The “glue padding” is the padding that the original hash would have added after the hidden prefix and message. The attacker includes it so the hash continuation lines up correctly.

Why “glue”?

It glues the original hashed content to the attacker’s extension while preserving the internal block boundary expected by the Merkle-Damgård process.

Safer constructions

The usual answer is not “never use hashes.” The answer is “use the right construction for the job.”

NeedUseWhy
Plain integrity fingerprintHash functionDetects changes when the expected digest is trusted.

Key lesson

A hash function is a building block. Authentication requires an authentication construction, not just a raw hash with a secret pasted beside it.

HMAC preview

HMAC stands for Hash-based Message Authentication Code. It is designed to avoid the naive secret-prefix problem by using the key in a more careful two-layer structure.

\[ HMAC(K,M)=H((K'\oplus opad)\,\|\,H((K'\oplus ipad)\,\|\,M)) \]

Do not worry if that formula looks crowded. The important idea is that HMAC does not simply compute \(H(K\,\|\,M)\). It wraps the message in an inner hash and an outer hash using derived key pads.

Common mistakes

Mistake 1: thinking padding is irrelevant

Padding controls the exact block sequence processed by the hash. It is part of the cryptographic structure.

Mistake 2: thinking length extension reverses the hash

It does not reverse the hash. It continues the hash from a known digest state.

Mistake 3: using Hash(secret || message) as a MAC

This can be vulnerable with Merkle-Damgård-style hashes. Use HMAC instead.

Mistake 4: thinking all hash uses are broken

Length extension is about specific constructions and misuse patterns, not every use of every hash.

Practice problems

  1. What are the major pieces of Merkle-Damgård-style padding?
    Show solution

    The original message, a 1 bit marker, enough 0 bits to align the final length field, and an encoding of the original message length.

  2. Why is the original message length encoded?
    Show solution

    It helps bind the padded block sequence to the original message length and avoids ambiguity.

  3. What is the central insight behind length extension attacks?
    Show solution

    The digest can act like a final chaining value, allowing an attacker to continue hashing extra blocks from that state.

  4. Does length extension reveal the secret in \(H(secret\,\|\,message)\)?
    Show solution

    No. The attacker does not learn the secret. They continue the hash state forward to compute a valid tag for a longer message.

  5. Why does glue padding appear in the forged message?
    Show solution

    It represents the padding that the original hash computation would have inserted before the attacker’s extension, so the continuation lines up with the block structure.

  6. What should be used instead of a naive secret-prefix hash MAC?
    Show solution

    Use HMAC or another standard message authentication code construction.

Why this matters for cryptography

Length extension attacks are a perfect example of why cryptographic primitives must be used in the right construction. A hash function may be excellent at producing digests and still be unsafe when used in a naive homemade authentication scheme. This lesson is the bridge from hash functions to MACs and prepares us directly for HMAC.

  • Padding defines the exact block sequence.
  • Length encoding binds padding to the original message length.
  • Merkle-Damgård digests can expose a resumable final state.
  • Length extension continues the hash; it does not reverse it.
  • Naive Hash(secret || message) constructions can be dangerous.
  • HMAC is the standard safe direction for keyed hashing.

One-page recap

What to remember
  • Merkle-Damgård padding commonly has the shape \(M\,\|\,1\,\|\,0\ldots0\,\|\,\text{len}(M)\).
  • The 1 bit marks where padding begins.
  • The 0 bits fill space before the length field.
  • The length field records the original message length.
  • Padding is part of the cryptographic grammar, not decoration.
  • In basic Merkle-Damgård, the digest is the final chaining value.
  • That final digest may let an attacker continue hashing additional blocks.
  • Length extension computes a digest for \(M\,\|\,pad(M)\,\|\,X\) from \(H(M)\), given the needed length information.
  • Length extension does not recover the secret and does not reverse the hash.
  • Naive \(H(secret\,\|\,message)\) authentication can be vulnerable.
  • Use HMAC for keyed hash-based authentication.
