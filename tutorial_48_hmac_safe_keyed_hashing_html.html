HMAC: Safe Keyed Hashing

How HMAC turns a hash function into a Message Authentication Code by using a secret key in a careful inner-and-outer structure, avoiding the naive length-extension trap.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 47 showed why a naive construction such as Hash(secret || message) can be dangerous with Merkle-Damgård-style hashes. This tutorial introduces HMAC, the standard construction that uses a hash function safely for keyed message authentication.

HMAC is not just “hash plus secret.” It is a carefully folded envelope: the message is sealed once with an inner keyed pad, then the result is sealed again with an outer keyed pad. The structure matters. The folds keep the length-extension goblin outside the door.

What you should know first

You should know that a plain hash provides an integrity fingerprint, while a Message Authentication Code, or MAC, provides integrity plus authenticity using a secret key. You should also know that length extension can make naive Hash(secret || message) constructions unsafe with Merkle-Damgård-style hashes.

MAC review

Definition: Message Authentication Code

A Message Authentication Code, or MAC, is a keyed value computed from a message. Someone who knows the secret key can verify that the message was created or approved by someone else who knows the same key, and that the message has not been changed.

ToolUses secret key?Main purpose
Plain hashNoIntegrity fingerprint when digest is trusted.
MACYesIntegrity plus authenticity.
HMACYesMAC built safely from a hash function.

The unsafe temptation

The tempting but unsafe idea is:

\[ tag=H(K\,\|\,M) \]

This looks reasonable at first: only someone with \(K\) should be able to compute the tag. But with Merkle-Damgård-style hashes, this can allow length extension if an attacker sees the tag and can guess the key length.

Problem

The tag can act like the final chaining value after processing \(K\,\|\,M\). That may allow an attacker to continue hashing extra data without knowing \(K\).

HMAC definition

Definition: HMAC

HMAC, or Hash-based Message Authentication Code, is a standard construction that uses a hash function and a secret key to produce a secure authentication tag.

\[ HMAC(K,M)=H((K'\oplus opad)\,\|\,H((K'\oplus ipad)\,\|\,M)) \]

This formula has two hash layers. First, an inner keyed hash processes the message. Then an outer keyed hash processes the inner result.

Plain-language version

HMAC says: mix the key with an inner pad, hash that with the message, then mix the key with an outer pad and hash the inner hash result.

What is \(K'\)?

HMAC needs the key to fit the hash function’s block size. So the original key \(K\) is first turned into a block-sized key \(K'\).

Key caseWhat HMAC does
Key is longer than block sizeHash the key first, then pad as needed.
Key is shorter than block sizePad it with zeros to the block size.
Key is exactly block sizeUse it as the block-sized key.
Why block size?

The pads ipad and opad are block-sized patterns. To XOR the key with those patterns, the key also needs a block-sized representation.

Why inner and outer pads?

The pads create two related but distinct keyed values. The inner keying material is different from the outer keying material.

Inner keyed prefix

\[ K'\oplus ipad \]

Used with the message in the inner hash.

Outer keyed prefix

\[ K'\oplus opad \]

Used with the inner digest in the outer hash.

This separation is part of what makes HMAC robust. The attacker does not get a raw digest that can simply be used as the final tag after a secret-prefix hash.

Why HMAC avoids the naive length-extension problem

In the unsafe construction \(H(K\,\|\,M)\), the visible tag may be the final state after hashing secret-plus-message. Length extension can continue from that state.

In HMAC, the visible tag is the output of the outer hash:

\[ H((K'\oplus opad)\,\|\,\text{inner digest}) \]

Even if someone imagines extending the outer hash, they would only extend the outer message, not the original protected message \(M\). The message itself was already compressed into the inner digest, and the attacker cannot recompute a valid new inner digest for \(M\,\|\,X\) without the inner keyed prefix.

\(H(K\,\|\,M)\)DangerousVisible tag may let attacker continue from secret-prefix state.
HMACDesigned to avoid this problemUses separate inner and outer keyed hashes.

Important precision

HMAC authenticates messages between parties who share a secret key. It does not provide public verifiability like a digital signature.

HMAC versus digital signatures

HMAC and digital signatures both help protect message authenticity, but they solve different problems.

FeatureHMACDigital signature

How verification works

To verify an HMAC tag, the receiver does not decrypt anything. They recompute the HMAC using the shared key and compare the result to the received tag.

StepVerifier action
1Receive message \(M\) and tag.
2Use shared key \(K\) to recompute \(HMAC(K,M)\).
3Compare recomputed tag with received tag.
4Accept only if they match.
Implementation note

Real systems should compare tags using constant-time comparison to avoid timing leaks. That is an implementation-security detail we will revisit later.

Toy HMAC-style example

Suppose Alice and Bob share a secret key. Alice sends a message and an HMAC tag.

Shared key:  K
Message:     transfer=100
Tag:         HMAC(K, "transfer=100")

If an attacker changes the message to transfer=900, the old tag should not verify, because the attacker cannot compute a valid HMAC tag without the key.

MessageTag available?Verification result
transfer=100Correct tagAccept
transfer=900Old tag from previous messageReject

Common mistakes

Mistake 1: thinking HMAC is just \(H(K\,\|\,M)\)

HMAC uses inner and outer keyed hash layers. That structure is the point.

Mistake 2: thinking HMAC encrypts the message

HMAC authenticates the message. It does not hide the message contents.

Mistake 3: thinking anyone can verify an HMAC

Verification requires the shared secret key. Digital signatures are for public verification.

Mistake 4: reusing weak keys

HMAC relies on a strong secret key and safe key management.

Why this matters for cryptography

HMAC is one of the most practically important constructions in applied cryptography. It appears in network protocols, APIs, key derivation designs, authentication systems, and secure messaging patterns. It also teaches a larger lesson: cryptographic primitives are rarely used raw. Safe systems use carefully designed constructions.

  • Plain hashes provide fingerprints, not secret-key authentication.
  • Naive secret-prefix hashes can be vulnerable to length extension.
  • HMAC uses inner and outer keyed hash layers.
  • HMAC provides message integrity and authenticity for shared-key settings.
  • HMAC does not encrypt and does not provide public verification.
  • Safe cryptography depends on constructions, not ingredients alone.
