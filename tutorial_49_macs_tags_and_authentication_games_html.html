MACs, Tags, and Authentication Games

How to think about message authentication formally: keys, tags, verification, forgery, attacker queries, and the game-based idea behind “secure against forgery.”

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 48 introduced HMAC as a safe keyed-hash construction. This tutorial steps back and asks a deeper question: what does it mean for any MAC to be secure? To answer that, we use a game between an attacker and a verifier.

Cryptography often defines security through games. Not games for amusement, exactly, but ritualized contests: the attacker gets certain powers, the defender sets rules, and security means the attacker still cannot win except with tiny probability. A MAC security game is the courtroom drama of message authentication.

What you should know first

You should know that a MAC uses a secret key to produce an authentication tag for a message. You should also know that HMAC is a standard MAC construction built from a hash function and a secret key.

Correctness: honest tags should verify

Definition: correctness

A MAC is correct if a tag honestly generated for a message under key \(K\) verifies successfully under the same key.

\[ Vrfy_K(M,Mac_K(M))=accept \]

This sounds obvious, but it matters. Before a MAC can be secure, it must at least work for honest users.

Integrity and authenticity

PropertyQuestion it answersMAC role
IntegrityWas the message changed?A changed message should fail verification with the old tag.
AuthenticityWas this approved by someone with the key?Only key holders should be able to make valid tags.
Memory hook

Integrity says “the wax seal is not broken.” Authenticity says “the seal came from someone who had the signet ring.”

What counts as a forgery?

A forgery is not merely making up a tag. A forgery is making a new message-tag pair that the verifier accepts.

Definition: forgery

A MAC forgery is a pair \((M^*,T^*)\) such that \(Vrfy_K(M^*,T^*)=accept\), where \(M^*\) is a message the attacker was not already given a legitimate tag for.

\[ Vrfy_K(M^*,T^*)=accept \]

The star symbols \(M^*\) and \(T^*\) are often used to mean the attacker’s attempted forged message and forged tag.

What does not count as a meaningful forgery?

AttemptForgery?Why?
Replay an old valid \((M,T)\)Usually not a new MAC forgeryThe attacker did not create a valid tag for a new message.
Guess random \(T\) for new \(M\), verifier rejectsNoThe pair is not accepted.
Ask for tag on \(M\), then submit same \((M,T)\)NoThe attacker already obtained that exact tag legitimately from the oracle.
Create valid tag for new \(M^*\)YesA new message-tag pair verifies.
Replay attacks are still real

Replaying an old valid message may still be dangerous in real systems. MAC security and replay protection are related but different. Systems often add nonces, timestamps, sequence numbers, or session context to prevent replay.

The authentication game

In cryptography, we often define MAC security with a game. The attacker gets to ask for tags on messages of their choice, then tries to forge a valid tag for a new message.

RoundWhat happens
1Challenger secretly chooses key \(K\).
2Attacker can query a tagging oracle for tags on messages \(M_1,M_2,\ldots\).
3Oracle returns valid tags \(T_i=Mac_K(M_i)\).
4Attacker outputs a candidate forgery \((M^*,T^*)\).
5Attacker wins if \(M^*\) was not previously queried and \(Vrfy_K(M^*,T^*)=accept\).
Key idea

The attacker is allowed to see valid tags for chosen messages. Security means they still cannot make a valid tag for a new message.

Chosen-message attacks

The authentication game gives the attacker a strong ability: they can choose messages and receive tags. This is called a chosen-message attack model.

Attack modelAttacker abilityWhy it matters
Passive observationSees some message-tag pairs.Weaker attacker model.
Chosen-message attackCan request tags for messages they choose.Stronger and more realistic for many APIs/protocols.
Forgery attemptOutputs new \((M^*,T^*)\).Tests whether tags can be created without the key.

This is a demanding standard. A good MAC should remain secure even after the attacker has collected many legitimate tags for messages of their choice.

Unforgeability under chosen-message attack

The standard phrase is often abbreviated as UF-CMA, meaning UnForgeability under Chosen-Message Attack.

Definition: UF-CMA security

A MAC is UF-CMA secure if even an attacker who can obtain valid tags for messages of their choice cannot produce a valid tag for a new message, except with negligible probability.

PieceMeaning
UFUnforgeability: attacker should not create a valid new tag.
CMAChosen-message attack: attacker may request tags for chosen messages.
Negligible probabilitySo tiny that it is considered infeasible at the security level.

Tag length and guessing

If a tag is \(n\) bits long and behaves unpredictably, a random tag guess succeeds with probability about:

\[ \frac{1}{2^n} \]

That is why tag length matters. Short tags can be guessed more easily, especially if an attacker gets many attempts.

Tag lengthRandom forgery probability per tryComment
32 bits\(1/2^{32}\)Often too short for many modern settings.
64 bits\(1/2^{64}\)Better, but context matters.
128 bits\(1/2^{128}\)Strong generic forgery resistance.
256 bits\(1/2^{256}\)Extremely large margin for random guessing.

Many attempts change the risk

If an attacker gets \(q\) independent forgery attempts against an \(n\)-bit tag, a rough upper estimate for random guessing success is:

\[ \frac{q}{2^n} \]

This is why systems often rate-limit failed verification attempts and choose tag lengths large enough for the expected threat model.

Approximation note

The expression \(q/2^n\) is a simple approximation when \(q\) is much smaller than \(2^n\). It is enough for intuition here.

Tag truncation

Some systems use shortened MAC tags to save space. This is called tag truncation. It can be safe when done according to a standard and with enough bits, but truncation reduces guessing resistance.

ActionEffect
Use full HMAC tagMaximum tag length from the chosen hash output.
Truncate to 128 bitsOften still strong in many settings.
Truncate to 32 bitsMay be risky if many attempts are possible.
Truncate without analysisBad engineering habit.
Design warning

Short tags are tiny doors. If an attacker can knock many times, eventually the odds stop feeling tiny.

MACs do not encrypt

A MAC authenticates a message. It does not hide the message.

GoalToolResult
ConfidentialityEncryptionHides message contents.
Integrity and authenticityMACDetects tampering and verifies key-holder approval.
BothAuthenticated encryptionEncrypts and authenticates together.

MACs do not automatically stop replay

If an attacker records a valid message-tag pair and sends it again later, the MAC may still verify. That is a replay attack, not necessarily a MAC forgery.

Replay defensePurpose
NonceUnique value used once.
TimestampLimits how long a message is valid.
Sequence numberEnsures messages arrive in expected order.
Session bindingTies message to a specific protocol session.
Memory hook

A MAC can prove the ticket is genuine. Replay protection checks whether that ticket has already been used.

Common mistakes

Mistake 1: thinking a MAC encrypts

A MAC authenticates. It does not hide the message.

Mistake 2: thinking replay is the same as forgery

Replay can be dangerous, but it is not the same as creating a valid tag for a new message.

Mistake 3: allowing short tags without analysis

Shorter tags mean easier random guessing, especially across many attempts.

Mistake 4: forgetting chosen-message attacks

A strong MAC should resist attackers who can obtain tags for messages they choose.

Practice problems

  1. What are the three conceptual parts of a MAC system?
    Show solution

    Key generation, tagging, and verification.

  2. What does \(T=Mac_K(M)\) mean?
    Show solution

    It means the tag \(T\) is generated for message \(M\) using secret key \(K\).

  3. What counts as a MAC forgery?
    Show solution

    A new message-tag pair \((M^*,T^*)\) that verifies successfully, where the attacker did not already obtain a legitimate tag for \(M^*\).

  4. What does UF-CMA mean?
    Show solution

    Unforgeability under Chosen-Message Attack.

  5. What is the random guessing probability for one attempt against a 128-bit tag?
    Show solution

    About \(1/2^{128}\).

  6. Why do MACs not automatically stop replay attacks?
    Show solution

    A previously valid message-tag pair may still verify unless the system includes freshness controls such as nonces, timestamps, or sequence numbers.

Why this matters for cryptography

MACs are where hash mathematics meets protocol discipline. A secure MAC does not merely produce complicated-looking tags. It must resist forgery even when attackers can request tags for chosen messages. This game-based way of thinking will appear again in encryption, signatures, key exchange, and modern proofs of security.

  • A MAC has key generation, tagging, and verification.
  • Correct tags should verify.
  • Forgery means producing a valid tag for a new message.
  • UF-CMA is the standard security idea for MACs.
  • Tag length affects guessing resistance.
  • MACs authenticate but do not encrypt.
  • Replay protection requires freshness mechanisms.

One-page recap

What to remember
  • A MAC provides message integrity and authenticity using a secret key.
  • A tag is the MAC output attached to a message.
  • Tagging is written \(T=Mac_K(M)\).
  • Verification is written \(Vrfy_K(M,T)\).
  • Correctness means honest tags verify: \(Vrfy_K(M,Mac_K(M))=accept\).
  • A forgery is a new valid message-tag pair.
  • Replay of an old valid pair can be dangerous, but it is not the same as creating a new forgery.
  • The authentication game lets the attacker request tags, then try to forge one new valid pair.
  • UF-CMA means Unforgeability under Chosen-Message Attack.
  • An \(n\)-bit unpredictable tag has random guessing probability about \(1/2^n\) per try.
  • With \(q\) attempts, rough guessing risk is about \(q/2^n\).
  • MACs do not encrypt and do not automatically prevent replay.
