How to use this workbook

Think like a security game designer. A MAC is not secure merely because its tags look random. It is secure when an attacker with realistic powers still cannot create a valid tag for a new message.

Quick reference

Core MAC notation
\[ T=Mac_K(M) \] \[ Vrfy_K(M,T) \in \{accept,reject\} \] \[ Vrfy_K(M,Mac_K(M))=accept \]
Forgery and guessing
\[ Vrfy_K(M^*,T^*)=accept \] \[ P(\text{random forgery})\approx\frac{1}{2^n} \] \[ P(\text{success after }q\text{ tries})\approx\frac{q}{2^n} \]

Note for self-study

The central security idea is not “can an attacker produce something that looks tag-like?” It is “can the attacker produce a valid tag for a new message after seeing tags for chosen messages?”

Part A: Warm-up vocabulary

QuestionAnswer
A1. What does \(K\) represent?The shared secret key.
A2. What does \(M\) represent?The message being authenticated.
A3. What does \(T\) represent?The authentication tag.
A4. What does \(Mac_K(M)\) mean?The tagging algorithm applied to message \(M\) using key \(K\).
A5. What does \(Vrfy_K(M,T)\) mean?The verification algorithm checks whether tag \(T\) is valid for message \(M\) under key \(K\).
A6. What does accept mean?The verifier says the tag is valid.
A7. What does reject mean?The verifier says the tag is invalid.
A8. What is a forgery?A new message-tag pair that verifies successfully without legitimate tagging for that message.
A9. What is an oracle in this context?A black-box helper that returns valid tags for messages the attacker chooses.

Part B: Three parts of a MAC system

PartNotationPurpose
Key generation\(K\leftarrow Gen()\)Create a secret key.
Tagging\(T=Mac_K(M)\)Produce a tag for a message.
Verification\(Vrfy_K(M,T)\)Accept or reject a message-tag pair.

B1 sample answer. A secret key is generated and shared by legitimate parties. The sender uses the key to compute a tag for the message. The receiver uses the same key to verify that the tag matches the message.

Part C: Correctness

C1 sample answer. If a tag is honestly computed for message \(M\) using key \(K\), then verification of that message-tag pair using the same key should accept.

C2 sample answer. A MAC must work correctly for honest users before we can meaningfully ask whether attackers can break it.

Part D: Integrity and authenticity

PropertyQuestion it answersMAC role
IntegrityWas the message changed?A changed message should fail verification with the old tag.
AuthenticityWas this approved by someone with the key?Only key holders should be able to make valid tags.

Part E: What counts as forgery?

AttemptForgery?Why?
Replay an old valid \((M,T)\)Usually not a new MAC forgeryThe attacker did not create a valid tag for a new message.
Guess random \(T\) for new \(M\), verifier rejectsNoThe pair is not accepted.
Ask for tag on \(M\), then submit same \((M,T)\)NoThe attacker already obtained that exact tag legitimately from the oracle.
Create valid tag for new \(M^*\)YesA new message-tag pair verifies.

E1 sample answer. Replay can be dangerous because an old valid command might still cause harm if accepted again, such as repeating a payment. It is different from a new forgery because the attacker reused a previously valid pair instead of creating a valid tag for a new message.

F1 sample answer. This models a strong realistic attacker who may be able to obtain valid tags from a system. A secure MAC should still prevent the attacker from tagging a new message.

Part G: UF-CMA

PieceMeaning
UFUnforgeability: attacker should not create a valid new tag.
CMAChosen-message attack: attacker may request tags for chosen messages.
Negligible probabilitySo tiny that it is considered infeasible at the security level.

G1 sample answer. UF-CMA security means that even after an attacker obtains valid tags for messages they choose, they still cannot create a valid tag for a new message except with negligible probability.

Part H: Tag length and guessing

Tag lengthRandom forgery probability per tryComment
32 bits\(1/2^{32}\)Often too short for many modern settings.
64 bits\(1/2^{64}\)Better, but context matters.
128 bits\(1/2^{128}\)Strong generic forgery resistance.
256 bits\(1/2^{256}\)Extremely large margin for random guessing.

H1 sample answer. Longer tags have more possible values, making random guessing less likely to succeed. An \(n\)-bit tag has about \(2^n\) possible values.

Part J: Tag truncation

J1 sample answer. Truncation can be acceptable in standardized designs with enough remaining bits, but shortening the tag directly increases random guessing risk. It should be based on threat model and standards, not convenience alone.

Part L: Scenario judgment

ScenarioLesson
An attacker submits a message-tag pair they already saw earlier.This is replay, not necessarily a new MAC forgery.
An attacker creates a valid tag for a new message they never queried.This is a successful forgery.
A system uses 32-bit tags and allows unlimited online attempts.Dangerous; short tags plus many attempts increase guessing risk.
A protocol needs to reject old valid messages sent again later.Use replay defenses such as nonces, timestamps, sequence numbers, or session binding.
A user thinks a MAC hides the message contents.Mistaken. MACs authenticate but do not encrypt.

Part N: Concept checks

N1 sample answer. A MAC security game models an attacker who can request valid tags for chosen messages and then tries to produce a valid tag for a new message. It defines what it means for the MAC to resist forgery.

N2 sample answer. Forgery means producing a valid tag for a new message. Replay means resending an old valid message-tag pair. Replay can still be harmful, but it is a different problem.

N3 sample answer. Tag length determines the chance of one random guess succeeding, while the number of attempts determines how many chances the attacker gets. Short tags become especially dangerous when many attempts are allowed.

N4 sample answer. This tutorial explained MACs as keyed authentication systems with key generation, tagging, and verification. It defined correctness, forgery, and the UF-CMA security game, where an attacker may request tags for chosen messages but should not be able to tag a new message. It also explained tag-length guessing, truncation, replay protection, and the difference between authentication and encryption.
