Authenticated Encryption: Encrypt-then-MAC and AEAD

How modern cryptographic systems combine confidentiality, integrity, and authenticity: why encryption alone is not enough, why Encrypt-then-MAC is the safest classic composition, and how AEAD modes package the whole job.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 49 explained MACs, tags, and forgery games. Now we connect authentication to encryption. In real systems, secrecy alone is not enough. We usually need both confidentiality and tamper detection. Authenticated encryption is the disciplined way to get both.

Encryption is the curtain. Authentication is the tamper-evident seal. Authenticated encryption is the locked crate with both the curtain and the seal, plus a manifest taped to the outside when some context must remain visible.

What you should know first

You should know that encryption hides message contents, while a MAC authenticates a message and detects tampering. You should also know that a MAC tag is verified before trusting a message-tag pair.

Three security goals

GoalQuestion answeredTool
ConfidentialityCan outsiders read it?Encryption.
IntegrityWas it changed?MAC or authenticated encryption tag.
AuthenticityWas it produced by someone with the right key?MAC or authenticated encryption tag.
Key distinction

Encryption without authentication may hide the message but still allow dangerous tampering. A sealed opaque box is not enough if an attacker can reshape what is inside without being detected.

Why encryption alone is not enough

Some encryption modes are malleable, meaning an attacker may be able to modify the ciphertext in a way that causes controlled or semi-controlled changes in the decrypted plaintext. Even when the attacker cannot read the message, tampering can be dangerous.

ProblemWhy it matters
Ciphertext tamperingThe receiver might decrypt altered data.
MalleabilityChanges in ciphertext may produce meaningful changes in plaintext.
Padding or format errorsDifferent errors may leak information.
No sender authenticationEncryption alone does not prove who created the ciphertext.
Design warning

“It is encrypted” does not mean “it is safe to process.” First verify authenticity, then decrypt or release plaintext.

Three classic composition orders

If encryption and MAC are separate tools, there are several ways to combine them. These names describe the order.

CompositionBasic shapeGeneral status
Encrypt-and-MACSend \(C=Enc(P)\) and \(T=Mac(P)\)Can leak information through the tag depending on MAC and message structure.
MAC-then-EncryptEncrypt \(P\,\|\,T\)Historically used, but error handling and padding can be fragile.
Encrypt-then-MACEncrypt \(P\), then MAC \(C\)Preferred generic composition when done correctly.

Encrypt-then-MAC

Definition: Encrypt-then-MAC

Encrypt-then-MAC is a composition where the sender first encrypts the plaintext to produce ciphertext, then computes a MAC tag over the ciphertext and relevant context. The receiver verifies the tag before decrypting.

\[ C=Enc_{K_E}(N,P) \] \[ T=Mac_{K_M}(N\,\|\,C) \] \[ \text{send } (N,C,T) \]

Verify before decrypt

The receiver should verify the tag before decrypting the ciphertext. If verification fails, reject the message and do not release plaintext.

Receiver stepActionReason
1Receive \((N,C,T)\).Collect nonce, ciphertext, and tag.
2Check \(T\) against \(N\,\|\,C\).Detect tampering before decryption.
3If tag fails, reject.Do not process untrusted ciphertext.
4If tag passes, decrypt.Only authenticated ciphertext is decrypted.
Practical rule

Authentication failure should be boring: reject cleanly, reveal as little as possible, and do not provide detailed error clues.

Use separate keys or derived subkeys

Encrypt-then-MAC is normally described with separate encryption and MAC keys:

\[ K_E \neq K_M \]

In practice, both keys may be derived from a master secret using a key derivation function. The point is that encryption and authentication should use distinct key material.

Reason

Separation reduces the risk that the use of one primitive interferes with the security assumptions of another. It is a form of compartmentalization.

AEAD: Authenticated Encryption with Associated Data

Definition: AEAD

AEAD stands for Authenticated Encryption with Associated Data. It is an encryption interface that provides confidentiality for plaintext and authenticity/integrity for both the ciphertext and additional associated data.

\[ (C,T)=AEAD.Enc_K(N,AAD,P) \] \[ P=AEAD.Dec_K(N,AAD,C,T) \quad \text{or reject} \]

AEAD modes are preferred in modern protocol design because they package encryption and authentication into one studied construction instead of asking every developer to assemble the pieces by hand.

Associated Data: authenticated but not encrypted

Associated Data, often written \(AAD\), is data that should remain visible but still be protected from tampering.

Associated data exampleWhy visible?Why authenticate?
Packet headerRouters or protocol handlers may need it.Attackers should not alter routing/context fields undetected.
Protocol versionReceiver needs to parse correctly.Prevents downgrade or confusion attacks.
Session IDUsed to identify connection context.Prevents messages from being moved between sessions.
Record numberUsed for ordering.Helps prevent replay or reordering.
Memory hook

AAD is like the address label on a sealed package. It must stay readable, but if someone changes it, the receiver should know.

Common AEAD modes

Current practice note

In modern systems, prefer well-studied AEAD modes through trusted libraries. Avoid designing your own authenticated encryption composition unless you are doing it as a learning exercise.

The nonce warning

Many AEAD modes require a nonce that must not repeat with the same key.

\[ \text{Same key }K + \text{same nonce }N = \text{danger} \]
Nonce behaviorEffect
Unique nonce per keyExpected safe use for nonce-based AEADs.
Repeated nonce with same keyCan reveal relationships between messages or break authentication depending on mode.
Random nonce too shortCollision risk may become non-negligible.
Counter nonce managed correctlyOften reliable if never reset under same key.
Design warning

Many modern encryption disasters are not caused by weak ciphers, but by using strong ciphers with repeated nonces. The sword is fine; the scabbard is on fire.

AEAD decryption either returns plaintext or rejects

AEAD decryption should not release unauthenticated plaintext. The ideal interface is simple:

\[ AEAD.Dec_K(N,AAD,C,T)=P \quad \text{or} \quad reject \]

If the tag is invalid, decryption fails. The receiver should not use partial plaintext or reveal detailed reasons for failure.

Common mistakes

Mistake 1: thinking encryption automatically authenticates

Encryption hides data. It does not automatically prove the ciphertext was not tampered with.

Mistake 2: decrypting before verifying

In Encrypt-then-MAC and AEAD-style thinking, authenticate before releasing plaintext.

Mistake 3: forgetting to authenticate visible context

Headers, versions, counters, and session identifiers may need authentication even when they remain visible.

Mistake 4: reusing nonces

Nonce reuse with the same key can destroy the security of many AEAD modes.

Why this matters for cryptography

Authenticated encryption is one of the most important practical lessons in modern cryptography. Many historical failures came from encrypting without authenticating, authenticating the wrong thing, handling errors poorly, or reusing nonces. AEAD modes exist because developers need safe, unified interfaces for real protocols.

  • Encryption provides confidentiality.
  • MACs provide integrity and authenticity.
  • Modern secure messaging usually needs both.
  • Encrypt-then-MAC is the safest classic generic composition.
  • AEAD modes combine encryption and authentication.
  • AAD is authenticated but not encrypted.
  • Nonce uniqueness is critical in many AEAD modes.
  • Reject invalid tags before releasing plaintext.

What to remember

  • A MAC provides integrity and authenticity, not confidentiality.
  • Authenticated encryption provides confidentiality, integrity, and authenticity together.

  • Encryption and MAC should use separate keys or derived subkeys.
  • AEAD stands for Authenticated Encryption with Associated Data.
  • AEAD encryption takes a key, nonce, associated data, and plaintext.
  • AAD is authenticated but not encrypted.
