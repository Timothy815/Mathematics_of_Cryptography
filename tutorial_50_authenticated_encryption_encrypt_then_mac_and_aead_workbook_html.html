How to use this workbook

Keep the jobs separate in your mind: encryption hides, authentication detects tampering and proves key-holder approval, and authenticated encryption does both in one disciplined construction.

Quick reference

Encrypt-then-MAC
\[ C=Enc_{K_E}(N,P) \] \[ T=Mac_{K_M}(N\,\|\,C) \] \[ \text{send }(N,C,T) \]
AEAD
\[ (C,T)=AEAD.Enc_K(N,AAD,P) \] \[ AEAD.Dec_K(N,AAD,C,T)=P \quad \text{or} \quad reject \]

Note for self-study

The central lesson is this: modern secure messages usually need both secrecy and tamper detection. AEAD gives a clean interface for doing both at once.

Part B: Security goals

GoalQuestion answeredTool
ConfidentialityCan outsiders read it?Encryption.
IntegrityWas it changed?MAC or authenticated encryption tag.

B1 sample answer. Encryption hides the message contents, while authentication detects tampering and proves that the message was produced by someone with the correct key. A message can be hidden but still unauthenticated.

Part C: Why encryption alone is not enough

ProblemWhy it matters
Ciphertext tamperingThe receiver might decrypt altered data.
MalleabilityChanges in ciphertext may produce meaningful changes in plaintext.
Padding or format errorsDifferent errors may leak information.
No sender authenticationEncryption alone does not prove who created the ciphertext.

C1 sample answer. Encrypted data may still be modified by an attacker. Without authentication, the receiver may process altered ciphertext or reveal useful error information.

Part D: Composition orders

CompositionBasic shapeGeneral status
Encrypt-and-MACSend \(C=Enc(P)\) and \(T=Mac(P)\)Can leak information through the tag depending on MAC and message structure.
MAC-then-EncryptEncrypt \(P\,\|\,T\)Historically used, but error handling and padding can be fragile.
Encrypt-then-MACEncrypt \(P\), then MAC \(C\)Preferred generic composition when done correctly.

Part E: Encrypt-then-MAC flow

FormulaMeaning
\(C=Enc_{K_E}(N,P)\)Encrypt plaintext \(P\) using encryption key \(K_E\) and nonce \(N\).
\(T=Mac_{K_M}(N\,\|\,C)\)Compute a MAC tag over the nonce and ciphertext using MAC key \(K_M\).
\(\text{send }(N,C,T)\)Send the nonce, ciphertext, and tag.

E1 sample answer. Authenticating the ciphertext lets the receiver check whether the encrypted data was altered before decrypting it. This prevents processing unauthenticated ciphertext.

Part F: Verify before decrypt

OrderReceiver actionReason
4If tag passes, decrypt.Only authenticated ciphertext is decrypted.
1Receive \((N,C,T)\).Collect nonce, ciphertext, and tag.
3If tag fails, reject.Do not process untrusted ciphertext.
2Check \(T\) against \(N\,\|\,C\).Detect tampering before decryption.

F1 sample answer. Detailed error messages can leak information. A clean, boring rejection avoids giving attackers clues about padding, formatting, or partial decryption behavior.

Part G: Separate keys or derived subkeys

G1 sample answer. Encryption and authentication have different security roles. Separate keys or derived subkeys compartmentalize the system so that using one primitive does not interfere with the assumptions of the other.

Part H: AEAD definition

FormulaMeaning
\((C,T)=AEAD.Enc_K(N,AAD,P)\)Encrypt plaintext \(P\) and produce ciphertext \(C\) plus tag \(T\), authenticating the associated data.
\(AEAD.Dec_K(N,AAD,C,T)=P\)If verification succeeds, return plaintext.
\(AEAD.Dec_K(N,AAD,C,T)=reject\)If verification fails, reject and release no plaintext.

H1 sample answer. AEAD modes are preferred because they combine encryption and authentication in one studied interface, reducing the chance that developers combine primitives incorrectly.

Part K: Nonce warning

Nonce behaviorEffect
Unique nonce per keyExpected safe use for nonce-based AEADs.
Repeated nonce with same keyCan reveal relationships between messages or break authentication depending on mode.
Random nonce too shortCollision risk may become non-negligible.
Counter nonce managed correctlyOften reliable if never reset under same key.

K1 sample answer. Many modes rely on the nonce to ensure fresh keystream or authentication conditions. Reusing a nonce with the same key can expose message relationships or even destroy authentication security.

Part P: Concept checks

P1 sample answer. Authenticated encryption combines encryption and authentication so that a message is both hidden and protected from undetected tampering. It provides confidentiality, integrity, and authenticity together.

P2 sample answer. Verify-before-decrypt prevents the system from processing altered ciphertext or leaking information through decryption errors. If authentication fails, the system should reject without releasing plaintext.

P3 sample answer. Plaintext is the readable message. Ciphertext is the encrypted message. A tag authenticates the ciphertext and context. A nonce is a value used once with the key. AAD is visible context that is authenticated but not encrypted.

P4 sample answer. This tutorial explained authenticated encryption as the combination of confidentiality, integrity, and authenticity. It compared composition orders and emphasized Encrypt-then-MAC as the safest classic generic pattern. It introduced AEAD as the preferred modern interface, where plaintext is encrypted, associated data is authenticated, and decryption returns plaintext or reject. It also warned that nonce reuse can break otherwise strong modes.
