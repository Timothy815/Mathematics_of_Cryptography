AEAD Case Study: AES-GCM and the Mathematics of GHASH

A guided look inside AES-GCM: AES counter mode provides confidentiality, GHASH provides authentication over ciphertext and associated data, and finite-field arithmetic ties the whole design together.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 50 introduced authenticated encryption and AEAD. This tutorial studies AES-GCM as a concrete case. AES-GCM is useful because it shows two major cryptographic ideas working together: counter-mode encryption for secrecy and Galois-field hashing for authentication.

AES-GCM is a two-engine machine. One engine turns counters into keystream blocks and hides the plaintext. The other engine stirs associated data and ciphertext through finite-field multiplication to build an authentication value. One side wears the cloak; the other checks the wax seal.

What you should know first

You should know the goals of AEAD: confidentiality, integrity, and authenticity. You should also know the basic idea of finite-field arithmetic from earlier tutorials: elements can be represented as bit strings, added by XOR, and multiplied modulo an irreducible polynomial.

Counter mode review

AES-GCM encrypts using AES in counter mode. Instead of encrypting the plaintext directly, AES encrypts counter blocks to make keystream blocks. The keystream is XORed with plaintext.

\[ S_i=AES_K(ctr_i) \] \[ C_i=P_i\oplus S_i \]
StepActionMeaning
1Build counter block \(ctr_i\).Nonce plus counter information.
2Compute \(S_i=AES_K(ctr_i)\).AES produces a keystream block.
3Compute \(C_i=P_i\oplus S_i\).XOR hides plaintext.
4Decrypt using \(P_i=C_i\oplus S_i\).XOR with same keystream reverses encryption.
CTR mode warning

Reusing the same nonce and key can reuse keystream blocks. Reused keystream can expose relationships between plaintexts. This is one reason nonce uniqueness matters so fiercely in AES-GCM.

What GHASH does

GHASH authenticates the associated data and ciphertext. It does not encrypt. Instead, it combines blocks using XOR and multiplication in \(GF(2^{128})\).

Definition: GHASH

GHASH is a finite-field hash function used inside GCM. It processes associated data, ciphertext, and length information using multiplication by a hash subkey \(H\) in \(GF(2^{128})\).

\[ H=AES_K(0^{128}) \]

The value \(H\) is called the hash subkey. It is secret in practice because it is derived from the AES key.

The finite-field connection

GHASH treats 128-bit blocks as elements of \(GF(2^{128})\). Addition is XOR. Multiplication is polynomial multiplication reduced modulo a fixed irreducible polynomial.

OperationIn GHASHEarlier course connection
AdditionXOR of 128-bit blocksSame as addition in binary polynomial fields.
MultiplicationCarryless multiply then reduceSame idea as \(GF(2^8)\), scaled to 128 bits.
ReductionModulo a fixed irreducible polynomialKeeps result inside the field.
Hash subkey\(H=AES_K(0^{128})\)Used as the multiplier in the field hash.
Memory hook

GHASH is like a 128-bit polynomial mixing bowl. Each block is stirred in with XOR, then folded through multiplication by \(H\).

The GHASH recurrence

After the associated data and ciphertext are formatted into blocks \(X_1,X_2,\ldots,X_m\), GHASH can be described with a recurrence:

\[ Y_0=0 \] \[ Y_i=(Y_{i-1}\oplus X_i)\cdot H \] \[ GHASH_H(X_1,\ldots,X_m)=Y_m \]

This is the authentication-side cousin of the chaining-value idea we saw in hash functions. The state \(Y_i\) keeps absorbing blocks, but the arithmetic is field multiplication by \(H\).

StepFormulaMeaning
Start\(Y_0=0\)Start from zero.
Absorb block\(Y_{i-1}\oplus X_i\)XOR the next block into the state.
Mix\((Y_{i-1}\oplus X_i)\cdot H\)Multiply by hash subkey in \(GF(2^{128})\).
Finish\(Y_m\)Output GHASH value.

How the final tag is made

GHASH produces an authentication value, but the final tag is masked using AES output on the initial counter block \(J_0\).

\[ S=GHASH_H(A,C) \] \[ T=AES_K(J_0)\oplus S \]

The final tag is therefore tied both to the GHASH result and to AES under the secret key.

Decryption and verification

AES-GCM decryption should verify the tag. If the tag is wrong, the receiver rejects. If it is correct, the ciphertext can be decrypted with AES-CTR.

Receiver stepAction
1Receive nonce \(N\), associated data \(A\), ciphertext \(C\), and tag \(T\).
2Recompute GHASH over \(A\), \(C\), and length information.
3Recompute expected tag using \(AES_K(J_0)\oplus GHASH\).
4If tag does not match, reject.
5If tag matches, decrypt ciphertext using AES-CTR.

Why nonce reuse is catastrophic

AES-GCM requires nonce uniqueness under the same key. Reusing the same nonce with the same key can damage both confidentiality and authentication.

FailureWhat can go wrong?
CTR keystream reuseTwo ciphertexts may reveal relationships between two plaintexts.
GHASH structure exposureRepeated nonces can help attackers derive information about authentication equations.
Forgery riskAuthentication security may collapse under serious nonce misuse.
Protocol failureThe mode may no longer provide its promised AEAD security.
Design warning

AES-GCM is powerful, but it is not forgiving. Reusing a nonce under the same key is not a scratch on the paint; it is a crack through the engine block.

Toy GHASH over a tiny field

Real GHASH uses \(GF(2^{128})\), which is too large for hand practice. We can see the structure with a tiny toy recurrence over 4-bit values.

\[ Y_0=0 \] \[ Y_i=(Y_{i-1}\oplus X_i)\cdot H \]

Suppose \(H=3\), and our toy multiplication is ordinary multiplication modulo 16 just for demonstration. This is not real GHASH field arithmetic, but it shows the absorb-then-multiply rhythm.

Startnone\(Y_0=0\)0

Toy warning

This example is not real GHASH arithmetic. Real GHASH uses carryless multiplication in \(GF(2^{128})\), not ordinary multiplication modulo 16. The toy example only illustrates the state update pattern.

Why GHASH is fast

GHASH was designed to be efficient on modern processors. Its operations are XOR and carryless multiplication, both of which can be implemented very quickly with hardware support.

FeatureWhy it helps
XOR additionFast bitwise operation.
Carryless multiplicationEfficient on processors with dedicated instructions.
Parallelizable encryptionCTR blocks can be generated independently.

Common mistakes

Mistake 1: thinking GCM is only encryption

GCM is authenticated encryption. It provides both encryption and authentication when used correctly.

Mistake 2: ignoring AAD

Associated data is not encrypted, but it is authenticated. Changing it should break verification.

Mistake 3: reusing nonces

Nonce reuse under the same key can destroy AES-GCM security.

Mistake 4: treating GHASH as a normal cryptographic hash

GHASH is a keyed authentication component inside GCM, not a standalone replacement for SHA-256 or HMAC.

Why this matters for cryptography

AES-GCM is a beautiful meeting point between block ciphers, stream-like encryption, authentication, and finite-field arithmetic. It also teaches a practical lesson that appears throughout real-world cryptography: strong primitives can fail when their usage rules are violated, especially nonce uniqueness.

  • AES-GCM is an AEAD mode.
  • AES-CTR hides the plaintext.
  • GHASH authenticates associated data and ciphertext.
  • GHASH works over \(GF(2^{128})\).
  • The hash subkey is \(H=AES_K(0^{128})\).
  • The tag combines GHASH with \(AES_K(J_0)\).
  • Nonce uniqueness under the same key is essential.
  • GHASH is not a general-purpose hash like SHA-256.
