Nonce Misuse, Replay, and Protocol Context

Why “unique,” “fresh,” and “bound to context” are not decorative protocol words: how nonce reuse breaks modes, how replay reuses old truth, and how associated data ties messages to the right situation.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 51 showed that AES-GCM can fail badly when a nonce is reused. This tutorial widens that lesson. Real protocols do not only need strong primitives. They need messages that are unique, fresh, ordered when necessary, and bound to the correct protocol context.

A cryptographic message is not just a sealed envelope. It is a sealed envelope with a date stamp, a serial number, a destination, a sender role, and a warning label saying which ceremony it belongs to. Without that context, an old honest message can become tomorrow’s counterfeit ghost.

What you should know first

You should know that AEAD modes authenticate ciphertext and associated data, and that many modes require a nonce that must not repeat with the same key. You should also know that a MAC or AEAD tag proves that a message was valid under a key, but does not automatically prove that the message is new.

What is a nonce?

Definition: nonce

A nonce is a value intended to be used once in a particular cryptographic context, usually under a specific key.

The word “nonce” often means “number used once,” but the important phrase is under the same key and same purpose. A nonce may be random, counter-based, or constructed from protocol fields, but it must satisfy the uniqueness requirements of the mode.

Nonce typeHow it worksMain risk
Counter nonceIncrease a counter for every message.Resetting the counter under the same key.
Random nonceChoose a random value large enough to avoid collisions.Collision if too short or random source fails.
Structured nonceCombine fields such as device ID and counter.Bad field design or overlapping ranges.

Unique does not mean secret

A nonce usually does not need to be secret. It usually needs to be unique under the key.

PropertyNonce requirement
Secret?Usually no.
Unique under same key?Usually yes, and often absolutely required.
Unpredictable?Sometimes, depending on the scheme.
Authenticated?Often included in tag computation or AEAD processing.
Memory hook

A nonce is less like a password and more like a ticket number. Everyone can see it, but disaster begins when the same ticket number is used twice at the same counter.

What breaks when a nonce is reused?

The exact failure depends on the construction. In stream-cipher-like modes and counter modes, nonce reuse can reuse the keystream. In AEAD modes such as AES-GCM, nonce reuse can damage both confidentiality and authentication.

Mode or constructionNonce reuse danger
CTR modeSame nonce/counter can repeat keystream.
Stream cipher modesRepeated keystream can expose relationships between plaintexts.
AES-GCMCan break confidentiality and authentication.
ChaCha20-Poly1305Can repeat keystream and break authentication assumptions.
Misuse-resistant AEADDesigned to reduce harm, but not a license for carelessness.

The two-time pad pattern

If the same keystream \(S\) is used to encrypt two plaintexts, then:

\[ C_1=P_1\oplus S \] \[ C_2=P_2\oplus S \]

XORing the two ciphertexts cancels the keystream:

\[ C_1\oplus C_2=(P_1\oplus S)\oplus(P_2\oplus S) \] \[ C_1\oplus C_2=P_1\oplus P_2 \]
Why this is bad

The attacker may not immediately know both plaintexts, but they now have a relationship between them. If one plaintext is guessed or partially known, the other can leak. The keystream was supposed to vanish from sight, but reuse makes it cancel like a magician’s trapdoor.

Replay attacks

Definition: replay attack

A replay attack occurs when an attacker records a valid message, ciphertext, or message-tag pair and sends it again later in a context where it is incorrectly accepted as fresh.

ExampleWhy dangerous?
Repeat “transfer $100” messagePayment may happen twice.
Repeat “unlock door” commandOld authorization may open the door again.
Replay login tokenAttacker may reuse a captured session artifact.
Replay protocol handshake dataMay confuse state or downgrade context if not protected.
Important distinction

Replay is not the same as forging a new tag. The old tag may be completely valid. The failure is that the receiver lacks a freshness rule.

Freshness tools

Protocols use freshness mechanisms to reject old or repeated messages.

ToolPurposeTypical check
NonceEnsure value is used once.Reject repeated nonce under same key/context.
Sequence numberOrder messages.Reject old, repeated, or out-of-window numbers.
TimestampLimit validity period.Reject messages outside allowed time window.
Challenge-responseProve liveness.Respond to a fresh challenge from verifier.
Session IDBind message to a session.Reject messages from another session.

Protocol context

A message should often be authenticated together with the context that gives it meaning. This prevents attackers from moving a valid message into the wrong place.

Context fieldWhy bind it?
Protocol namePrevents using a message from one protocol in another.
Protocol versionPrevents downgrade or version confusion.
Sender and receiver rolesPrevents reflection or role-confusion attacks.
Session IDPrevents moving messages between sessions.
Message typePrevents treating one kind of message as another.
Sequence numberPrevents replay and reordering.
Key principle

Authenticate not only the bytes, but the meaning of the bytes.

AAD as context binding

In AEAD systems, associated authenticated data is the natural place to bind visible protocol context.

\[ (C,T)=AEAD.Enc_K(N,AAD,P) \]

The plaintext \(P\) is encrypted. The associated data \(AAD\) remains visible, but the tag protects it from undetected tampering.

AAD might include:
  protocol = "example-secure-channel-v2"
  session_id = 7F2A...
  sender_role = client
  receiver_role = server
  sequence_number = 1042
  message_type = application_data
Memory hook

AAD is the message’s passport. It does not hide the traveler, but it binds the traveler to a name, place, role, and route.

Domain separation

Definition: domain separation

Domain separation means including labels, constants, or context values so that cryptographic outputs for one purpose cannot be confused with outputs for another purpose.

Without domain separation, the same key or primitive might accidentally produce values that can be reused in another context. Labels help keep ceremonies separate.

Without labelWith label
\(MAC_K(data)\)\(MAC_K("login"\,\|\,data)\)
\(MAC_K(data)\)\(MAC_K("payment"\,\|\,data)\)
\(Hash(data)\)\(Hash("transcript"\,\|\,data)\)
\(KDF(secret)\)\(KDF(secret,"encryption-key")\), \(KDF(secret,"mac-key")\)

State management

Many nonce and replay protections depend on state. The system must remember what has been used, what counter comes next, or what sequence numbers are acceptable.

State failureConsequence
Counter resets after rebootNonce reuse under same key.
Multiple devices share key and counter rangeTwo devices may emit same nonce.
No replay cacheOld valid messages may be accepted again.
Loose timestamp windowReplay may remain possible inside the window.
Out-of-order handling too permissiveAttackers may replay older sequence numbers.
Engineering truth

Cryptography is not only equations. It is also bookkeeping. Bad bookkeeping can make good equations cry in hexadecimal.

Toy protocol example

Suppose a secure channel sends this AEAD-protected message:

AAD:
  protocol = "ToyChannel-v1"
  session_id = 91B4
  sender = client
  receiver = server
  sequence_number = 0007
  message_type = transfer_request

Plaintext:
  amount=100; recipient=Alice

The associated data is not secret, but it is authenticated. If an attacker changes the sequence number, role, session ID, protocol version, or message type, verification should fail.

Attacker changes...Should receiver accept?Why?
sequence_number from 0007 to 0005NoAAD changed and may indicate replay.
receiver from server to adminNoRole/context changed.
message_type from transfer_request to balance_checkNoMessage meaning changed.
nothingYes, if tag verifies and sequence is freshAuthenticated and fresh message.

Common mistakes

Mistake 1: thinking nonces must be secret

Nonces usually need uniqueness, not secrecy.

Mistake 2: authenticating the message but not the context

A valid message in the wrong context can still be dangerous.

Mistake 3: treating replay as forgery

Replay reuses old truth. Forgery creates new falsehood. Both can be dangerous, but the defenses differ.

Mistake 4: resetting counters under the same key

Counter reset is a common path to nonce reuse.

Why this matters for cryptography

Nonce management, replay protection, and context binding are where cryptographic theory meets protocol engineering. A cipher can be mathematically strong and still fail if nonces repeat, old messages are accepted, or authenticated data is detached from its intended meaning.

  • Nonces are usually public but must be unique under the relevant key.
  • Nonce reuse can break stream-like encryption and AEAD modes.
  • Replay attacks reuse old valid messages.
  • Freshness mechanisms reject old or repeated messages.
  • AAD binds visible protocol context into authentication.
  • Domain separation prevents cross-purpose confusion.
  • Good cryptography requires good state management.
