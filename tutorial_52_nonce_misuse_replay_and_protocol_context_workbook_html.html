Part B: Nonce basics

B1 sample answer. A nonce is a value intended to be used once under a specific key and cryptographic context.

Nonce typeHow it worksMain risk
Counter nonceIncrease a counter for every message.Resetting the counter under the same key.
Random nonceChoose a random value large enough to avoid collisions.Collision if too short or random source fails.
Structured nonceCombine fields such as device ID and counter.Bad field design or overlapping ranges.

Part C: Unique does not mean secret

PropertyNonce requirement
Secret?Usually no.
Unique under same key?Usually yes, and often absolutely required.
Unpredictable?Sometimes, depending on the scheme.
Authenticated?Often included in tag computation or AEAD processing.

C1 sample answer. A nonce can usually be visible like a ticket number. The important issue is that the same ticket number should not be reused in the same context under the same key.

Part D: What breaks when a nonce is reused?

Stream cipher modesRepeated keystream can expose relationships between plaintexts.
AES-GCMCan break confidentiality and authentication.

Part E: Two-time pad pattern

E1 work.

\[ C_1\oplus C_2=(P_1\oplus S)\oplus(P_2\oplus S) \] \[ =P_1\oplus P_2\oplus S\oplus S \] \[ =P_1\oplus P_2\oplus 0 \] \[ =P_1\oplus P_2 \]

E2 sample answer. Knowing \(P_1\oplus P_2\) gives a relationship between the two plaintexts. If one plaintext is guessed, partly known, or structured, information about the other may leak.

Part F: Replay attacks

F1 sample answer. A replay attack happens when an attacker records a valid message, ciphertext, or tag and sends it again later where the receiver wrongly accepts it as fresh.

ExampleWhy dangerous?
Repeat “transfer $100” messagePayment may happen twice.
Repeat “unlock door” commandOld authorization may open the door again.
Replay login tokenAttacker may reuse a captured session artifact.
Replay protocol handshake dataMay confuse state or downgrade context if not protected.

F2 sample answer. Replay reuses an old valid message-tag pair. Forgery creates a valid tag for a new message. Replay can still be harmful because the receiver may need freshness checks.

Part H: Protocol context

Context fieldWhy bind it?
Protocol namePrevents using a message from one protocol in another.
Protocol versionPrevents downgrade or version confusion.
Sender and receiver rolesPrevents reflection or role-confusion attacks.
Session IDPrevents moving messages between sessions.
Message typePrevents treating one kind of message as another.
Sequence numberPrevents replay and reordering.

H1 sample answer. The same bytes can mean different things in different contexts. Authentication should bind the data to its intended protocol, session, role, and message type so it cannot be moved or reinterpreted safely by an attacker.

Part I: AAD as context binding

I1 sample answer. AAD remains visible but is protected by the authentication tag. That makes it ideal for protocol fields that must be readable but must not be changed undetected.

I2 sample answer. Examples include protocol version, session ID, sender role, receiver role, sequence number, and message type.

Part J: Domain separation

Without labelWith labelWhy better?
\(MAC_K(data)\)\(MAC_K("login"\,\|\,data)\)Prevents login tags from being confused with other tags.
\(MAC_K(data)\)\(MAC_K("payment"\,\|\,data)\)Binds the tag to payment use.
\(Hash(data)\)\(Hash("transcript"\,\|\,data)\)Marks the hash as a transcript hash.

J1 sample answer. Domain separation means using labels or context values so cryptographic outputs meant for one purpose cannot be confused with outputs meant for another purpose.

Part O: Concept checks

O1 sample answer. Nonce misuse happens when a value that should be unique under a key is reused or generated incorrectly. In stream-like modes and AEAD modes, this can repeat keystream, leak relationships between plaintexts, or damage authentication.

O2 sample answer. A replay attack reuses an old valid message or tag in a later context. Freshness tools such as nonces, sequence numbers, timestamps, challenge-response, and session IDs help receivers reject old or repeated messages.

O3 sample answer. Protocol context binding means authenticating fields that explain what a message means, such as protocol version, session ID, roles, and message type. Domain separation uses labels or context to keep cryptographic purposes from being confused.

O4 sample answer. This tutorial explained that nonce uniqueness, replay protection, and protocol context are essential to secure cryptographic systems. Nonces usually need to be unique rather than secret, and nonce reuse can cause two-time-pad-style leakage. Replay attacks reuse old valid messages, so systems need freshness tools. AAD and domain separation bind visible context and purpose into authentication.
