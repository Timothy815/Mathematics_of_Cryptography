Key Derivation Functions: Turning Secrets into Separate Keys

How cryptographic systems turn one shared secret into separate, labeled, context-bound keys for encryption, authentication, AEAD, sessions, and protocol roles.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 52 showed why protocol context and domain separation matter. This tutorial explains a major tool for doing that correctly: Key Derivation Functions, or KDFs. A KDF turns one secret into separate keys, each labeled for a specific job.

A KDF is a cryptographic key workshop. You bring in one rough ingot of secret material, then stamp out separate tools: one for encryption, one for authentication, one for client messages, one for server messages, one for this session and not that session. The labels on the tools matter.

What you should know first

You should know that encryption and authentication should often use separate keys or derived subkeys. You should also know that protocol context, labels, session IDs, and roles help prevent cryptographic confusion.

What is a KDF?

Definition: Key Derivation Function

A Key Derivation Function, or KDF, takes secret input material and context information, then produces one or more cryptographic keys suitable for specific purposes.

\[ OKM=KDF(IKM, salt, info, length) \]
InputPurpose
\(IKM\)The secret input material.
saltHelps randomize, separate, or strengthen extraction.
infoNames the purpose and protocol context.
lengthHow many bytes of output key material are needed.

Why not use the same secret for everything?

Using one secret directly for many jobs is dangerous. Encryption, authentication, client traffic, server traffic, session resumption, and handshake confirmation are different ceremonies. They should not all share the same raw key material.

Bad habitRisk
Same key for encryption and MACPrimitive interactions may violate assumptions.
Same key for client and server trafficReflection or direction-confusion risks.
Same key across sessionsCompromise or nonce reuse may spread damage.
No purpose labelsKeys or tags may be confused across contexts.
Memory hook

One master secret should be a seed, not a skeleton key used on every lock in the kingdom.

HKDF as the main mental model

HKDF stands for HMAC-based Key Derivation Function. It is a widely used KDF built from HMAC. For this course, HKDF is the clearest model because it explicitly separates extraction and expansion.

\[ PRK=HKDF\text{-}Extract(salt,IKM) \] \[ OKM=HKDF\text{-}Expand(PRK,info,L) \]
Important distinction

HKDF is not for storing passwords. Passwords are usually low-entropy human secrets and need password-hashing or password-based KDFs such as Argon2id, scrypt, bcrypt, or PBKDF2 depending on the setting.

The power of labels: info

The info field tells the KDF what the derived material is for. This is how a protocol practices domain separation.

Derived keyExample info labelPurpose
\(K_E\)"ToyChannel v1 client encryption"Encrypt client-to-server messages.
\(K_M\)"ToyChannel v1 client mac"Authenticate client-to-server messages.
Server encryption key"ToyChannel v1 server encryption"Encrypt server-to-client messages.
Handshake key"ToyChannel v1 handshake finished"Authenticate handshake transcript.
Principle

Different job, different label, different derived key.

What does the salt do?

A salt is usually not secret. Its purpose depends on the KDF setting, but in HKDF-style extraction it helps make the extraction stage more robust and separates derivations.

Salt propertyMeaning
Usually publicIt does not need to be hidden like a key.
Often random or context-specificHelps separate derivation instances.
Used during extractContributes to producing the PRK.
Not a password replacementA salt does not make a weak password strong by itself.

KDFs and key schedules

In protocol design, a key schedule is the organized process for deriving all keys needed by the protocol. A KDF is often the engine inside that schedule.

shared_secret
  ↓ HKDF-Extract with salt
PRK
  ↓ HKDF-Expand(info="client app traffic")
client_application_key
  ↓ HKDF-Expand(info="server app traffic")
server_application_key
  ↓ HKDF-Expand(info="exporter")
exporter_secret

The important idea is that all derived keys are related to the same root secret, but they are separated by labels and context.

Password KDFs are a related but different creature

HKDF assumes the input keying material is already reasonably strong, such as a Diffie-Hellman shared secret. Human passwords are usually weak and guessable, so they need a different class of function.

ToolInput typeMain goal
HKDFHigh-entropy secret materialDerive separate keys.
Argon2idPasswordSlow/memory-hard password hashing.
scryptPasswordMemory-hard password-based derivation.
PBKDF2PasswordIterated password-based derivation.
Common confusion

HKDF is excellent for expanding strong secrets into multiple keys. It is not the right mental model for making a bad human password safe against offline guessing.

Good KDF habits

HabitReason
Use a standard KDFAvoid designing your own key workshop with crooked tools.
Use clear labelsSupports domain separation.
Separate directionsPrevents client/server role confusion.
Separate purposesEncryption keys and MAC keys should not be casually reused.
Include protocol contextBinds derived keys to the correct protocol/session/version.
Rotate or refresh keys as neededLimits damage over time and across sessions.

Common mistakes

Mistake 1: slicing a hash without labels

Deriving keys by simply hashing a secret and cutting it into pieces can lack clear domain separation and protocol binding.

Mistake 2: using the same derived key for multiple purposes

If two jobs are different, derive two labeled keys.

Mistake 3: confusing salt with secrecy

A salt is usually public. It is not a second password.

Mistake 4: using HKDF for weak passwords

Use password-hashing/password-based KDF tools for passwords.

Why this matters for cryptography

KDFs are the quiet machinery behind serious protocols. They turn one negotiated secret into a set of separated, labeled keys. Without them, systems drift toward key reuse, role confusion, protocol confusion, and long-term damage from a single compromised value.

  • A KDF turns secret input material into usable output key material.
  • HKDF uses extract and expand stages.
  • Labels and context provide domain separation.
  • Encryption and authentication should use separate keys or subkeys.
  • Client and server directions often need separate keys.
  • Salts are usually public and not the same as passwords.
  • Password KDFs solve a different problem than HKDF.
  • Key schedules organize derived keys for a protocol.
