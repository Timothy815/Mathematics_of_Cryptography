How to use this workbook

Track the purpose of each key. A KDF is not merely producing random-looking bytes. It is producing separated, labeled key material for specific cryptographic jobs.

Quick reference

Generic KDF shape
\[ OKM=KDF(IKM,salt,info,length) \]
HKDF extract and expand
\[ PRK=HKDF\text{-}Extract(salt,IKM) \] \[ OKM=HKDF\text{-}Expand(PRK,info,L) \]

Note for self-study

The core idea is controlled separation: one source secret can safely feed many cryptographic jobs only when a standard KDF derives purpose-specific keys with labels and context.

InputPurpose

B1 sample answer. A Key Derivation Function takes secret input material plus context and produces one or more usable cryptographic keys for specific purposes.

Part C: Why not use the same secret for everything?

Bad habitRisk
Same key for encryption and MACPrimitive interactions may violate assumptions.
Same key for client and server trafficReflection or direction-confusion risks.
Same key across sessionsCompromise or nonce reuse may spread damage.
No purpose labelsKeys or tags may be confused across contexts.

C1 sample answer. A master secret should feed a derivation process that creates separate purpose-specific keys. It should not be used directly for every cryptographic lock because one misuse could affect many jobs.

Part D: Extract-and-expand pattern

StageFormula shapePurpose
Extract\(PRK=Extract(salt,IKM)\)Clean up input secret material into a pseudorandom key.
Expand\(OKM=Expand(PRK,info,length)\)Produce labeled output key material of desired length.

D1 sample answer. Extract turns the input secret material into a clean pseudorandom key. Expand uses that pseudorandom key with labels and length requests to create the actual output key material.

Part E: HKDF

E1 sample answer. HKDF stands for HMAC-based Key Derivation Function.

E2 sample answer. HKDF is clear because it explicitly separates the job of cleaning up the input secret from the job of expanding it into labeled keys.

Part F: The power of labels

Derived keyExample info labelPurpose
\(K_E\)"ToyChannel v1 client encryption"Encrypt client-to-server messages.

F1 sample answer. Different cryptographic jobs should receive distinct derived keys with clear labels. This prevents a key meant for one purpose from being reused or confused with a key meant for another purpose.

Part G: Salt

G1 sample answer. Salt is usually public and helps separate or randomize derivations. It is not secret key material and should not be treated as a password.

Part H: KDFs and key schedules

H1 sample answer. A key schedule is the organized process that derives all cryptographic keys needed by a protocol from one or more root secrets.

H2 sample answer. Labels and context bind each derived key to its protocol role, session, direction, and purpose, reducing key confusion and cross-use problems.

Part J: Password KDFs are different

ToolInput typeMain goal
HKDFHigh-entropy secret materialDerive separate keys.
Argon2idPasswordSlow/memory-hard password hashing.
scryptPasswordMemory-hard password-based derivation.
PBKDF2PasswordIterated password-based derivation.

J1 sample answer. HKDF assumes strong input keying material. Human passwords are usually low-entropy and need password-hashing or password-based KDFs designed to slow offline guessing.

Part N: Concept checks

N1 sample answer. A KDF takes secret input material and derives one or more usable cryptographic keys. It helps separate keys by purpose, direction, session, and protocol context.

N2 sample answer. HKDF first extracts a clean pseudorandom key from input keying material and salt. It then expands that pseudorandom key into output key material using labels, context, and a requested length.

N3 sample answer. Labels and context tell the KDF what each key is for. This prevents encryption keys, MAC keys, client keys, server keys, and handshake keys from being confused or reused for the wrong job.

N4 sample answer. This tutorial explained Key Derivation Functions as tools for turning one secret into separate purpose-specific keys. HKDF uses extract and expand stages, with salt helping extraction and info labels binding keys to context. KDFs support key schedules, direction separation, purpose separation, and domain separation. Password KDFs are related but different because human passwords need slow or memory-hard protection against guessing.
