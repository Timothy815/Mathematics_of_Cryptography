Password Hashing and Password-Based Key Derivation

Why passwords need special treatment: salts, slow hashing, memory hardness, offline guessing, password storage, and the difference between HKDF and password-based KDFs.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 53 explained KDFs for deriving separate keys from strong secret material. Passwords are different. Human passwords are often low-entropy and guessable, so they need deliberately slow and sometimes memory-hard protection against offline attackers.

A normal KDF is a key workshop. A password hashing system is a fortified toll bridge. Every guess must pay time, memory, or both. The defender cannot make bad passwords magically strong, but can make mass guessing far more expensive.

What you should know first

You should know that ordinary cryptographic hashes such as SHA-256 are designed to be fast. That speed is good for file integrity and signatures, but bad for password storage because attackers can test guesses rapidly. You should also know that salts are usually public and help prevent precomputed or shared attacks across users.

Passwords are different from random keys

A cryptographic key might be 128 or 256 random bits. A human password might be a word, phrase, pattern, reused secret, or slightly modified dictionary entry. That means attackers can often guess likely passwords before trying impossible random values.

Secret typeTypical sourceAttacker strategy
Random keyCryptographic random generatorBrute force is usually infeasible.
Human passwordMemory, habit, reuse, patternsDictionary, mutation, leaked password lists, guessing.
Key idea

Password hashing does not make a weak password mathematically random. It makes each guess more expensive.

Bad password storage patterns

Storage patternWhy bad?
Plaintext passwordDatabase theft immediately reveals passwords.
Fast unsalted hashAttackers can guess quickly and use precomputed tables.
Same salt for everyoneSame passwords still reveal matching hashes across users.
Outdated weak hash settingsAttackers may test guesses cheaply with modern hardware.
Homebrew password schemeUntested designs often fail in quiet, creative ways.

Salt: unique public seasoning

Definition: salt

A salt is a unique, usually random, usually public value stored with a password hash. It ensures that the same password does not produce the same stored hash across users or accounts.

\[ stored = salt\,\|\,PasswordHash(password,salt,cost) \]
Salt propertyMeaning
Unique per password/accountPrevents identical passwords from sharing identical hashes.
PublicMay be stored with the hash.
Random enoughMakes precomputed tables impractical across many salts.
Not a substitute for slow hashingSalt separates hashes but does not slow guesses by itself.

Offline guessing attacks

The scariest password-storage scenario is database theft. Once attackers have the stored hashes and salts, they can test guesses on their own machines without asking the real server.

A good password hashing scheme makes each guess expensive enough that large-scale guessing becomes harder.

Why slow hashing helps

Ordinary cryptographic hashes are designed to be fast. Password hashing deliberately adds cost.

Design goalWhy?
Slow enough for attackersReduces guesses per second.
Fast enough for real usersLogin should remain acceptable.
Adjustable costCan increase difficulty as hardware improves.
Memory cost if possibleMakes GPU/ASIC mass guessing more expensive.
The defender’s balancing act

A password hash should be tolerable for one login but painful for millions of guesses.

Memory hardness

Definition: memory-hard password hashing

A memory-hard password hashing function is designed to require significant memory during computation, making large-scale parallel guessing more expensive.

Cost typeWhat it charges the attacker
Time costMore computation per guess.
Memory costMore RAM per guess.
Parallelism settingControls how work may be split across threads/cores.

Memory hardness matters because attackers often rely on massive parallelism. If every guess needs meaningful memory, the attack becomes harder to scale.

Common password hashing and password-based derivation tools

ToolMain ideaTypical role
Argon2idMemory-hard password hashing with hybrid data access pattern.Strong modern default for password storage when available.
scryptMemory-hard password-based derivation.Password storage or password-derived keys.
bcryptSlow password hashing based on Blowfish key setup.Longstanding password storage option.
PBKDF2Repeated pseudorandom function, often HMAC.Widely standardized, but not memory-hard.
Practical note

Use trusted libraries and current parameters. The algorithm name alone is not enough; weak parameters can turn a good tool into cardboard armor.

Password storage flow

PhaseAction

Pepper: optional server-side secret

Definition: pepper

A pepper is an additional secret value used with password hashing and stored separately from the password database.

ValueSecret?Stored where?
SaltNoWith the password hash.
PepperYesSeparate secret store or application environment.
Password hashNo, but sensitivePassword database.

Pepper can add defense if the database is stolen without the application secret, but it also creates key-management responsibilities.

Password-based key derivation

Sometimes a password is used to derive an encryption key, such as for an encrypted file vault. In that case, a password-based KDF stretches the password into key material.

\[ K=PBKDF(password,salt,parameters) \]

This still does not make a weak password strong. It makes guessing more expensive and produces a properly sized key for the encryption system.

Online versus offline attacks

Attack typeWhere guesses are testedMain defenses
Online guessingAgainst the live login service.Rate limiting, lockouts, multifactor authentication, monitoring.
Offline guessingAgainst stolen hashes on attacker hardware.Strong password hashing, salts, memory hardness, strong passwords.
Memory hook

Online guessing knocks on the front door. Offline guessing steals the lock and takes it to a warehouse full of power tools.

Common mistakes

Mistake 1: using SHA-256 directly for passwords

SHA-256 is fast. Attackers like fast.

Mistake 2: using one salt for all users

Salts should be unique per password/account.

Mistake 3: thinking salt is secret

Salt is usually stored with the hash. Pepper is the optional secret.

Mistake 4: ignoring parameters

Algorithm choice matters, but cost parameters matter too.

Practice problems

  1. Why are passwords different from random cryptographic keys?
    Show solution

    Passwords are human-chosen and often low-entropy or pattern-based, while cryptographic keys are generated randomly and have much higher entropy.

  2. What is a salt?
    Show solution

    A unique, usually public value stored with a password hash to separate users/accounts and prevent identical passwords from having identical hashes.

  3. Why is a fast hash bad for password storage?
    Show solution

    Attackers can test many guesses per second during offline attacks.

  4. What does memory-hard mean?
    Show solution

    It means the function requires significant memory to compute, making large-scale parallel guessing more expensive.

  5. What is the difference between salt and pepper?
    Show solution

    Salt is public and stored with the hash; pepper is secret and stored separately from the database.

  6. What is the difference between online and offline guessing?
    Show solution

    Online guessing tests passwords against a live service; offline guessing tests guesses against stolen hashes on attacker-controlled hardware.

Why this matters for cryptography

Password hashing is where cryptography meets human weakness. The mathematics cannot assume the secret is uniformly random. Instead, the system must slow attackers, separate users with salts, choose memory-hard tools when appropriate, and store enough parameters to verify later and upgrade over time.

  • Passwords are usually weaker than random keys.
  • Do not store plaintext passwords.
  • Do not use fast unsalted hashes for password storage.
  • Use unique salts.
  • Use slow or memory-hard password hashing.
  • Store algorithm, parameters, salt, and hash.
  • Argon2id is a strong modern password hashing choice when available.
  • HKDF and password hashing solve different problems.

One-page recap

What to remember
  • Passwords are usually lower-entropy than random cryptographic keys.
  • Password hashing makes guesses expensive.
  • Salt is unique, public, and stored with the password hash.
  • Salt prevents identical passwords from producing identical stored hashes.
  • Salt does not make guessing slow by itself.
  • Fast hashes such as raw SHA-256 are bad for password storage.
  • Offline guessing happens after attackers steal password hashes.
  • Slow hashing reduces guesses per second.
  • Memory-hard hashing increases RAM cost per guess.
  • Argon2id, scrypt, bcrypt, and PBKDF2 are password-focused tools.
  • Pepper is an optional server-side secret stored separately.
  • HKDF is for deriving keys from strong secret material, not for ordinary password storage.
