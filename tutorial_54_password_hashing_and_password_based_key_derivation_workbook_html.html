How to use this workbook

Keep this distinction sharp: HKDF expands strong secrets, while password hashing defends weak human secrets against guessing. Password hashing cannot make a bad password perfect, but it can make each guess more expensive.

Note for self-study

The central idea is cost. Password hashing protects against guessing by making each password attempt expensive, while salts separate users and peppers add optional server-side secret protection.

Part A: Warm-up vocabulary

QuestionAnswer
A1. What is a password?A human-chosen secret, usually lower entropy than a random cryptographic key.
A2. What is a salt?A unique, usually public value stored with a password hash.
A3. What is the stored hash?The password hashing output used as a verifier instead of storing the password itself.
A4. What is a pepper?An additional server-side secret stored separately from the password database.
A5. What is a work factor?A cost setting controlling time, iterations, or memory cost.
A6. What is an offline attack?An attack where guesses are tested against stolen hashes away from the live server.
A7. What does memory-hard mean?The function requires significant memory to compute.
A8. What is Argon2id?A modern memory-hard password hashing function and common strong default when available.
A9. What does PBKDF stand for?Password-Based Key Derivation Function.

Part B: Passwords are different from random keys

Secret typeTypical sourceAttacker strategy
Random keyCryptographic random generatorBrute force is usually infeasible.
Human passwordMemory, habit, reuse, patternsDictionary, mutation, leaked password lists, guessing.

B1 sample answer. A weak password still comes from a small or guessable space. Password hashing cannot change the user’s original entropy; it can only make each guess slower or more expensive.

OrderAction

E1 sample answer. Offline guessing is dangerous because attackers do not need to interact with the live server. Rate limits and account lockouts no longer stop them from testing many guesses on their own hardware.

Part F: Why slow hashing helps

Design goalWhy?
Slow enough for attackersReduces guesses per second.
Fast enough for real usersLogin should remain acceptable.
Adjustable costCan increase difficulty as hardware improves.
Memory cost if possibleMakes GPU/ASIC mass guessing more expensive.

F1 sample answer. A single user login can tolerate a small delay, but an attacker trying millions or billions of guesses pays that delay repeatedly. That turns small cost into large attack expense.

Part G: Memory hardness

Cost typeWhat it charges the attacker
Time costMore computation per guess.
Memory costMore RAM per guess.
Parallelism settingControls how work may be split across threads/cores.

G1 sample answer. Attackers often rely on massive parallel guessing. If each guess needs meaningful RAM, then running many guesses at once becomes more expensive and harder to scale.

H1 sample answer. Password hashing depends on parameters such as time cost, memory cost, iterations, and parallelism. A good algorithm with weak settings may still be too cheap to attack.

J1 sample answer. Pepper can add protection if the database is stolen without the application secret. It also creates a responsibility to manage, protect, rotate, and recover that secret carefully.

Part K: Password-based key derivation

K1 sample answer. A file vault may need to turn a user password into an encryption key. A password-based KDF stretches the password into key material with salt and cost parameters.

K2 sample answer. A weak password still comes from a small guessable space. The KDF makes guessing more expensive, but it does not add true randomness to the user’s choice.

Part L: Online versus offline attacks

Attack typeWhere guesses are testedMain defenses
Online guessingAgainst the live login service.Rate limiting, lockouts, multifactor authentication, monitoring.
Offline guessingAgainst stolen hashes on attacker hardware.Strong password hashing, salts, memory hardness, strong passwords.

L1 sample answer. Online guessing is like knocking on the front door where the server can slow, block, or monitor attempts. Offline guessing means the attacker stole the verifier and can test guesses on their own hardware without rate limits.

Part O: Concept checks

O1 sample answer. Password hashing stores a verifier for a password instead of the password itself. A good password hash uses a unique salt and cost parameters so attackers must spend significant time or memory for each guess.

O2 sample answer. A salt is a unique public value stored with the hash to separate accounts. A pepper is an optional secret stored separately from the database. A work factor controls how expensive each hash computation is.

O3 sample answer. Offline guessing means attackers can test guesses on their own machines after stealing hashes. The live server can no longer rate-limit those guesses, so the hash itself must be expensive to compute.

O4 sample answer. This tutorial explained that passwords are weaker than random keys and need special protection. Password hashing uses salts and cost parameters to make offline guessing harder. Memory-hard functions such as Argon2id and scrypt make mass guessing more expensive. Password records should store algorithm, parameters, salt, and hash, but never plaintext passwords. HKDF and password hashing solve different problems.
