Randomness, Entropy, and Secure Random Number Generation

Why cryptography depends on unpredictable values: entropy, random keys, salts, nonces, initialization vectors, CSPRNGs, and the quiet catastrophe of bad randomness.

Roadmap Previous Tutorial Practice One-page recap

Why this tutorial exists

Purpose

Tutorial 54 showed why password systems need special protection because human secrets are weak. This tutorial studies the opposite ideal: strong cryptographic randomness. Keys, salts, nonces, challenges, and many protocol values depend on random generation or unpredictable state.

Randomness is the well cryptography drinks from. If the well is poisoned, the castle can have perfect walls and still fall because every key was copied before the doors were built.

What you should know first

You should know that cryptographic systems rely on secret keys, unique nonces, salts, and unpredictable challenges. You should also know that ordinary programming-language random functions are often designed for simulations or games, not for cryptographic security.

Randomness versus pseudorandomness

Definition: pseudorandomness

A pseudorandom generator is deterministic, but its output should look random to someone who does not know its internal state or seed.

TypeSourceCryptographic question
True randomnessPhysical uncertainty, hardware events, operating-system entropy sources.Is the source measured and handled safely?
PseudorandomnessDeterministic algorithm from secret seed/state.Is it secure even after many outputs are observed?

In practice, computers collect uncertain events into an entropy pool, then use a secure generator to produce random bytes for applications.

Entropy: measuring uncertainty

Entropy is a way to talk about uncertainty. One fair coin flip has one bit of entropy because there are two equally likely outcomes.

\[ \text{possibilities}=2^n \]

If a value has \(n\) bits of ideal entropy, an attacker faces about \(2^n\) equally likely possibilities.

EntropyPossible valuesFriendly meaning
1 bit\(2^1=2\)One fair coin flip.
8 bits\(2^8=256\)One random byte.
32 bits\(2^{32}\)Large for humans, small for attackers.
128 bits\(2^{128}\)Strong cryptographic scale.
256 bits\(2^{256}\)Enormous security margin for random guessing.

Uniformity and unpredictability

Good cryptographic randomness should avoid bias and be unpredictable. A generator can look messy but still be predictable, and a value can be unique but not random.

PropertyMeaningExample
UniformAll values are equally likely.Each 128-bit key has the same chance.
UnpredictableAttacker cannot guess the next value.Secure session key.
UniqueValue does not repeat in context.Counter nonce.
SecretAttacker does not know it.Private key or symmetric key.
Important distinction

A counter can be unique without being unpredictable. A key must be unpredictable and secret. A salt should be unique, but usually does not need secrecy.

CSPRNG: cryptographic random generation

Definition: CSPRNG

A Cryptographically Secure Pseudorandom Number Generator is a generator designed so that its outputs are unpredictable to attackers, even if they observe many previous outputs.

What bad randomness breaks

Bad randomness problemPossible consequence
Predictable keysAttackers can guess or reconstruct secret keys.
Repeated noncesAEAD or stream-like modes may fail.
Repeated saltsPassword hashes lose some separation benefit.
Predictable challengesAuthentication protocols may be replayed or precomputed.
Low-entropy seedsCSPRNG output may become guessable.
Design warning

A cryptosystem with weak randomness can fail even if every algorithm inside it is famous, standardized, and mathematically elegant.

Modulo bias

A common mistake is taking random bytes and using modulo reduction to force them into a smaller range. If the range does not divide evenly into the source size, some outputs become more likely than others.

\[ x=randomByte()\bmod 10 \]

Since one byte has 256 possible values and 10 does not divide 256 evenly, values 0 through 5 occur one extra time compared with 6 through 9.

Output rangeHow many byte values map there?
0 through 526 each
6 through 925 each
Safer habit

Use secure library functions for bounded random integers. Good libraries use rejection sampling or another method to avoid bias.

Practical random API habits

Randomness health and failure thinking

Secure systems often include health checks, reseeding behavior, and careful handling of generator state. The details are subtle, but the design goal is straightforward: do not let weak or repeated state silently produce security-critical values.

ConcernWhy it matters
Early boot entropyDevices may start before enough uncertainty is available.
Virtual machine cloningCloned systems may duplicate random-generator state.
Embedded devicesMay lack strong entropy sources.
State compromiseIf generator state leaks, future or past outputs may be at risk depending on design.

Toy example: weak seed, weak keys

Suppose a toy system seeds a PRNG using the current second of the day. There are only 86,400 possible seconds in a day.

\[ 86400 < 2^{17} \]

That means the attacker might only need to test fewer than \(2^{17}\) possible seeds, which is tiny compared with a 128-bit key space.

Seed sourceApproximate possibilitiesSecurity result
Second of the dayLess than \(2^{17}\)Far too guessable.
128-bit secure random seed\(2^{128}\)Cryptographic scale.

Common mistakes

Mistake 1: using ordinary random functions for keys

Game/simulation random functions are not designed to resist attackers.

Mistake 2: using timestamps as secrets

Timestamps are easy to guess or narrow down.

Mistake 3: confusing uniqueness with unpredictability

A counter is unique but predictable. That may be fine for some nonces, but not for keys.

Mistake 4: introducing modulo bias

Use library functions for bounded random values rather than careless modulo reduction.

  2. How many possible values are represented by \(n\) bits of ideal entropy?
    Show solution

  3. What is a CSPRNG?
    Show solution

    A Cryptographically Secure Pseudorandom Number Generator, designed to produce outputs unpredictable to attackers.

  4. Why is a timestamp a bad secret seed?
    Show solution

    Because timestamps are guessable and usually come from a small search range.

  5. Does a nonce always need to be secret?
    Show solution

    No. A nonce usually needs uniqueness under the same key; unpredictability depends on the scheme.

  6. What is modulo bias?
    Show solution

    Modulo bias occurs when random values are reduced into a range that does not divide evenly, causing some outputs to be more likely than others.

Why this matters for cryptography

Randomness is not a side detail. It creates the secrets and freshness values that the rest of cryptography depends on. Strong algorithms cannot rescue predictable keys, repeated nonces, biased random choices, or weak generator seeds.

  • Entropy measures uncertainty.
  • \(n\) bits of entropy means about \(2^n\) possible values.
  • Keys need secrecy and unpredictability.
  • Salts need uniqueness, not secrecy.
  • Nonces usually need uniqueness; unpredictability depends on the mode.
  • CSPRNGs are designed for cryptographic randomness.
  • Ordinary PRNGs should not be used for cryptographic keys.
  • Modulo bias can make some random choices more likely than others.
