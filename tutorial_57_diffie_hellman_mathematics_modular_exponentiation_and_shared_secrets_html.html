Diffie-Hellman Mathematics: Modular Exponentiation and Shared Secrets

A step-by-step look at classical Diffie-Hellman: modular exponentiation, public parameters, private exponents, matching shared secrets, and the discrete logarithm problem.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 56 introduced public-key cryptography and previewed Diffie-Hellman. This tutorial slows down the mathematics. We will use small toy numbers to see exactly how Alice and Bob compute the same shared secret while an eavesdropper sees only public values.

Diffie-Hellman feels like a stage illusion at first: Alice has one secret, Bob has another, they shout public numbers across the room, and somehow both end up holding the same hidden value. The trick is not magic; it is modular exponentiation wearing a velvet cape.

What you should know first

You should know modular arithmetic, especially the idea that values wrap around after a modulus. You should also know that public-key cryptography depends on operations that are easy in one direction and hard to reverse without secret information.

Modular exponentiation

Definition: modular exponentiation

Modular exponentiation means computing a power and then reducing it modulo a number.

\[ g^a \bmod p \]

This means: multiply \(g\) by itself \(a\) times, then keep the remainder after division by \(p\). In real cryptography, the numbers are huge. In this tutorial, we use tiny numbers so the machinery is visible.

ExpressionPlain powerModulo result
\(5^1 \bmod 23\)55
\(5^2 \bmod 23\)252
\(5^3 \bmod 23\)12510
\(5^4 \bmod 23\)6254

Diffie-Hellman setup

Classical Diffie-Hellman begins with public parameters. Everyone may know them.

ParameterPublic or private?Meaning
\(p\)PublicPrime modulus.
\(g\)PublicGenerator or base.
\(a\)PrivateAlice’s secret exponent.
\(b\)PrivateBob’s secret exponent.
Toy warning

Small values are useful for learning but completely insecure. Real Diffie-Hellman uses carefully chosen large parameters or elliptic-curve groups.

Computing public values

Alice and Bob each choose a private exponent. They do not reveal these. Instead, they publish modular exponentiation results.

\[ A=g^a \bmod p \] \[ B=g^b \bmod p \]
PartyPrivate exponentPublic value
Alice\(a\)\(A=g^a\bmod p\)
Bob\(b\)\(B=g^b\bmod p\)

Computing the shared secret

Alice receives Bob’s public value \(B\) and raises it to her private exponent \(a\). Bob receives Alice’s public value \(A\) and raises it to his private exponent \(b\).

\[ S_A=B^a\bmod p \] \[ S_B=A^b\bmod p \]

Since \(B=g^b\) and \(A=g^a\), the two computations match:

\[ S_A=(g^b)^a\bmod p=g^{ba}\bmod p \] \[ S_B=(g^a)^b\bmod p=g^{ab}\bmod p \] \[ g^{ab}=g^{ba} \]
Why they match

Alice and Bob do the exponents in opposite order, but multiplication of the exponents gives the same product. The shared secret meets in the middle.

Worked toy example

Let the public parameters be:

\[ p=23, \quad g=5 \]

Alice chooses \(a=6\). Bob chooses \(b=15\).

StepAliceBob
Private exponent\(a=6\)\(b=15\)
Public value\(A=5^6\bmod23=8\)\(B=5^{15}\bmod23=19\)
Shared secret\(S_A=19^6\bmod23=2\)\(S_B=8^{15}\bmod23=2\)

Alice and Bob both get the shared secret \(S=2\).

What does the eavesdropper see?

An eavesdropper can see public parameters and public values:

Visible to eavesdropperHidden from eavesdropper
\(p\)\(a\)
\(g\)\(b\)
\(A=g^a\bmod p\)shared secret \(S\)
\(B=g^b\bmod p\)derived traffic keys

To compute the shared secret from the public information, the attacker would need to solve the hard reverse problem in the group.

The discrete logarithm problem

Definition: Discrete Logarithm Problem

Given \(g\), \(p\), and \(A=g^a\bmod p\), the Discrete Logarithm Problem is to recover \(a\).

\[ A=g^a\bmod p \quad \text{easy if you know } a \] \[ a=\log_g(A) \bmod p \quad \text{hard in suitable groups} \]

In ordinary arithmetic, logarithms undo exponentiation. In modular arithmetic, finding the exponent from the result can be very hard when parameters are chosen properly.

Memory hook

Modular exponentiation is a blender. The discrete logarithm problem asks someone to recover the original fruit arrangement after the smoothie is poured.

Why the toy example is insecure

With \(p=23\), an attacker can simply try possible exponents until one matches Alice’s public value. That is tiny enough to do by hand.

Using the shared secret

The raw Diffie-Hellman shared value should usually go into a Key Derivation Function. The KDF binds the exchange to context and derives usable keys.

\[ shared\ secret = DH(a,B)=DH(b,A) \] \[ traffic\ keys=KDF(shared\ secret, transcript, roles, algorithm\ identifiers) \]
KDF inputWhy include it?
Shared secretCore secret created by the exchange.
TranscriptBinds keys to messages exchanged.
RolesSeparates client/server directions.
Algorithm identifiersPrevents confusion about negotiated algorithms.

Again: authentication still matters

Diffie-Hellman protects against passive eavesdroppers, but unauthenticated Diffie-Hellman is vulnerable to man-in-the-middle attacks.

SituationProblemNeeded fix
Alice and Bob exchange public values without authentication.Attacker can substitute their own public values.Authenticate public values or transcript.
Alice does not know whether \(B\) belongs to Bob.Alice may share a secret with the attacker.Certificates, signatures, fingerprints, or trusted keys.
Shared secret is not bound to context.Keys may be confused across sessions or protocols.Use KDF with transcript/context.

Common mistakes

Mistake 1: thinking public parameters are secret

\(p\), \(g\), \(A\), and \(B\) are public. The private exponents and derived secrets must remain protected.

Mistake 2: thinking small examples are secure

Toy examples are for learning. Real parameters must be large and carefully chosen.

Mistake 3: using raw shared secret directly

Feed the shared secret into a KDF with transcript and context.

Mistake 4: ignoring authentication

Diffie-Hellman without authentication can be intercepted by an active attacker.

Practice problems

  1. In Diffie-Hellman, which values are public?
    Show solution

    \(p\), \(g\), Alice’s public value \(A\), and Bob’s public value \(B\) are public.

  2. Which values are private?
    Show solution

    Alice’s private exponent \(a\), Bob’s private exponent \(b\), and the derived shared secret should remain secret.

  3. Why do Alice and Bob compute the same shared secret?
    Show solution

    Because \((g^b)^a=(g^a)^b=g^{ab}\).

  4. What is the Discrete Logarithm Problem?
    Show solution

    Given \(g\), \(p\), and \(A=g^a\bmod p\), recover \(a\).

  5. Why are small Diffie-Hellman examples insecure?
    Show solution

    Because an attacker can try possible exponents until the public value matches.

  6. What should happen to the raw shared secret?
    Show solution

    It should usually be fed into a KDF with transcript and context to derive traffic keys.

Why this matters for cryptography

Diffie-Hellman is one of the great bridges in cryptographic history. It shows how two parties can build a shared secret over a public channel. The mathematics is modular exponentiation. The security rests on the hardness of reversing that operation in suitable groups. The engineering still requires authentication, parameter validation, and key derivation.

  • Diffie-Hellman uses public parameters \(p\) and \(g\).
  • Alice keeps \(a\) private and sends \(A=g^a\bmod p\).
  • Bob keeps \(b\) private and sends \(B=g^b\bmod p\).
  • Alice computes \(B^a\bmod p\).
  • Bob computes \(A^b\bmod p\).
  • Both get \(g^{ab}\bmod p\).
  • The hard reverse problem is the Discrete Logarithm Problem.
  • Raw shared secrets should feed a KDF.
  • Authentication is needed against active attackers.

What to remember
  • Diffie-Hellman is a key exchange protocol.
  • It lets Alice and Bob compute the same shared secret over a public channel.
  • Classical Diffie-Hellman uses modular exponentiation.

  • \(g\) is a public generator/base.
  • \(a\) and \(b\) are private exponents.
  • \(A=g^a\bmod p\) is Alice’s public value.
  • \(B=g^b\bmod p\) is Bob’s public value.

  • Bob computes \(S=A^b\bmod p\).
  • The values match because \((g^b)^a=(g^a)^b=g^{ab}\).
  • The Discrete Logarithm Problem asks attackers to recover the secret exponent from the public value.
  • Small examples are insecure and only for learning.
