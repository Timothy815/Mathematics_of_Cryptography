RSA Worked Examples: Encryption, Decryption, and Signature Intuition

Hands-on toy RSA examples for seeing public exponentiation, private exponentiation, encryption/decryption reversal, signature verification intuition, and why padding separates learning math from real security.

Why this tutorial exists

Purpose

Tutorial 58 built the RSA machine. This tutorial turns the crank slowly. We will encrypt, decrypt, sign, and verify with tiny toy values so that the inverse-exponent structure becomes visible.

RSA is best learned with arithmetic smoke cleared away. Tiny numbers let us watch the gears rotate. Real RSA uses enormous numbers and carefully designed padding, but the toy examples reveal the skeleton beneath the armor.

What you should know first

You should know that RSA uses a public key \((n,e)\) and a private exponent \(d\). You should also know that \(d\) is chosen so that \(ed\equiv1\pmod{\varphi(n)}\), which lets the private operation undo the public operation under RSA’s modular structure.

Textbook RSA encryption shape

Textbook RSA encryption raises the message number to the public exponent:

\[ c\equiv m^e\pmod n \]

With \((n,e)=(55,3)\), this becomes:

\[ c\equiv m^3\pmod{55} \]
Language precision

This is textbook RSA. Real RSA encryption uses padding, usually RSA-OAEP, before the RSA operation happens.

Example 1: encrypt \(m=7\)

Start with message number:

\[ m=7 \]

Encrypt using \(e=3\) and \(n=55\):

\[ c\equiv7^3\pmod{55} \] \[ 7^3=343 \] \[ 343=6\cdot55+13 \] \[ c=13 \]
StepValue

Raise to public exponent\(7^3=343\)

Example 1: decrypt \(c=13\)

Textbook RSA decryption raises the ciphertext to the private exponent:

\[ m\equiv c^d\pmod n \]

Here:

\[ m\equiv13^{27}\pmod{55} \]

Computing \(13^{27}\) directly is enormous. Modular arithmetic lets us reduce along the way. A useful repeated-squaring path is:

PowerReduction mod 55
\(13^1\)13
\(13^2=169\)4
\(13^4\equiv4^2=16\)16
\(13^8\equiv16^2=256\)36
\(13^{16}\equiv36^2=1296\)31

Since \(27=16+8+2+1\):

\[ 13^{27}=13^{16}\cdot13^8\cdot13^2\cdot13^1 \] \[ 13^{27}\equiv31\cdot36\cdot4\cdot13\pmod{55} \] \[ 31\cdot36\cdot4\cdot13\equiv7\pmod{55} \]

The original message returns:

\[ m=7 \]

Example 2: encrypt and decrypt \(m=12\)

Encrypt with \((n,e)=(55,3)\):

\[ c\equiv12^3\pmod{55} \] \[ 12^3=1728 \] \[ 1728\equiv23\pmod{55} \]

So:

\[ c=23 \]

Decryption should return \(12\):

\[ 23^{27}\equiv12\pmod{55} \]

Message limits in toy RSA

In textbook RSA, the message representative must be an integer modulo \(n\). That means:

\[ 0\le m

In our toy example, \(n=55\), so message representatives must be from 0 through 54. Real RSA does not directly encrypt arbitrary long messages. Instead, real systems use padding and usually encrypt a symmetric key or use hybrid encryption.

RSA signature intuition

RSA signatures use the private exponent to create a value that the public exponent can verify.

\[ s\equiv m^d\pmod n \] \[ m\equiv s^e\pmod n \]

In real RSA signatures, you do not sign raw messages this way. A real signature scheme hashes and encodes the message first, then uses a secure padding scheme such as RSA-PSS.

Toy signature example with \(m=7\)

Sign with the private exponent \(d=27\):

\[ s\equiv7^{27}\pmod{55} \]

The result is:

\[ s=28 \]

Verify with public exponent \(e=3\):

\[ s^e\equiv28^3\pmod{55} \] \[ 28^3\equiv7\pmod{55} \]

The verification recovers the message representative \(7\), so the toy signature checks out.

Better wording

Avoid saying “encrypt with the private key.” Say “sign with the private key” and “verify with the public key.” The math may look related, but the protocol goal is different.

Why padding matters

Textbook RSA has dangerous properties. It is deterministic: the same message encrypted with the same key gives the same ciphertext. It also has mathematical structure that attackers can exploit.

Textbook RSA weaknessWhy dangerous?
Deterministic encryptionSame message gives same ciphertext.
No randomnessAttackers can test guesses for small message spaces.
Malleable structureMathematical relationships can leak or manipulate meaning.
Raw message handlingReal messages need encoding, padding, and length handling.
Rule of thumb

Textbook RSA is a blackboard diagram. Real RSA needs OAEP for encryption and PSS for signatures.

RSA in hybrid encryption

Real systems often do not use RSA to encrypt the full message. Instead, RSA protects or transports a symmetric key, and the symmetric key protects the actual data.

Generate random symmetric key K
Encrypt data using AEAD with K
Encrypt or wrap K using RSA-OAEP

Common mistakes

Mistake 1: treating toy RSA as secure

Tiny moduli are factorable and useful only for learning.

Mistake 2: using textbook RSA directly

Textbook RSA lacks the padding and security structure real systems require.

Mistake 3: saying signatures are “private-key encryption”

Signatures are a separate primitive with different security goals and formatting rules.

Mistake 4: trying to RSA-encrypt large messages directly

RSA operates on bounded-size representatives. Use hybrid encryption for real data.

Why this matters for cryptography

RSA’s formulas are simple to write but dangerous to misuse. The worked examples show why the inverse exponents recover the message and why the same exponent machinery can support signature verification intuition. The security lesson is just as important: real cryptography is not only the core equation; it is the equation plus padding, encoding, randomness, parameter size, and protocol context.

  • RSA encryption shape: \(c\equiv m^e\pmod n\).
  • RSA decryption shape: \(m\equiv c^d\pmod n\).
  • RSA signing intuition: \(s\equiv m^d\pmod n\).
  • RSA verification intuition: \(m\equiv s^e\pmod n\).
  • Message representatives must be less than \(n\).
  • Textbook RSA is for learning, not direct use.
  • Real RSA encryption uses OAEP.
  • Real RSA signatures use PSS or another secure signature scheme.
  • Real systems usually combine RSA with symmetric encryption.
