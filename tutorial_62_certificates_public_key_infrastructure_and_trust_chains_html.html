Certificates, Public Key Infrastructure, and Trust Chains

How public keys become trustworthy: certificates, certificate authorities, root stores, intermediate certificates, trust chains, hostname validation, expiration, revocation, and Certificate Transparency.

🏠 Dashboard

Why this tutorial exists

Purpose

Earlier tutorials showed how public-key mathematics can create shared secrets and digital signatures. But one question remains: how do you know a public key belongs to the person, server, or organization you think it belongs to? Public Key Infrastructure answers that question with certificates and chains of trust.

A public key by itself is a nameless key on a table. A certificate is the tag tied to it, stamped by an authority that says, “This key belongs to this subject, for this purpose, during this time.” PKI is the bureaucracy of trust, tiny wax seals all the way up.

What you should know first

You should know public/private key pairs, digital signatures, ECDH, and the problem of authentication. A certificate does not make encryption “stronger” by itself. Its main job is to bind an identity or name to a public key using a trusted signature.

Certificate big picture

A certificate is a signed statement about a public key. It says, in effect: this public key belongs to this subject, is valid during this date range, is allowed for these uses, and was signed by this issuer.

Certificate fieldQuestion it answers

Subject public keyWhat public key is being certified?

Validity periodWhen is this certificate valid?
Subject Alternative NameWhich domain names or identities does this certificate cover?
Key usage / extended key usageWhat may this certificate be used for?
SignatureHow can the certificate be verified?
Definition: certificate

A certificate is a signed data object that binds a public key to a subject and includes metadata such as issuer, validity period, permitted uses, and names covered.

The signature on a certificate

A CA signs a certificate using the CA’s private key. A client verifies that signature using the CA’s public key.

\[ signature = Sign_{CA\ private\ key}(certificate\ data) \] \[ Verify_{CA\ public\ key}(certificate\ data, signature)=valid/invalid \]

This is the same digital-signature idea from earlier tutorials. The certificate’s contents are protected by the issuer’s signature. If someone changes the certificate data, the signature check fails.

Memory hook

The certificate data is the letter. The CA’s signature is the wax seal. Break or rewrite the letter, and the seal no longer proves it.

Public Key Infrastructure

Definition: Public Key Infrastructure

Public Key Infrastructure, or PKI, is the system of certificate authorities, certificates, policies, root stores, validation rules, and revocation mechanisms used to bind public keys to trusted identities.

PKI componentJob
Root CAActs as a trust anchor.
Intermediate CAIssues certificates while keeping root keys safer.
Leaf certificateIdentifies a server, user, device, or service.
Root storeHolds trusted root certificates.
Validation softwareChecks signatures, names, dates, usage, and revocation information.

Chain of trust

A website usually does not present a certificate signed directly by a root CA. Instead, it presents a leaf certificate signed by an intermediate CA. That intermediate certificate is signed by a root CA. The client already trusts the root CA because it is in the root store.

Root stores and trust anchors

A root store is a list of root CA certificates that a system, browser, or application trusts. A root certificate is usually self-signed, but it is not trusted because it signed itself. It is trusted because the root store says it is trusted.

QuestionAnswer
Why trust a root CA?Because it is included as a trust anchor in the root store.
Who maintains root stores?Operating systems, browsers, applications, and platform vendors.
Can roots be removed?Yes, root programs can distrust or remove CAs that violate rules.
Self-signed does not automatically mean trusted

A self-signed certificate can be useful in a private environment, but public browsers trust only roots that are accepted into their trust stores or explicitly installed by the user or organization.

Hostname validation

A certificate must match the name you are trying to visit. For websites, the client checks the hostname against the certificate’s Subject Alternative Name extension.

Browser visitsCertificate SAN containsResult
www.example.comwww.example.comName matches.
shop.example.comwww.example.comName mismatch.
mail.example.com*.example.comWildcard may match one label.
a.b.example.com*.example.comWildcard usually does not match multiple labels.
Critical point

A certificate can be signed by a trusted CA and still fail validation if the hostname does not match.

Validity periods and key usage

Certificates are not valid forever. They have a not-before time and a not-after time. They also contain usage rules that say what the certificate is allowed to do.

FieldQuestion answered
Not BeforeHas the certificate become valid yet?
Not AfterHas the certificate expired?
Key UsageWhat cryptographic operations are allowed?
Extended Key UsageWhat application purposes are allowed, such as server authentication?

Validation is a bundle of checks, not just one signature check. The certificate must be signed correctly, within its valid date range, appropriate for the intended use, and match the expected name.

Revocation

Sometimes a certificate must be distrusted before its expiration date. For example, the private key may be stolen, the certificate may have been issued incorrectly, or the domain owner may no longer control the domain.

MechanismMeaningBasic idea

Expiration is a badge naturally timing out. Revocation is security taking the badge away early.

Certificate Transparency

Certificate Transparency, or CT, creates public logs of issued certificates. These logs make it easier to detect certificates that were issued mistakenly or maliciously.

How this fits into TLS

In modern TLS, certificates are usually used to authenticate the server. The actual session keys typically come from an ephemeral key exchange, such as ECDHE. So the certificate does not usually encrypt all the traffic itself. It authenticates the handshake.

1. Client connects to server.
2. Server sends certificate chain.
3. Client validates the certificate chain and hostname.
4. Client and server perform authenticated key exchange.
5. KDF derives traffic keys.
6. Symmetric AEAD protects the application data.
Important distinction

Certificates usually authenticate public keys and identities. Symmetric traffic keys protect the actual data after the handshake.

Private and internal PKI

Public websites usually use publicly trusted certificates. Organizations can also run private PKI for internal services, devices, VPNs, Wi-Fi authentication, code signing, and more.

PKI typeTrusted byTypical use
Public PKIBrowsers and operating systems by defaultPublic HTTPS websites.
Private PKIDevices where the organization installs the rootInternal systems, enterprise devices, VPNs, labs.
Self-signed leaf certificateOnly if explicitly trustedTesting, labs, small private systems.

Common mistakes

Mistake 1: thinking “signed” means “safe”

A certificate can be signed but still fail hostname, date, usage, or trust-chain checks.

Mistake 2: trusting any self-signed certificate

Self-signed certificates are not automatically trusted. Trust depends on the root store or explicit installation.

Mistake 3: ignoring hostname validation

The certificate must match the name being visited.

Mistake 4: thinking certificates encrypt all traffic by themselves

Certificates authenticate keys and identities. Session traffic is usually protected by symmetric keys derived during the handshake.

Practice problems

  1. What does a certificate bind together?
    Show solution

    A certificate binds a subject or name to a public key, along with metadata, under an issuer’s signature.

  2. What is a certificate authority?
    Show solution

    A Certificate Authority is an entity that issues and signs certificates.

  3. What is a chain of trust?
    Show solution

    A sequence of certificates where each certificate is signed by the next issuer, ending at a trusted root.

  4. Why does hostname validation matter?
    Show solution

    It ensures the certificate actually covers the site or service name being visited.

  5. What is certificate revocation?
    Show solution

    Revocation marks a certificate as no longer valid before its expiration date.

  6. What does a certificate usually do in modern TLS?
    Show solution

    It authenticates the server’s public key/identity during the handshake; traffic is then protected by derived symmetric keys.

Why this matters for cryptography

Public-key cryptography solves powerful mathematical problems, but authentication is the human-system layer that tells us whose key we are using. Certificates and PKI are the bridge between mathematics and trust. Without that bridge, a perfect key exchange could still happen with the wrong party.

  • Certificates bind public keys to identities or names.
  • Certificate authorities sign certificates.
  • Clients verify certificate signatures using issuer public keys.
  • Trust chains usually run from leaf to intermediate to root.
  • Root stores define trust anchors.
  • Hostname validation checks whether the certificate covers the requested name.
  • Validity periods limit when certificates are accepted.
  • Key usage limits what a certificate may do.
  • Revocation attempts to distrust certificates early.
  • Certificate Transparency makes issuance more visible.
  • Modern TLS uses certificates for authentication and key exchange for session keys.

What to remember
  • A certificate is a signed statement about a public key.
  • A certificate binds a subject/name to a public key.
  • A Certificate Authority, or CA, signs certificates.
  • A root CA is trusted because it is in a root store.
  • An intermediate CA is signed by another CA and often issues leaf certificates.
  • A leaf certificate is the end-entity/server certificate.
  • A chain of trust connects leaf → intermediate → trusted root.
  • Certificate signatures protect certificate contents from tampering.
  • Hostname validation checks whether the certificate covers the requested name.
  • Certificates have validity periods.
  • Key usage and extended key usage restrict certificate purposes.
  • Revocation can distrust a certificate before expiration.
  • OCSP and CRLs support revocation checking.
  • Certificate Transparency logs make certificate issuance more visible.
  • In modern TLS, certificates usually authenticate the handshake; symmetric traffic keys protect the data.
