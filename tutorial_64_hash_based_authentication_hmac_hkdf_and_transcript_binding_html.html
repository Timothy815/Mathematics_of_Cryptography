Hash-Based Authentication: HMAC, HKDF, and Transcript Binding

How hashes become keyed proof-machines: Message Authentication Codes, HMAC, HKDF extract-and-expand, domain separation, labels, context, and transcript binding in secure protocols.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 63 showed that TLS uses transcript hashes, MACs, and KDFs inside the handshake. This tutorial slows down those pieces. We will see how an ordinary hash becomes a keyed authenticator with HMAC, and how HKDF turns raw shared material into clean, separated keys.

A plain hash is a fingerprint. HMAC is a fingerprint stamped with a secret ring. HKDF is a careful key-forge: it melts raw secret ore into uniform metal, then casts separate tools for separate jobs.

What you should know first

You should know that hash functions produce fixed-size digests, that TLS keeps a transcript hash, and that key exchanges produce raw shared material. You should also know that real protocols need different keys for different purposes, not one master blob reused everywhere.

Plain hash versus keyed authentication

A plain hash can detect accidental or malicious changes if you already know the correct digest. But anyone can compute a plain hash. That means a plain hash alone does not prove that the sender knew a secret.

ConstructionUses secret key?What it proves
\(H(m)\)NoOnly a digest of the message.
\(HMAC(K,m)\)YesMessage was authenticated by someone who knows \(K\).
Important distinction

A hash gives integrity only if you already trust the digest. A MAC gives integrity and authenticity under a shared secret key.

Message Authentication Codes

Definition: Message Authentication Code

A Message Authentication Code, or MAC, is a keyed function that produces a tag for a message. A verifier with the same secret key can recompute the tag and check whether it matches.

\[ tag=MAC(K,m) \] \[ Verify(K,m,tag)=valid/invalid \]

If the message changes, the tag should no longer verify. If the attacker does not know the key, they should not be able to forge a valid tag for a new message.

HMAC big picture

HMAC is a standard way to build a MAC from a cryptographic hash function. It combines the key and message in a careful two-layer construction rather than merely hashing \(K\parallel m\).

\[ HMAC(K,m)=H((K'\oplus opad)\parallel H((K'\oplus ipad)\parallel m)) \]
PieceMeaning
\(K'\)The key adjusted to the hash block size.
\(ipad\)Inner padding constant.
\(opad\)Outer padding constant.
\(\oplus\)Exclusive OR.
\(\parallel\)Concatenation.
Why not just hash key plus message?

Naively hashing \(K\parallel m\) can interact badly with some hash constructions and protocol assumptions. HMAC was designed to be robust and analyzable.

HKDF big picture

Definition: HKDF

HKDF, or HMAC-based Key Derivation Function, is a two-stage process that extracts a strong pseudorandom key from input material and expands it into one or more context-labeled output keys.

StageFormula shapePurpose
Extract\(PRK=HMAC(salt,IKM)\)Condense raw input into a clean pseudorandom key.
Expand\(OKM=HKDF\text{-}Expand(PRK,info,L)\)Generate output key material for specific purposes.
Memory hook

Extract is the refining furnace. Expand is the casting mold.

HKDF-Extract

The extract stage takes input keying material, which may not be perfectly uniform, and produces a pseudorandom key.

\[ PRK=HMAC(salt,IKM) \]
InputRole
\(IKM\)Raw input keying material, such as an ECDHE shared secret.
\(salt\)Optional non-secret value that helps extraction.
\(PRK\)Pseudorandom key output from extraction.

In protocols, the input keying material may come from ECDHE or another shared secret source. HKDF-Extract turns that material into a more uniform internal key.

HKDF-Expand

The expand stage uses the pseudorandom key, context information, and desired length to produce output key material.

\[ OKM=HKDF\text{-}Expand(PRK,info,L) \]
InputRole
\(PRK\)Clean pseudorandom key from Extract.
\(info\)Context label describing the purpose.
\(L\)Length of output key material needed.
\(OKM\)Output key material.

Domain separation with labels

Domain separation means using different labels or context strings so that keys for one purpose cannot be confused with keys for another purpose.

PurposeExample label / info value
Client handshake traffic key\(info=\text{"client handshake traffic"}\)
Server handshake traffic key\(info=\text{"server handshake traffic"}\)
Client application traffic key\(info=\text{"client application traffic"}\)
Server application traffic key\(info=\text{"server application traffic"}\)
Why labels matter

If two derived keys use the same inputs and labels, they may collide into the same output. Labels make the derivation purpose-aware.

Transcript binding

A protocol transcript is the ordered record of handshake messages. Hashing the transcript and including it in key derivation or authentication binds the keys to the exact conversation.

\[ transcript\ hash=H(m_1\parallel m_2\parallel m_3\parallel\cdots) \] \[ key=HKDF\text{-}Expand(PRK, transcript\ hash\parallel label, L) \]

If an attacker modifies the transcript, the transcript hash changes. Derived keys or authentication checks will no longer match.

Finished messages as HMAC-style checks

In secure handshakes, a Finished message is an authenticated check over the transcript. It proves that the party knows the derived finished key and agrees on the transcript.

\[ Finished=MAC(finished\ key, transcript\ hash) \]

This gives the handshake a final self-test. If either the transcript or keys differ, verification fails.

Common mistakes

Mistake 1: using a plain hash as a MAC

A plain hash is not keyed. Use HMAC or a proper MAC construction for authenticity.

Mistake 2: deriving all keys with the same label

Labels and context separate keys by purpose and direction.

Mistake 3: skipping extraction

Raw shared material may not be in the ideal form for direct expansion. HKDF-Extract helps clean it into a pseudorandom key.

Mistake 4: forgetting transcript binding

Without transcript binding, a protocol may fail to detect changes in negotiation or identity messages.

Practice problems

  1. What is the main difference between \(H(m)\) and \(HMAC(K,m)\)?
    Show solution

    \(H(m)\) is unkeyed; anyone can compute it. \(HMAC(K,m)\) uses a secret key and authenticates the message for those who know \(K\).

  2. What does MAC stand for?
    Show solution

    Message Authentication Code.

  3. What does HKDF stand for?
    Show solution

    HMAC-based Key Derivation Function.

  4. What is the purpose of HKDF-Extract?
    Show solution

    To condense raw input keying material into a clean pseudorandom key.

  5. What is the purpose of HKDF-Expand?
    Show solution

    To derive output key material of a desired length for a specific context or purpose.

  6. Why include a transcript hash in key derivation?
    Show solution

    It binds derived keys to the exact handshake messages, so tampering changes the hash and breaks verification.

Why this matters for cryptography

Hashes, MACs, and KDFs are the connective tissue of modern protocols. They do not just summarize data; when keyed and contextualized, they authenticate messages, derive separated keys, bind transcripts, and make handshakes self-checking. Public-key cryptography often creates the raw shared material, but HMAC and HKDF help turn that material into a disciplined key ecosystem.

  • Plain hashes are not authentication by themselves.
  • MACs authenticate messages under a shared secret key.
  • HMAC builds a robust MAC from a hash function.
  • HKDF uses HMAC for extract-and-expand key derivation.
  • Extract turns raw input keying material into a pseudorandom key.
  • Expand produces purpose-specific output key material.
  • Labels and context provide domain separation.
  • Transcript hashes bind keys to protocol conversations.
  • Finished messages authenticate transcript agreement.

One-page recap

What to remember
  • A plain hash is unkeyed.
  • A MAC uses a secret key to authenticate a message.
  • HMAC is a hash-based MAC construction.
  • A tag verifies only if the verifier has the same secret key and message.
  • HKDF stands for HMAC-based Key Derivation Function.
  • HKDF has two main stages: Extract and Expand.
  • Extract produces \(PRK\), a pseudorandom key.
  • Expand produces \(OKM\), output key material.
  • \(IKM\) means Input Keying Material.
  • \(salt\) helps extraction and is usually non-secret.
  • \(info\) labels the purpose/context of derived keys.
  • Domain separation prevents key-purpose confusion.
  • Transcript binding ties keys to the exact protocol conversation.
  • Finished messages authenticate transcript agreement.
