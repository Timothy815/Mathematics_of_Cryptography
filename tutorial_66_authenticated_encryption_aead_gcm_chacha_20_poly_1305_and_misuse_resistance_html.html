Authenticated Encryption: AEAD, GCM, ChaCha20-Poly1305, and Misuse Resistance

How modern encryption protects both secrecy and integrity: authenticated encryption, associated data, authentication tags, nonce rules, Galois/Counter Mode, ChaCha20-Poly1305, and safer designs when humans make mistakes.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 65 explained nonces, salts, and initialization vectors. This tutorial uses those ideas inside authenticated encryption. Modern systems rarely want encryption alone. They need encryption that also detects tampering, authenticates associated data, and refuses to release plaintext when verification fails.

Encryption without authentication is a locked chest with a soft wax hinge. AEAD hardens both the lid and the hinge: it hides the contents, binds the label, and screams if someone fiddles with either.

What you should know first

You should know symmetric encryption, XOR, nonces, initialization vectors, MACs, HMAC, and TLS traffic keys. You do not need to fully understand Galois-field multiplication yet, but it helps to remember that GCM uses finite-field arithmetic to authenticate ciphertext and associated data.

Why encryption alone is not enough

Encryption hides plaintext, but hiding is not the same as proving that the ciphertext was not changed. Some encryption modes can be malleable: an attacker may alter ciphertext in a way that causes controlled or semi-controlled changes in decrypted plaintext.

GoalEncryption aloneAuthenticated encryption
ConfidentialityYes, if used correctly.Yes, if used correctly.
IntegrityNot necessarily.Yes.
AuthenticityNot necessarily.Yes, under the shared key.
Tamper detectionNot guaranteed.Yes, via tag verification.
Core warning

Do not build “encrypt-only” protocols when attackers can modify ciphertexts. Use authenticated encryption or a proven encrypt-then-MAC construction.

Authenticated Encryption with Associated Data

Definition: AEAD

Authenticated Encryption with Associated Data is a symmetric encryption interface that encrypts plaintext, authenticates ciphertext, and authenticates additional public data without encrypting that public data.

\[ (C,tag)=AEAD\_Encrypt(K,N,AAD,P) \] \[ P=AEAD\_Decrypt(K,N,AAD,C,tag) \]

Decryption should return plaintext only if the tag verifies. If verification fails, the system should reject and reveal no plaintext.

Associated Data

Associated data is public metadata that must not be changed. For example, a network packet may have headers that routers need to read, but the receiver still needs to know those headers were not tampered with.

ScenarioPlaintextAAD
TLS recordApplication dataRecord header or protocol metadata.
Encrypted database fieldStored sensitive valueRow ID, table name, version.
Secure messageMessage bodySender, recipient, timestamp, protocol version.
Memory hook

AAD is the luggage tag. Everyone may read it, but no one should be able to swap it without the seal breaking.

Tag verification

The authentication tag is checked before plaintext is accepted. If the tag fails, the ciphertext or associated data may have been modified, the wrong key may have been used, the wrong nonce may have been supplied, or the record may be corrupted.

Bad eventExpected AEAD behavior
Ciphertext bit changedTag verification fails.
AAD changedTag verification fails.
Wrong key usedTag verification fails.
Wrong nonce usedTag verification fails or decrypts incorrectly and fails tag check.
Tag changedTag verification fails.
Reject means reject

A system must not release unverified plaintext. “Decrypt then complain later” is how protocol dragons hatch.

GCM: Galois/Counter Mode

Galois/Counter Mode, or GCM, is a common AEAD mode often used with AES. It combines counter-mode encryption with a finite-field polynomial authenticator called GHASH.

Part of GCMJob
Counter-mode encryptionTurns AES into a stream of blocks that encrypt plaintext by XOR.
GHASHAuthenticates ciphertext and associated data using finite-field arithmetic.
Authentication tagDetects tampering with ciphertext or AAD.
NonceMust be unique under the same key.
GCM nonce warning

Reusing a GCM nonce with the same key is dangerous. It can compromise confidentiality and authentication.

The math shape of GCM

At a high level, GCM does two jobs in parallel: encrypt with counter mode and authenticate with GHASH.

\[ C=P\oplus KS \] \[ tag=GHASH_H(AAD,C)\oplus mask \]

The exact GCM construction has important details, but the course-level intuition is enough for now: counter mode provides confidentiality, while GHASH provides authentication over both ciphertext and associated data.

ChaCha20-Poly1305

ChaCha20-Poly1305 is another major AEAD construction. ChaCha20 is a stream cipher, and Poly1305 is a one-time authenticator. Together they provide fast authenticated encryption, especially useful on systems without hardware acceleration for AES.

PartJob
ChaCha20Generates a keystream to encrypt plaintext.
Poly1305Authenticates ciphertext and associated data.
NonceMust be unique under the same key.
TagDetects tampering.
\[ C=P\oplus ChaCha20(K,N) \] \[ tag=Poly1305(one\text{-}time\ key,AAD,C) \]

Nonce requirements in AEAD

For common AEAD modes like AES-GCM and ChaCha20-Poly1305, the nonce is usually public but must be unique under the same key. It does not normally need to be secret, but reuse can be catastrophic.

Nonce behaviorSecurity result
Unique per keyExpected safe use.
Repeated with same keyDangerous; may break confidentiality and/or authenticity.
Public but uniqueUsually acceptable.
Secret but repeatedStill dangerous. Secrecy does not repair reuse.

Misuse resistance

Definition: misuse resistance

Misuse resistance is a design property where a cryptographic mode limits the damage caused by some implementation mistakes, such as accidental nonce reuse.

Misuse-resistant modes are not magic shields. They do not make bad practice good. But they may fail less catastrophically when a common mistake occurs.

Mode typeIf nonce is reusedLesson
Nonce-sensitive AEADMay suffer severe security failure.Nonce management must be correct.
Misuse-resistant AEADDamage may be reduced for some mistakes.Safer design, but still follow rules.
Examples to know later

AES-SIV and related Synthetic IV designs are common examples of nonce-misuse-resistant authenticated encryption. We do not need the full construction yet, only the design goal.

Common mistakes

Mistake 1: encrypting but not authenticating

Attackers may modify ciphertexts if there is no authentication check.

Mistake 2: ignoring AAD

Public metadata still needs tamper protection.

Mistake 3: reusing AEAD nonces

Public is fine. Reused under the same key is not.

Mistake 4: releasing plaintext before tag verification

Decryption must fail closed. No verified tag, no trusted plaintext.

Mistake 5: assuming misuse resistance means mistake-proof

Misuse resistance reduces some damage. It does not repeal cryptographic rules.

    Show solution

    No. AAD is authenticated but not encrypted.

  3. What should happen if AEAD tag verification fails?
    Show solution

    The message should be rejected and plaintext should not be released or trusted.

  4. What are the two main jobs inside AES-GCM?
    Show solution

    Counter-mode encryption provides confidentiality, and GHASH provides authentication.

  5. What are the two main parts of ChaCha20-Poly1305?
    Show solution

    ChaCha20 provides stream-cipher encryption, and Poly1305 provides authentication.

  6. Does making a nonce secret fix nonce reuse?
    Show solution

    No. For nonce-sensitive AEAD, reuse under the same key is dangerous even if the nonce was hidden.

Why this matters for cryptography

AEAD is one of the practical centerpieces of modern cryptography. TLS, secure messaging, VPNs, APIs, and storage systems all need confidentiality and integrity together. AEAD gives a clean interface: key, nonce, associated data, plaintext in; ciphertext and tag out. But its safety depends on correct nonce use, tag verification, and careful binding of metadata.

  • Encryption alone does not always provide integrity.
  • AEAD provides confidentiality, integrity, and authenticity under a shared key.
  • AAD is authenticated but not encrypted.
  • Tags detect tampering with ciphertext or AAD.
  • Plaintext should not be released if tag verification fails.
  • AES-GCM combines counter-mode encryption with GHASH.
  • ChaCha20-Poly1305 combines a stream cipher with a polynomial authenticator.
  • Common AEAD nonces must be unique per key.
  • Misuse-resistant modes reduce damage from some mistakes but do not make rules optional.

One-page recap

What to remember
  • AEAD stands for Authenticated Encryption with Associated Data.
  • AEAD encrypts plaintext and authenticates ciphertext.
  • AEAD also authenticates AAD without encrypting it.
  • The authentication tag detects tampering.
  • Failed tag verification means reject the message.
  • Do not release unverified plaintext.
  • AES-GCM uses AES counter mode for encryption.
  • GCM uses GHASH for authentication.
  • ChaCha20-Poly1305 uses ChaCha20 for encryption.
  • ChaCha20-Poly1305 uses Poly1305 for authentication.
  • Common AEAD nonces are public but must be unique per key.
  • Nonce reuse can be catastrophic.
  • Misuse resistance reduces damage from some mistakes.
  • Use vetted libraries instead of hand-rolling modes.
