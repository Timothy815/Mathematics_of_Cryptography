Formal Security Goals: IND-CPA, IND-CCA, INT-CTXT, and AE Security

Turning “secure” into testable attacker games: indistinguishability, chosen-plaintext attacks, chosen-ciphertext attacks, ciphertext integrity, and the combined goals behind authenticated encryption.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 66 introduced authenticated encryption as a practical tool. This tutorial explains how cryptographers make the word “secure” more precise. Rather than saying “this seems hard to break,” formal security goals ask: what can the attacker query, what must the attacker guess or forge, and how much better than chance can the attacker do?

Formal security games are courtroom dramas for algorithms. The attacker gets rules, powers, and a challenge. The scheme is not judged by vibes, but by whether the attacker can win more often than random guessing or trivial luck should allow.

What you should know first

You should know symmetric encryption, AEAD, nonces, authentication tags, MACs, and the difference between confidentiality and integrity. The math here is not heavy algebra; it is careful adversarial thinking.

Why security games?

Security games give us a disciplined way to define attacker success. They describe what the attacker is allowed to do and what counts as winning.

Game ingredientMeaning
Attacker powersWhat the attacker may query or observe.
ChallengeThe test the attacker must solve.
Winning conditionWhat counts as success.
AdvantageHow much better than chance the attacker performs.
Memory hook

A security definition is not a slogan. It is a game with rules, a scoreboard, and a suspiciously clever opponent.

Attacker advantage

Many confidentiality games ask the attacker to guess a hidden bit \(b\), often meaning “was message 0 encrypted or message 1 encrypted?” Random guessing succeeds with probability \(1/2\).

\[ Advantage = \left|Pr[attacker\ wins]-\frac{1}{2}\right| \]

If the attacker wins about half the time, they have no useful distinguishing power. If they win much more often than half, the scheme leaks something useful.

Tiny advantage matters at scale

Formal cryptography often treats even small advantages carefully because attackers may run many trials, exploit structure, or combine leaks with other weaknesses.

IND-CPA: indistinguishability under chosen-plaintext attack

Definition: IND-CPA

IND-CPA means that even if an attacker can ask for encryptions of chosen plaintexts, they still cannot distinguish which of two chosen challenge messages was encrypted, except with negligible advantage.

IND-CPA game shape
attacker
queries
encryption oracle

\(m_0,m_1\)

1. Attacker may ask for encryptions of chosen messages.
2. Attacker chooses two equal-length messages m0 and m1.
3. Challenger flips secret bit b.
4. Challenger returns Enc(K, mb).
5. Attacker guesses b.
6. Secure means attacker cannot do much better than 50%.

IND-CPA captures confidentiality against passive observation plus chosen encryption queries. It says ciphertexts should not reveal which plaintext was chosen.

Why deterministic encryption usually fails IND-CPA

If encryption is deterministic, the same plaintext under the same key always produces the same ciphertext. Then an attacker can often compare challenge ciphertexts to earlier encryption queries.

Scheme behaviorIND-CPA problem
Same plaintext always gives same ciphertext.Attacker can build a codebook of encryptions.
Nonce/randomness not used correctly.Patterns may leak.
Repeated messages are visible.Equality information leaks.
Core lesson

Confidential encryption usually needs randomness, nonces, or state so identical plaintexts do not simply announce themselves.

IND-CCA: indistinguishability under chosen-ciphertext attack

Definition: IND-CCA

IND-CCA means the attacker still cannot distinguish challenge encryptions even when given controlled access to a decryption oracle, except that they cannot simply ask the oracle to decrypt the challenge ciphertext itself.

1. Attacker can ask encryption and decryption questions.
2. Attacker chooses m0 and m1.
3. Challenger returns challenge ciphertext C* = Enc(K, mb).
4. Attacker may continue asking allowed decryption questions, but not for C* itself.
5. Attacker guesses b.
6. Secure means attacker cannot do much better than 50%.

IND-CCA is stronger than IND-CPA. It models attackers who can feed modified ciphertexts into a system and observe whether decryption succeeds, fails, or leaks information.

Memory hook

CPA asks, “Can you learn from chosen encryptions?” CCA asks, “Can you also poke the decryption machine and learn from its reactions?”

Malleability and why CCA matters

A malleable encryption scheme lets attackers modify ciphertexts in ways that cause meaningful changes to plaintexts. Even if attackers cannot directly decrypt, the system’s reactions may leak information.

PropertyMeaningRisk
MalleableCiphertext changes can cause controlled plaintext changes.Attackers may manipulate messages.
Decrypt-and-report errorsSystem reveals different failure reasons.Can become an oracle.
Authenticate before releaseInvalid ciphertexts are rejected.Limits decryption-oracle leakage.

INT-CTXT: integrity of ciphertexts

Definition: INT-CTXT

INT-CTXT means an attacker cannot create a new ciphertext that decrypts successfully unless that ciphertext was previously produced by the encryption oracle.

1. Attacker may ask for encryptions of chosen messages.
2. Attacker sees valid ciphertexts and tags.
3. Attacker tries to output a new ciphertext/tag pair.
4. The attacker wins if the new pair decrypts successfully.
5. Secure means successful forgery should be infeasible.

INT-CTXT is about tamper resistance. It says attackers should not be able to manufacture valid ciphertexts, even if they have seen many valid examples.

Authenticated Encryption security

Authenticated Encryption aims to provide both confidentiality and integrity. In simplified course language:

GoalFormal flavorPlain-language meaning
ConfidentialityIND-CPA or strongerAttacker cannot learn which plaintext was encrypted.
IntegrityINT-CTXTAttacker cannot forge a valid new ciphertext.
Strong active-attack protectionIND-CCA-like behaviorInvalid ciphertexts should not leak useful information.
AE security intuition

A good AEAD scheme hides the plaintext and refuses forged ciphertexts. It is not just a locked box. It is a locked box with a tamper-evident seal and a rule that broken seals are never opened.

Encrypt-then-MAC intuition

One historically important way to think about authenticated encryption is encrypt-then-MAC: first encrypt the plaintext, then compute a MAC over the ciphertext and relevant associated data.

\[ C=Enc(K_E,N,P) \] \[ tag=MAC(K_M,AAD\parallel N\parallel C) \]

During decryption, verify the tag first. Only if the tag is valid should plaintext be released.

Why this pattern matters

Encrypt-then-MAC helps avoid decryption-oracle behavior because invalid ciphertexts are rejected before plaintext processing.

Common mistakes

Mistake 1: thinking confidentiality implies integrity

IND-CPA confidentiality does not mean attackers cannot modify ciphertexts.

Mistake 2: thinking integrity implies secrecy

A MAC can authenticate data without hiding it.

Mistake 3: treating decryption errors casually

Error behavior can become an oracle if it leaks distinctions attackers can exploit.

Mistake 4: assuming AEAD fixes nonce misuse automatically

AEAD security depends on the mode’s assumptions, often including nonce uniqueness.

Why this matters for cryptography

Formal security goals let cryptographers talk precisely about what a scheme does and does not protect. They also reveal why “encrypted” is not enough. IND-CPA gives a confidentiality target. IND-CCA strengthens the attacker model. INT-CTXT gives an integrity target. AEAD combines these ideas into the modern interface used by secure channels and protocols.

  • Security games define attacker powers and winning conditions.
  • Advantage measures success above chance.
  • IND-CPA is confidentiality under chosen-plaintext attack.
  • Deterministic encryption often fails IND-CPA.
  • IND-CCA models attackers with decryption-query power.
  • Malleability can make decryption behavior dangerous.
  • INT-CTXT prevents valid ciphertext forgeries.
  • AEAD should provide confidentiality plus integrity.
  • Encrypt-then-MAC gives useful intuition for fail-closed decryption.
