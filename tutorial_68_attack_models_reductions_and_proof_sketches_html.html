Attack Models, Reductions, and Proof Sketches

How cryptographers reason from attacker powers to security claims: threat models, adversary games, reductions, assumptions, proof sketches, tightness, and what a proof does not prove.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 67 introduced formal security goals such as IND-CPA, IND-CCA, and INT-CTXT. This tutorial explains how cryptographers connect those goals to assumptions and proof sketches. The central move is called a reduction: if an attacker can break our scheme, then we can use that attacker as a tool to solve some hard problem.

A reduction is a trapdoor argument. You say, “Suppose the attacker has a lockpick for my scheme. Then I can use that lockpick to open a vault everyone believes should stay closed.” If the vault is truly hard, the lockpick probably does not exist.

What you should know first

You should know the idea of security games, attacker advantage, IND-CPA, IND-CCA, INT-CTXT, AEAD, and hard problems such as factoring, discrete logarithms, or Diffie-Hellman-style assumptions. This tutorial is about reasoning structure more than computation.

Attack models

An attack model says what the attacker can do. This matters because a scheme may be secure against one kind of attacker and insecure against another.

ModelAttacker can...Example formal goal
Passive attackerObserve traffic.Basic confidentiality.
Chosen-plaintext attackerAsk for encryptions of chosen messages.IND-CPA.
Chosen-ciphertext attackerAsk for decryptions of chosen ciphertexts under rules.IND-CCA.
Forgery attackerTry to create a valid new ciphertext, tag, or signature.INT-CTXT or EUF-CMA.
Side-channel attackerObserve timing, power, cache behavior, or errors.Implementation security, not just mathematical security.
Memory hook

An attack model is the attacker’s equipment list. You cannot judge a castle without knowing whether the enemy has ladders, cannons, spies, or a dragon named Timing Leak.

Threat model versus attack model

These phrases are related but not identical. A threat model is broader and practical. An attack model is often narrower and formal.

ConceptQuestion it answersExample
Threat modelWhat real-world risks are we designing against?Network attacker, stolen database, malicious insider, compromised client.
Attack modelWhat powers are allowed in the formal game?Chosen-plaintext queries, chosen-ciphertext queries, signing queries.

Reduction big picture

A reduction tries to show that breaking a cryptographic scheme would imply solving a hard problem. This does not prove the hard problem is impossible. It says the scheme is at least connected to that hardness assumption under the model.

Reduction shape
Adversary \(A\)
breaks scheme
used by
Reduction \(R\)
to solve
Hard problem
\[ A\ breaks\ scheme \Longrightarrow R^A\ solves\ hard\ problem \]

The notation \(R^A\) means the reduction \(R\) uses the adversary \(A\) like a subroutine or oracle.

The contrapositive shape

Reductions often reason by contrapositive. The proof shape sounds like this:

If an efficient attacker A can break the scheme,
then an efficient reduction R can solve hard problem H.
But we assume H is hard for efficient algorithms.
Therefore, no efficient attacker A should be able to break the scheme.
This is conditional security

The proof is only as strong as its assumptions, model, definitions, and implementation match. It is not a holy spell; it is a conditional bridge.

Ingredients of a reduction

IngredientQuestion
AdversaryWhat break are we assuming exists?
Hard problem instanceWhat challenge is the reduction trying to solve?
SimulationHow does the reduction pretend to be the real security game?
EmbeddingWhere is the hard problem hidden inside the adversary’s view?
ExtractionHow does the reduction turn the adversary’s success into a hard-problem solution?
Loss analysisHow much advantage or running time is lost?

A toy reduction pattern

Suppose a public-key encryption scheme is claimed secure based on a hard problem \(H\). A proof sketch might say:

Assume attacker A can distinguish encryptions of m0 and m1.
Build reduction R that receives an instance of hard problem H.
R embeds that hard instance into the public key or challenge ciphertext.
R runs A and answers A's allowed oracle queries.
When A guesses the hidden bit correctly, R uses that guess to solve H.
Therefore, if A has useful advantage, R has useful advantage against H.
Memory hook

The reduction hides a diamond inside the attacker’s puzzle box. If the attacker opens the box, the reduction steals the diamond.

Advantage and tightness

A reduction may lose some of the attacker’s advantage. A tight reduction loses little. A loose reduction loses a lot, which may require larger security parameters.

\[ Adv_R \geq \frac{Adv_A}{q} \]

This example says the reduction’s advantage is at least the adversary’s advantage divided by some loss factor \(q\), perhaps the number of oracle queries or guesses needed by the proof.

Reduction typeMeaningPractical effect
TightLittle security loss.Security estimate transfers cleanly.
LooseLarge security loss.May need larger parameters or more caution.

Hybrid arguments

A hybrid argument proves two worlds are indistinguishable by moving through a chain of small changes. If each step is hard to notice, the beginning and end are hard to distinguish.

Game 0: Real scheme.
Game 1: Replace one value with a pseudorandom value.
Game 2: Replace another value with random.
Game 3: Ideal world.

If no efficient attacker can distinguish adjacent games,
then no efficient attacker can distinguish Game 0 from Game 3.
Memory hook

Hybrid proofs are cryptographic ship repairs. Replace one plank at a time. If no one can tell which plank changed, no one can tell when the whole ship became a different ship.

Proof sketches

A proof sketch gives the structure of a proof without every technical detail. It should still make clear what is being assumed, what is being proven, and how the reduction connects them.

Proof-sketch pieceQuestion to answer
ClaimWhat security goal is being proven?
AssumptionWhat hard problem or primitive security is assumed?
AdversaryWhat kind of attacker is assumed to exist?
Construction of reductionHow is the attacker used?
SimulationWhy does the attacker’s view look real?
ConclusionWhy would attacker success contradict the assumption?

What proofs do not prove

A proof can be powerful and still limited. It proves a statement inside a model under assumptions. It does not automatically prove that every implementation, parameter choice, random number generator, or hardware environment is safe.

A proof may not cover...Example
Bad implementationTiming leaks, memory bugs, wrong nonce handling.
Bad randomnessPredictable keys or reused signing nonces.
Wrong modelProof assumes CPA, real attacker has CCA or side-channel access.
Weak parametersSmall keys, obsolete curves, short tags.
Composition mistakesSecure primitive used in an unsafe protocol.
Proofs are maps, not armor

A map can be accurate and still not stop rain, thieves, or a broken wagon wheel. Formal proofs guide design, but implementations still need engineering discipline.

Common mistakes

Mistake 1: treating a proof as unconditional

Most cryptographic proofs are conditional on assumptions and models.

Mistake 2: ignoring the attack model

A scheme proven secure against CPA attacks may not be secure against CCA or side-channel attacks.

Mistake 3: forgetting reduction loss

A very loose reduction can weaken the practical meaning of a security estimate.

Mistake 4: confusing proof sketch with full proof

A proof sketch explains the shape. A full proof must handle all details, edge cases, and probabilities.

Practice problems

  1. What is an attack model?
    Show solution

    A formal description of what powers the attacker has, such as encryption queries, decryption queries, signing queries, or observation access.

  2. What is a reduction?
    Show solution

    An algorithm that uses an attacker against one scheme to solve another problem, usually a problem assumed to be hard.

  3. What is the basic reduction shape?
    Show solution

    If attacker \(A\) breaks the scheme, reduction \(R\) uses \(A\) to solve the hard problem.

  4. What does tightness describe?
    Show solution

    How much advantage or efficiency is lost when translating the attacker’s success into a hard-problem solution.

  5. What is a hybrid argument?
    Show solution

    A proof technique that moves through a sequence of slightly different games or worlds, showing adjacent worlds are hard to distinguish.

  6. Name one thing a formal proof may not cover.
    Show solution

    Examples include implementation bugs, side channels, bad randomness, wrong parameters, or unsafe composition.

Why this matters for cryptography

Reductions and proof sketches are the bridge from experimental confidence to structured confidence. They do not make cryptography simple, and they do not eliminate engineering risk. But they give cryptographers a disciplined way to say why a scheme’s security follows from a particular assumption in a particular model.

  • Attack models define attacker powers.
  • Threat models connect formal games to real systems.
  • Reductions use attackers as subroutines.
  • Security proofs are usually conditional.
  • Simulation makes the adversary’s view look real.
  • Embedding hides the hard problem inside the game.
  • Extraction turns attacker success into a hard-problem solution.
  • Tightness measures loss in the reduction.
  • Hybrid arguments move from real to ideal worlds one step at a time.
  • Proofs do not automatically cover bad implementations or wrong models.

Core formulas and patterns
\[ A\ breaks\ scheme \Longrightarrow R^A\ solves\ hard\ problem \] \[ Adv_R \geq \frac{Adv_A}{q} \] 
Reduction proof shape:
Assume attacker A breaks the scheme.
Build reduction R that uses A.
R simulates the game for A.
R embeds the hard problem into A's view.
A succeeds.
R extracts a solution to the hard problem.
Therefore, if the hard problem is hard, A should not exist.
