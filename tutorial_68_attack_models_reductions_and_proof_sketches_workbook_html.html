How to use this workbook

Think like a proof architect. Each exercise asks you to identify the attacker’s powers, the assumption being used, how the attacker is simulated, and how attacker success becomes a solution to a hard problem.

Quick reference

Reduction shape
\[ A\ breaks\ scheme \Longrightarrow R^A\ solves\ hard\ problem \]

Read: the reduction \(R\) uses adversary \(A\) as a subroutine.

Tightness shape
\[ Adv_R \geq \frac{Adv_A}{q} \]

A larger \(q\) means more reduction loss.

Part E: Contrapositive proof shape

If an efficient attacker A can break the scheme,
then an efficient reduction R can solve hard problem H.
But we assume H is hard for efficient algorithms.
Therefore, no efficient attacker A should be able to break the scheme.

Part G: Toy reduction pattern

Assume attacker A can distinguish encryptions of m0 and m1.
Build reduction R that receives an instance of hard problem H.
R embeds that hard instance into the public key or challenge ciphertext.
R runs A and answers A's allowed oracle queries.
When A guesses the hidden bit correctly, R uses that guess to solve H.
Therefore, if A has useful advantage, R has useful advantage against H.
G1. What is being assumed about attacker \(A\) at the beginning of the proof?
G2. What does the reduction do with the hard problem instance?
G3. What does the reduction extract from the attacker’s success?

Note for self-study

The central idea is that a reduction is a conditional bridge: if an attacker exists, then a solver for a hard problem exists. If the hard problem is truly hard, the attacker should not exist in that model.

Part A: Warm-up vocabulary

QuestionAnswer
A1. What is an attack model?A formal description of what the attacker can query, observe, modify, or control.
A2. What is a threat model?A broader real-world description of attackers, assets, risks, environment, and goals.
A3. What is an adversary \(A\)?The attacker algorithm trying to win a security game.
A4. What is a challenger?The referee that runs the security game and answers allowed queries.
A5. What is an oracle?A controlled black-box service, such as encryption, decryption, signing, or verification.
A6. What is a reduction \(R\)?An algorithm that uses adversary \(A\) to solve a supposedly hard problem.
A7. What is an assumption?A computational problem or primitive security property believed to be hard or secure.
A8. What is advantage?How much better than baseline or chance an adversary performs.
A9. What is tightness?How much advantage or efficiency is preserved in the reduction.
A10. What is a proof sketch?A high-level outline of a proof’s structure.
A11. What is a hybrid argument?A proof technique using a sequence of slightly different games.
A12. What does negligible mean?Too small to matter at cryptographic security scale, formally shrinking very quickly with the security parameter.

Part B: Attack models

ModelAttacker can...Example formal goal
Passive attackerObserve traffic.Basic confidentiality.
Chosen-plaintext attackerAsk for encryptions of chosen messages.IND-CPA.
Chosen-ciphertext attackerAsk for decryptions of chosen ciphertexts under rules.IND-CCA.
Forgery attackerTry to create a valid new ciphertext, tag, or signature.INT-CTXT or EUF-CMA.
Side-channel attackerObserve timing, power, cache behavior, or errors.Implementation security.

B1 sample answer. A claim is only meaningful relative to attacker powers. A scheme secure against passive observation may fail against chosen-ciphertext attacks or side channels.

C1 sample answer. A threat model includes real-world assets, attackers, deployment assumptions, and environmental risks. An attack model narrows this into formal powers used in a security game.

Part D: Reduction big picture

D1 sample answer. A reduction tries to show that if someone can break the scheme, then that attacker can be used to solve a hard problem. Since the hard problem is assumed hard, the attacker should not exist.

D2 answer. \(R^A\) means reduction \(R\) uses adversary \(A\) as a subroutine or oracle.

E1 sample answer. It is conditional because the conclusion depends on the assumption that the hard problem really is hard and that the formal model matches the situation being claimed.

Part K: What proofs do not prove

A proof may not cover...Example
Bad implementationTiming leaks, memory bugs, wrong nonce handling.
Bad randomnessPredictable keys or reused signing nonces.
Wrong modelProof assumes CPA, real attacker has CCA or side-channel access.
Weak parametersSmall keys, obsolete curves, short tags.

K1 sample answer. A proof applies under specific assumptions and models. Real implementations can fail because of bad randomness, bugs, side channels, wrong parameters, or unsafe protocol composition.

Part N: Concept checks

N1 sample answer. An attack model formally describes the attacker’s powers in a security game, such as chosen-plaintext or chosen-ciphertext access. A threat model is broader and describes the real-world attackers, assets, risks, and environment.

N2 sample answer. A reduction shows that if an attacker can break a scheme, then that attacker can be used as a tool to solve a hard problem. Since the hard problem is assumed hard, the attacker should not exist in that model.

N3 sample answer. Tightness describes how much advantage or efficiency is lost when converting attacker success into a hard-problem solution. Hybrid arguments prove security by moving from a real game to an ideal game through small indistinguishable steps.

N4 sample answer. This tutorial explained how cryptographers reason about attack models, reductions, proof sketches, tightness, hybrid arguments, and proof limitations. It emphasized that proofs are conditional on assumptions, models, and correct implementation.
