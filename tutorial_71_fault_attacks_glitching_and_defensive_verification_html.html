Fault Attacks, Glitching, and Defensive Verification

How attackers disturb cryptographic computations: voltage glitches, clock glitches, laser faults, instruction skips, corrupted intermediate values, Bellcore-style RSA faults, redundancy, consistency checks, infective countermeasures, and defensive verification.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 70 explained side channels: attacks that listen to unintended leaks. Fault attacks are different. Here the attacker actively disturbs the computation and studies what happens. A device may run the correct algorithm, but if the attacker can make it skip a step, flip a bit, corrupt a register, or output a faulty result, the fault can become a crowbar against the secret.

Side channels ask, “What did the machine whisper?” Fault attacks ask, “What happens if I shove the machine at just the wrong moment?” The mathematics may be pure, but the hardware lives in a thunderstorm.

What you should know first

You should know side channels, timing leaks, modular arithmetic, public-key signatures, RSA at a high level, finite-field operations, MACs, AEAD verification, and implementation security basics. You do not need electrical-engineering expertise; the goal is to understand the attack logic and defensive habits.

What fault attacks are

Definition: fault attack

A fault attack intentionally causes a cryptographic computation to make an error, then uses the faulty result, changed behavior, or failure pattern to recover secret information or bypass security checks.

Fault attacks are especially important for smart cards, embedded devices, hardware wallets, secure elements, payment devices, Internet of Things devices, and any system where attackers may get physical access.

Ethical boundary

The goal here is defensive understanding. Real fault injection against systems you do not own or have permission to test is not acceptable. In legitimate labs, this belongs in controlled hardware-security research environments.

Instruction skips and bypasses

One common fault goal is making a device skip a critical instruction. If the skipped instruction is a security check, the attacker may bypass authentication, signature verification, or access control.

if verify_signature(message, signature, public_key) == false:
    reject()
allow_access()

If a fault skips the branch, skips the rejection, or corrupts the verification result, access might be allowed incorrectly.

Single-check fragility

A single critical check can become a single critical failure point. Defensive designs often add redundancy, consistency checks, or fail-closed behavior.

Bit flips and corrupted state

A fault may flip bits in an intermediate value. That corrupted value may produce an output related to the correct computation in a way that helps the attacker.

\[ correct\ output = F_K(x) \] \[ faulty\ output = F_K(x, fault) \] \[ difference = correct\ output \oplus faulty\ output \]

Differential fault analysis studies how differences between correct and faulty outputs reveal information about secret internal state or keys.

Differential fault analysis

Definition: differential fault analysis

Differential fault analysis compares correct and faulty outputs to infer secrets from the way faults propagate through the algorithm.

StepIdea
1. Obtain correct outputRun the device normally.
2. Induce faultDisturb one operation, round, register, or memory value.
3. Obtain faulty outputRecord the altered result.
4. Compare outputsStudy how the difference propagates.
5. Narrow secret candidatesUse the fault pattern to reduce the key/state search.

The Bellcore-style RSA fault idea

A famous family of fault attacks targets RSA implementations using the Chinese Remainder Theorem, or CRT, optimization. The details can become technical, but the lesson is powerful: one faulty signature may reveal a factor of the RSA modulus if the implementation releases it unchecked.

High-level RSA-CRT picture

RSA-CRT computes pieces of the signature modulo the secret primes \(p\) and \(q\), then recombines them. If one side is faulty and the result is released, the difference between the correct mathematical relation and the faulty output can expose a factor.

\[ N=pq \] \[ faulty\ signature\ may\ reveal\ gcd(N,\ difference) \]

The defensive lesson is simple: verify a signature before releasing it, especially when using optimized private-key computations.

Never release unchecked private-key results

If a device computes a signature, it can verify its own result before outputting it. If verification fails, it should reject safely.

Defensive verification

Defensive verification means checking that a cryptographic result is valid before releasing or trusting it. For example, after signing, verify the signature with the public key. After decrypting, verify padding, tags, structure, and consistency before allowing sensitive behavior.

OperationDefensive check
RSA signature generationVerify signature before output.
AEAD decryptionVerify tag before releasing plaintext.
Boot verificationCheck signature/hash before executing firmware.
Access-control decisionUse redundant checks or state consistency validation.
Key derivation or key loadCheck integrity, version, and expected state.

Redundancy and consistency checks

Redundancy means doing extra work to detect whether a fault occurred. This can include recomputing a value, computing it in a different way, checking invariants, or comparing duplicated state.

DefenseIdeaRisk if done poorly
Recompute and compareRun operation twice and compare outputs.Same fault may affect both runs.
Diverse computationCompute by two different methods.More complexity can create bugs.
Invariant checksVerify expected mathematical relationships.Incomplete checks may miss faults.
Control-flow integrityDetect skipped or reordered steps.Implementation must be carefully protected.
SensorsDetect voltage, clock, light, or temperature anomalies.Attackers may bypass or tune around sensors.

Infective countermeasures

Some defenses avoid a simple branch like “if fault detected, return error,” because that branch itself might be skipped. Infective countermeasures aim to mix the fault detection result into the computation so that if a fault occurs, the output becomes useless rather than merely triggering a branch.

Defense styleBehaviorMain concern
Detect and branchIf fault detected, reject.Branch may itself be skipped.
InfectiveIf fault detected, corrupt/poison the output.Must be designed carefully to avoid new leaks.
Memory hook

A detect-and-branch defense says, “Stop if poisoned.” An infective defense says, “If poisoned, make the whole dish inedible.”

Fault attacks and formal proofs

A formal proof may assume the algorithm runs correctly. Fault attacks challenge that assumption. If the attacker can make the device compute the wrong thing, skip checks, or reveal faulty outputs, the real system no longer matches the proof model.

Proof assumes...Fault attack may cause...
Correct computation.Corrupted intermediate state.
Verification check executes.Instruction skip bypasses check.
Output is honestly generated.Faulty output related to secret state.
Failure path is reliable.Error branch skipped or corrupted.

Common mistakes

Mistake 1: assuming attackers only observe

Physical attackers may disturb computation, not merely listen to it.

Mistake 2: releasing unchecked signatures

Faulty private-key outputs can reveal secrets. Verify before output where appropriate.

Mistake 3: relying on one critical branch

If a single reject branch is skipped, security may collapse.

Mistake 4: detailed fault errors

Fault handling can become an oracle if it reveals too much.

Mistake 5: adding defenses without testing their interactions

Redundancy, sensors, and infective logic can create new bugs if composed carelessly.

Why this matters for cryptography

Fault attacks show that real cryptography must protect not only the secrecy of values and the timing of computations, but also the correctness of execution. A cryptographic result should not be trusted merely because it came from the right algorithm. It should be checked, cross-checked, and withheld if the device may have been disturbed.

  • Fault attacks actively disturb computation.
  • Glitching can target voltage, clock, reset, or environmental conditions.
  • Instruction skips can bypass security checks.
  • Bit flips can corrupt internal state.
  • Differential fault analysis compares correct and faulty outputs.
  • RSA-CRT implementations need defensive verification.
  • Redundancy and consistency checks help detect faults.
  • Infective countermeasures try to make faulty outputs useless.
  • Fail-closed behavior is essential under uncertainty.
  • Formal proofs may not model faulty execution.
