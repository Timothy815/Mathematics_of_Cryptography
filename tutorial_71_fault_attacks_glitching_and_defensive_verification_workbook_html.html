Note for self-study

The central idea is that fault attacks break the assumption of correct execution. A secure implementation must detect, resist, or safely fail when computation has been disturbed.

Part A: Warm-up vocabulary

QuestionAnswer
A1. What is a fault attack?An attack that induces computational errors and uses faulty behavior or output.
A2. What is fault injection?A physical or logical method for forcing an error into a device or computation.
A3. What is glitching?A precisely timed disturbance to voltage, clock, reset, or environment.
A4. What is a voltage glitch?A temporary power disturbance that can cause incorrect computation.
A5. What is a clock glitch?An abnormal clock pulse or timing disturbance that can cause skipped or malformed operations.
A6. What is a laser fault?A focused light-induced disturbance to a chip region.
A7. What is an instruction skip?A fault causing an instruction not to execute correctly.
A8. What is a bit flip?A fault that changes one or more bits in state, memory, or computation.
A9. What is redundancy?Extra computation or data used to detect faults.
A10. What is a consistency check?A verification that output satisfies expected relationships.
A11. What is an infective countermeasure?A defense that makes faulty outputs useless rather than relying only on a reject branch.
A12. What is defensive verification?Checking cryptographic results before outputting or trusting them.

Part B: Fault attacks versus side channels

FeatureSide-channel attackFault attack
Attacker postureMostly observes.Actively disturbs.
Information sourceTiming, power, cache, errors, emissions.Faulty outputs, skipped checks, corrupted state.
Typical goalInfer secrets from leakage.Force useful errors or bypasses.
Defense styleConstant-time, masking, hiding.Redundancy, verification, sensors, fault detection.

B1 sample answer. Listening observes unintended leaks such as time or power. Disturbing intentionally changes the computation, causing skipped checks, corrupted state, or faulty outputs.

Part F: Differential fault analysis

StepIdea
1. Obtain correct outputRun the device normally.
2. Induce faultDisturb one operation, round, register, or memory value.
3. Obtain faulty outputRecord the altered result.
4. Compare outputsStudy how the difference propagates.
5. Narrow secret candidatesUse the fault pattern to reduce the key/state search.

F1 sample answer. Differential fault analysis compares normal and faulted outputs. The way the output changes can reveal information about internal state or key material.

Part G: Bellcore-style RSA fault idea

G1 sample answer. In RSA-CRT, a fault in one modular branch can produce an output mathematically related to only part of the secret factorization. Comparing the faulty relation can reveal a factor of \(N\).

G2 answer. Verify private-key results, such as signatures, before releasing them.

Defense styleBehaviorMain concern
Detect and branchIf fault detected, reject.Branch may itself be skipped.
InfectiveIf fault detected, corrupt/poison the output.Must be designed carefully to avoid new leaks.

J1 sample answer. If an attacker can skip the reject branch, a normal detect-and-branch defense may fail. Infective logic aims to make the output useless even if the branch is skipped.

Part O: Concept checks

O1 sample answer. Fault attacks intentionally disturb cryptographic computation. The attacker may use voltage, clock, laser, electromagnetic, temperature, or software techniques to cause skipped instructions, bit flips, corrupted state, faulty outputs, or bypassed checks.

O2 sample answer. Differential fault analysis compares correct and faulty outputs to learn how a fault moved through the algorithm. In RSA-CRT, a faulty signature can sometimes reveal a factor of the modulus if released unchecked, so implementations should verify signatures before output.

O3 sample answer. Defensive verification checks results before releasing or trusting them. Redundancy repeats or cross-checks computation to detect faults. Infective countermeasures try to poison faulty outputs so they are useless even if a reject branch is skipped.

O4 sample answer. This tutorial explained how fault attacks disturb computation and how glitches, instruction skips, bit flips, and faulty outputs can bypass checks or reveal secrets. It covered differential fault analysis, RSA-CRT fault intuition, verification, redundancy, infective countermeasures, and proof-model limits.
