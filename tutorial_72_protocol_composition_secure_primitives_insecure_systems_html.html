Protocol Composition: Secure Primitives, Insecure Systems

Why good cryptographic ingredients can still make a bad protocol: ordering mistakes, missing identity binding, replay attacks, downgrade attacks, state confusion, key reuse, context separation failures, error oracles, and composition discipline.

🏠 Dashboard

Why this tutorial exists

Purpose

Tutorial 71 showed that correct algorithms can fail when computation is disturbed. This tutorial shows a different danger: correct primitives can fail when combined incorrectly. A protocol is not merely a bag of strong parts. The order, context, identities, state, error behavior, and binding rules matter.

Cryptographic primitives are strong bricks. Protocol composition is architecture. A pile of excellent bricks can still become a house with doors that open into the moat and windows labeled “please attack here.”

What you should know first

You should know hashes, MACs, digital signatures, key exchange, AEAD, nonces, replay protection, formal security goals, side channels, and fail-closed behavior. The central idea is that systems require secure interaction, not just secure components.

Primitive security versus protocol security

A primitive can be secure in isolation while the protocol using it is insecure. The primitive promises a narrow property under specific assumptions. The protocol must ensure the right data is protected, the right keys are used, the right messages are accepted, and the right failures are handled safely.

LayerQuestionExample failure
PrimitiveIs this building block secure under its definition?Using a weak MAC or broken cipher.
ModeIs the primitive used safely?Nonce reuse in GCM.
ProtocolAre messages, identities, keys, and state bound correctly?Replay, downgrade, unknown-key-share, state confusion.
ImplementationDoes the code leak or fail incorrectly?Timing leak, fault attack, parsing oracle.

Binding: the protocol glue

Binding means making sure important facts cannot be separated or swapped. A protocol often needs to bind identities, public keys, negotiated algorithms, transcript messages, nonces, session identifiers, and intended usage.

\[ session\_key = KDF(shared\_secret, transcript, identities, context) \]

This formula is not a universal construction, but it shows the design instinct: the key should depend not only on the raw secret, but also on what conversation produced it and who it belongs to.

Bind this...To prevent...
Identity to keyUnknown-key-share or impersonation confusion.
Transcript to sessionMessage splicing or handshake tampering.
Algorithm choices to transcriptDowngrade attacks.
Nonce/counter to messageReplay or reordering confusion.
Context label to derived keyKey reuse across purposes.

Identity binding

Identity binding prevents a key or message from being accepted under the wrong identity. A signature over a public key may not be enough if the signature does not also bind the role, protocol, session, or intended peer.

Missing bindingPossible result
Public key not bound to identity.Impersonation.
Identity not included in key derivation.Unknown-key-share confusion.
Role not bound.Client/server reflection or role confusion.
Protocol name not bound.Cross-protocol attack.

Replay attacks

A replay attack occurs when an attacker records a valid message and sends it again later. Cryptographic validity alone does not prove freshness.

Bad pattern:
1. Alice sends signed command: "Pay Bob $100"
2. Attacker records it.
3. Attacker sends the same signed command again.
4. System accepts it because signature is valid.

Replay defenses include nonces, timestamps, counters, sequence numbers, session identifiers, and server-side replay caches.

Replay defenseIdea
Nonce challengeProve the response was made for this session.
Counter/sequence numberReject old or repeated messages.
TimestampAccept only within a time window.
Session identifierBind messages to one session.
Replay cacheRemember used tokens/messages.

Downgrade attacks

A downgrade attack tricks parties into using weaker algorithms, weaker protocol versions, or less secure options even though both support stronger choices.

Client: I support StrongVersion and WeakVersion.
Attacker modifies message:
Client: I support WeakVersion.
Server: OK, use WeakVersion.

A strong protocol authenticates the negotiation transcript so tampering with algorithm choices is detected.

Design instinct

Negotiation is security-sensitive data. Treat supported versions, chosen ciphers, groups, and extensions as part of the authenticated transcript.

State confusion

Protocols are state machines. A message valid in one stage may be dangerous in another. State confusion happens when a system accepts a message in the wrong phase or under the wrong assumptions.

State issueRisk
Accepting data before handshake complete.Unauthenticated data processed too early.
Allowing old handshake messages in a new session.Replay or transcript confusion.
Not separating client/server roles.Reflection or role-confusion attack.
Continuing after failed verification.Fail-open behavior.

Key separation and context separation

Key separation means using different keys for different jobs. Context separation means making sure data from one context cannot be interpreted as data from another context.

\[ K_{enc}=KDF(master,\ \text{``encryption''}\parallel transcript) \] \[ K_{mac}=KDF(master,\ \text{``authentication''}\parallel transcript) \]

Labels such as "encryption" and "authentication" are simple examples of domain separation. They prevent two keys from accidentally being the same when they serve different purposes.

Key reuse warning

Reusing the same key for encryption, authentication, export, and application-specific purposes can create interactions that the primitive security proof never promised to handle.

Parsing and authentication order

A common protocol mistake is parsing, decompressing, or acting on untrusted plaintext before authentication succeeds. This can create oracles, crashes, state changes, or side channels.

PatternSecurity meaning
Decrypt → parse → verifyDangerous. Unverified data affects behavior.
Verify tag → decrypt/release → parseSafer AEAD instinct.
Parse headers as AADPublic metadata can be authenticated before use.

Toy protocol example

Consider a toy authenticated key exchange. The unsafe version signs only the ephemeral key. The safer version signs the transcript and derives keys from the transcript, identities, and context.

Unsafe instinctSafer instinct
Sign only \(g^a\).Sign transcript: identities, roles, \(g^a\), \(g^b\), algorithms.
Derive key only from shared secret.Derive key from shared secret plus transcript/context.
Accept any valid signed old value.Require freshness and session binding.
Continue after negotiation mismatch.Fail closed on mismatch.
\[ unsafe:\ K=KDF(g^{ab}) \] \[ safer:\ K=KDF(g^{ab}, transcript, identities, algorithms, context) \]

Common mistakes

Mistake 1: trusting primitives to fix protocol design

Secure primitives do not automatically bind identities, freshness, roles, or state.

Mistake 2: authenticating too little

If the transcript, identities, or algorithms are not authenticated, attackers may tamper with the conversation.

Mistake 3: missing replay protection

A valid old message can still be dangerous if freshness is not checked.

Mistake 4: key reuse across purposes

A key for one role should not casually become a key for another role.

Mistake 5: permissive state machines

Protocol messages should only be accepted in the correct phase and context.

Why this matters for cryptography

Protocol composition is where cryptography becomes systems design. A cipher, MAC, KDF, signature, or hash may be strong, but the protocol must decide what gets protected, when, under whose identity, in what session, with which keys, and in what state. Most real cryptographic failures are not “someone solved the math”; they are “the system used the math incorrectly.”

  • Secure primitives do not automatically create secure protocols.
  • Binding links identities, transcripts, algorithms, roles, and context.
  • Freshness prevents replay.
  • Authenticated negotiation prevents downgrade attacks.
  • Strict state machines prevent phase confusion.
  • Key separation prevents cross-purpose interactions.
  • Verify before parsing or acting on untrusted data.
  • Generic failures reduce oracle behavior.
  • Protocol design requires both mathematical and engineering discipline.

  • A protocol is a structured conversation using primitives.
  • Secure primitives can compose into insecure systems.
  • Binding ties identities, keys, transcript, algorithms, roles, and context together.
  • A transcript is the ordered record of protocol messages.
  • Replay attacks reuse old valid messages.
  • Freshness requires nonces, counters, timestamps, session IDs, or replay caches.
  • Downgrade attacks force weaker choices.
  • Negotiation choices should be authenticated.
