Note for self-study

The central idea is that secure primitives are ingredients, not the whole meal. Protocol security depends on ordering, binding, freshness, state, context, and failure behavior.

Part A: Warm-up vocabulary

QuestionAnswer
A1. What is a primitive?A cryptographic building block such as a hash, MAC, signature, encryption scheme, KDF, or key exchange.
A2. What is a protocol?A structured sequence of messages and state transitions using primitives to achieve a security goal.
A3. What is composition?Combining primitives and protocol steps together.
A4. What is binding?Cryptographically tying important facts together, such as identities, keys, transcript, roles, and context.
A5. What is a transcript?The ordered record of protocol messages exchanged.
A6. What is a replay attack?An attack where an old valid message is recorded and accepted again later.
A7. What is a downgrade attack?An attack that tricks parties into using weaker protocol versions or algorithms.
A8. What is context separation?Preventing data or keys from one purpose/protocol context from being reused as another.
A9. What is key separation?Using different keys for different purposes, often derived with labels.
A10. What is a state machine?The protocol’s rulebook of stages and valid messages in each stage.
A11. What does fail closed mean?Invalid or uncertain state leads to safe rejection.

Part B: Primitive security versus protocol security

LayerQuestionExample failure
PrimitiveIs this building block secure under its definition?Using a weak MAC or broken cipher.
ModeIs the primitive used safely?Nonce reuse in GCM.
ProtocolAre messages, identities, keys, and state bound correctly?Replay, downgrade, unknown-key-share, state confusion.
ImplementationDoes the code leak or fail incorrectly?Timing leak, fault attack, parsing oracle.

B1 sample answer. Secure primitives provide narrow guarantees, but a protocol must still bind identities, prove freshness, authenticate negotiation, separate keys, enforce state, and handle failures safely.

Replay defenseIdea

E1 sample answer. A signature proves the message was signed by the key, but it does not prove the message is new, intended for this session, or not already used.

Part F: Downgrade attacks

F1 sample answer. If negotiation choices are authenticated as part of the transcript, attackers cannot silently remove strong options or force weaker versions without detection.

Part G: State confusion

State issueRisk
Accepting data before handshake complete.Unauthenticated data processed too early.
Allowing old handshake messages in a new session.Replay or transcript confusion.
Not separating client/server roles.Reflection or role-confusion attack.
Continuing after failed verification.Fail-open behavior.

G1 sample answer. A strict state machine ensures messages are accepted only in the correct phase and context, reducing replay, reflection, and phase-confusion risks.

Part H: Key separation and context separation

H1 sample answer. Labels make derived keys purpose-specific, so an encryption key and authentication key do not accidentally collide or get reused across roles.

H2 sample answer. One key used for many roles may create cross-purpose interactions that none of the primitive security definitions promised to handle.

Part O: Concept checks

O1 sample answer. Protocol composition is the process of combining cryptographic primitives and message steps into a larger system. Even if each primitive is secure alone, the protocol can fail if identities, freshness, state, keys, context, or errors are handled incorrectly.

O2 sample answer. Binding ties important protocol facts together so they cannot be swapped or separated. Replay protection proves messages are fresh using nonces, counters, timestamps, session identifiers, or caches. Downgrade protection authenticates negotiation choices so attackers cannot force weaker settings.

O3 sample answer. Key separation uses different derived keys for different purposes, while context separation prevents one protocol or role from being confused with another. Strict state machines accept only expected messages in the correct phase and reject everything else.

O4 sample answer. This tutorial explained why strong cryptographic primitives do not automatically create secure systems. It covered binding, transcript protection, replay attacks, downgrade attacks, state confusion, key separation, context separation, safe parsing order, and composition discipline.
