How to use this workbook

Treat each protocol as a design case study. Do not memorize every implementation detail. Instead, identify what problem the protocol solves, how it authenticates identity, how it derives and updates keys, what prevents replay or downgrade, and what remains outside its protection.

Quick reference

Signal-style ratchet idea
\[ new\_chain\_key, message\_key = KDF(old\_chain\_key) \]

Note for self-study

The central idea is that real protocols reveal different design answers to the same recurring questions: who is authenticated, how are keys derived, how is freshness proven, how are keys updated, and what remains outside the protection boundary?

Part A: Warm-up vocabulary

QuestionAnswer
A1. What is TLS?Transport Layer Security, widely used for HTTPS and secure transport.
A2. What is the Signal Protocol?A protocol family for end-to-end encrypted messaging using asynchronous setup and ratcheting.
A3. What is SSH?Secure Shell, used for secure remote login, file transfer, and tunneling.
A4. What is WireGuard?A modern VPN protocol using static public keys and a compact Noise-based design.
A5. What is a handshake?The setup conversation that negotiates parameters, authenticates parties, and derives keys.
A6. What is a transcript?The ordered record of handshake messages.
A7. What is forward secrecy?Protection of past sessions even if long-term keys are later compromised.
A8. What is a key schedule?A structured pipeline for deriving purpose-specific keys and secrets.
A9. What does TOFU mean?Trust On First Use.
A10. What is a ratchet?A mechanism that updates keys forward and discards old keys.
A11. What is Noise in this context?A framework for building authenticated key-exchange handshake patterns.

Part C: TLS case study

TLS design featureSecurity purpose
Server certificateBind server identity to a public key.
Ephemeral key exchangeProvide forward secrecy.
Transcript hashBind the whole handshake into authentication and key derivation.
Authenticated encryptionProtect application data after the handshake.
Version/cipher negotiation protectionReduce downgrade risk.
Key scheduleSeparate handshake, application, and exporter secrets.

C1 sample answer. The transcript hash binds the derived keys and authentication to the exact handshake messages, making tampering with the setup conversation detectable.

C2 sample answer. Certificate validation binds the server identity to its public key. If validation fails, the client may be talking securely to the wrong party.

Part E: Signal Protocol case study

Signal design featureSecurity purpose
Identity keysLong-term user identity anchor.
PrekeysAllow asynchronous session setup.
Triple/extended Diffie-Hellman-style setupMix long-term and ephemeral contributions.
Double RatchetContinuously update message keys.
Message keys deleted after useLimit damage from later compromise.
Safety numbersLet users verify identity keys out of band.

E1 sample answer. In messaging, both users may not be online at the same time. Prekeys let a sender start a secure session using key material published earlier by the recipient.

E2 sample answer. A ratchet keeps moving keys forward. Each message gets fresh key material, and old message keys are deleted so later compromise does not automatically reveal old messages.

Part G: SSH case study

SSH design featureSecurity purpose
Host keyAuthenticate the server across connections.
Trust On First UseRemember the first observed host key and warn on changes.
Key exchangeDerive session keys for encrypted transport.
User authenticationAuthenticate the client/user after server setup.
Strict host-key checkingDetect possible man-in-the-middle attacks after first trust.

G1 sample answer. TOFU trusts the first key it sees. If an attacker intercepts that first connection, the client may store the attacker’s key as trusted.

G2 sample answer. A changed host key may mean a legitimate server rebuild, but it may also indicate a man-in-the-middle attack. It should be checked before acceptance.

Part O: Concept checks

O1 sample answer. TLS provides secure client-server transport, especially for HTTPS. Its design lessons include certificate-based server identity, ephemeral key exchange, transcript binding, authenticated encryption, protected negotiation, and purpose-separated key schedules.

O2 sample answer. Signal is designed for end-to-end encrypted messaging, including asynchronous conversation setup. It uses identity keys, prekeys, Diffie-Hellman-style setup, ratcheting, deletion of old message keys, and safety numbers for out-of-band identity verification.

O3 sample answer. SSH often uses Trust On First Use for host keys, making the first connection especially important. WireGuard uses configured static public keys, which reduces negotiation complexity but makes correct key distribution and configuration essential.

O4 sample answer. This tutorial compared TLS, Signal, SSH, and WireGuard as real-world examples of protocol composition. It emphasized identity models, transcript binding, key schedules, ratchets, TOFU, minimalism, replay protection, downgrade resistance, metadata limits, and operational setup.
