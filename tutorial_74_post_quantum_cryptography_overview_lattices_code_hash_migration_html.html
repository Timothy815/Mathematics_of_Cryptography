Post-Quantum Cryptography: Lattices, Code-Based Systems, Hash-Based Signatures, and Migration

A first map of the post-quantum landscape: why quantum computers threaten RSA, Diffie-Hellman, and elliptic-curve cryptography; what ML-KEM, ML-DSA, and SLH-DSA are; how lattices, codes, and hash-based signatures fit together; and why migration is a protocol and engineering problem, not merely an algorithm swap.

Why this tutorial exists

Purpose

Volume 3 ended with real-world protocols. Volume 4 begins with the next great migration: replacing or supplementing classical public-key cryptography with systems believed to remain secure against quantum attackers. This tutorial is the map before the hike. We will not prove lattice hardness yet. We will name the kingdoms, describe their terrain, and explain why migration matters.

Post-quantum cryptography is not “quantum cryptography.” It is ordinary software cryptography designed to survive a future where attackers may own quantum machines. The battlefield changes, but the protocols still run on laptops, phones, servers, routers, and sleepy little embedded boxes blinking in closets.

What you should know first

You should know public-key encryption, key exchange, digital signatures, hash functions, security parameters, bits of security, protocol composition, TLS-style handshakes, and the basic difference between symmetric and public-key cryptography.

Why quantum computers change public-key cryptography

RSA depends on factoring being hard. Diffie-Hellman and elliptic-curve cryptography depend on discrete logarithms being hard. Shor’s algorithm threatens both factoring and discrete logarithms if a sufficiently large, fault-tolerant quantum computer becomes available.

Classical systemMain hard problemQuantum concern

Elliptic-curve Diffie-HellmanElliptic-curve discrete logarithmShor’s algorithm.

Shor attacks the trapdoor staircases of classical public-key cryptography. The stairs still look steep to ordinary computers, but a quantum attacker may have an elevator.

What about symmetric cryptography?

Symmetric cryptography is affected differently. Grover’s algorithm gives a quadratic speedup for brute-force search, which is often summarized as cutting an ideal \(n\)-bit search down toward about \(2^{n/2}\) quantum work. This does not destroy symmetric cryptography in the same way Shor threatens RSA and elliptic curves. It usually means choosing larger symmetric security margins.

\[ classical\ brute\ force\approx 2^n \] \[ Grover\text{-style search}\approx 2^{n/2} \]
Primitive familyQuantum impactTypical response
Symmetric encryptionQuadratic search speedup.Use larger keys when needed.
HashesSome goals get quantum speedups.Use adequate digest lengths and modern designs.
Public-key factoring/discrete-log systemsEfficient quantum attacks from Shor.Migrate to post-quantum assumptions.

KEMs versus signatures

Post-quantum migration is not one algorithm replacing everything. Key establishment and signatures are different jobs.

Common mistake

ML-KEM does not replace signatures. ML-DSA and SLH-DSA do not replace key establishment. They solve different protocol jobs.

ML-KEM: Module-Lattice-Based Key-Encapsulation Mechanism

What ML-KEM is for

ML-KEM is a post-quantum key encapsulation mechanism. It lets one party encapsulate a shared secret to another party’s public key. The other party decapsulates using the private key. Both end up with the same shared secret, which can feed symmetric encryption or a protocol key schedule.

\[ (ciphertext, shared\_secret)=Encapsulate(public\_key) \] \[ shared\_secret=Decapsulate(private\_key, ciphertext) \]

ML-DSA: Module-Lattice-Based Digital Signature Algorithm

What ML-DSA is for

ML-DSA is a post-quantum digital signature algorithm. It is used to sign data so that others can verify that the holder of the private key approved that data.

SLH-DSA: Stateless Hash-Based Digital Signature Algorithm

What SLH-DSA is for

SLH-DSA is a stateless hash-based signature algorithm. It relies mainly on hash-function security rather than lattice assumptions. This makes it conceptually conservative, though with larger signatures and different performance tradeoffs.

If lattice signatures are elegant new machinery, hash-based signatures are the sturdy old stone bridge. Heavy, maybe, but comforting.

Lattice intuition

A lattice is a regular grid of points formed by combining basis vectors with integer coefficients. Lattice cryptography often hides secrets inside high-dimensional noisy linear algebra. The attacker sees equations, but noise makes exact recovery hard.

\[ b = A s + e \pmod q \]

In this sketch, \(A\) is public structure, \(s\) is secret, \(e\) is small error or noise, and \(q\) is a modulus. Without the noise, solving for \(s\) would look like linear algebra. With carefully chosen noise in high dimensions, the problem becomes hard.

Analogy

Imagine hearing a precise melody through static. One noisy note is easy to guess. A huge orchestra of noisy equations becomes much harder to untangle.

Code-based cryptography intuition

Code-based cryptography is built from error-correcting codes. The friendly owner knows a code structure that can correct certain errors efficiently. The attacker sees something that looks like a random code and faces a hard decoding problem.

PieceMeaning
MessageOriginal data to protect.
Error-correcting codeStructure that can recover from certain errors.
Added errorsIntentional noise hiding the message.
Secret structureTrapdoor allowing efficient decoding.
Attacker problemDecode without knowing the hidden structure.

Migration: the hard part is not only mathematics

Post-quantum migration affects protocols, certificates, devices, libraries, packet sizes, storage formats, hardware acceleration, compliance, and operational workflows. A technically strong algorithm can still be hard to deploy safely.

Migration issueWhy it matters
Larger keys or signaturesCan affect packet sizes, certificates, memory, and bandwidth.
Protocol compatibilityOld systems may not understand new algorithms.
Hybrid handshakesCan combine classical and PQC security during transition.
Certificates and PKISignature algorithms and certificate chains must migrate.
Embedded devicesMay lack memory, CPU, or update paths.
Crypto-agilitySystems need safe upgrade paths for future changes.

Store-now, decrypt-later

The quantum threat is not only future communication. Attackers can record encrypted traffic today and decrypt it later if future quantum capabilities break the key exchange used to protect it. This is especially important for data that must remain secret for years or decades.

Long-lived secrecy warning

If a secret must remain confidential for twenty years, waiting until quantum computers are actually available may be too late. The ciphertext harvest may already be sitting in someone’s warehouse of patient doom.

Hybrid classical/post-quantum protocols

A hybrid protocol combines a classical method and a post-quantum method. The goal is to remain safe if one side later turns out weaker than expected.

\[ shared\_secret = KDF(classical\_secret \parallel pqc\_secret \parallel transcript) \]

Hybrid designs are not magic glue. They must bind the transcript, prevent downgrade, separate keys by purpose, and handle failures carefully.

Common mistakes

Mistake 1: thinking PQC means quantum computers run the protocol

Post-quantum cryptography is classical cryptography designed to resist quantum attackers.

Mistake 2: treating ML-KEM, ML-DSA, and SLH-DSA as interchangeable

ML-KEM is for key establishment. ML-DSA and SLH-DSA are for signatures.

Mistake 3: ignoring migration costs

Post-quantum objects can change sizes, performance, protocols, certificates, and storage.

Mistake 4: waiting too long because the quantum computer is not here yet

Store-now, decrypt-later attacks matter for long-lived secrets.

Mistake 5: forgetting crypto-agility

Today’s migration will not be the last. Systems should be built to change safely.

Why this matters for cryptography

Post-quantum cryptography is the next great public-key transition. It forces us to rethink what hardness assumptions we trust, how protocols establish keys, how certificates are signed, how long secrets must remain safe, and how systems migrate without collapsing under their own backward compatibility. It is also a reminder that cryptography is never finished. It evolves because attackers, machines, and mathematics evolve.

  • Shor threatens factoring and discrete logarithm systems.
  • Grover changes symmetric search estimates but does not destroy symmetric cryptography.
  • ML-KEM is for post-quantum key establishment.
  • ML-DSA and SLH-DSA are for post-quantum signatures.
  • Lattice cryptography often uses noisy linear-algebra assumptions.
  • Hash-based signatures rely mainly on hash-function security.
  • Code-based cryptography relies on hard decoding problems.
  • Migration requires protocol, engineering, and operational planning.
  • Store-now, decrypt-later makes the threat urgent for long-lived secrets.
  • Crypto-agility is no longer optional for serious systems.
